# 17.3.2.7 — `--preferred=0` beats `--membind=0` at SF300 by 12.9%

**Date:** 2026-05-04
**Workload:** SF300 (216 GB), --runs 5 each
**Status:** PASS — soft NUMA binding gives a new SF300 best of 31.3 GB/s

## Result

| Policy | Cold | Warm runs (2-5) | Median warm | GB/s |
|--------|------|-----------------|--------------|------|
| `--membind=0` (strict) | OOM at 2nd attempt; 19.7s on 1st | 7782, 7823, 7790, 7779 | 7790 ms | 27.7 |
| `--preferred=0` (soft) | 22106 ms | 6838, 6891, 6901, 6912 | **6896 ms** | **31.3** |

**Speedup of `--preferred` over `--membind`: 1.13×**
**Speedup of `--preferred` over default: ~1.96×** (default median was 13490 ms)

## Why `--preferred` wins

`--membind=0` forces ALL memory allocations to node 0. The 216 GB input
+ 216 GB output + 42 GB pinned + scratch = ~470-500 GB. Node 0 has
515 GB but typically only ~430 GB free, so the strict bind lands close
to the ceiling and OOMs intermittently (we saw the OOM on attempt 2).

`--preferred=0` uses node 0 first, then **spills to node 1** when full.
The likely outcome:
- Input mmap (216 GB read-mostly): on node 0 (preferred)
- Output buffer (216 GB write-mostly): probably spread across both
  nodes since node 0 is mostly full of input
- Pinned host buffers: node 0 first, then node 1

Then during the sort:
- Random reads from input → mostly local to node 0 threads
- Sequential writes to output → spread across BOTH nodes' DDR5 channels
  → 2× write bandwidth available

The output write bandwidth doubling outweighs the small remote-write
penalty when output pages happen to be on node 1.

## Variance comparison (4 warm runs)

| Policy | Range | StdDev / Median |
|--------|-------|------------------|
| Default | 7265 ms (10927-18192) | 28% |
| `--membind=0` | 44 ms | 0.2% |
| `--preferred=0` | 74 ms (6838-6912) | 0.4% |

Both numactl modes give tight variance. `--preferred` has slightly
wider but still 100× tighter than default.

## Recommended deployment recipe (UPDATED)

```bash
# Old recipe (17.3.2):
numactl --cpunodebind=0 --membind=0 ./external_sort_tpch_compact ...

# New recipe (17.3.2.7):
numactl --cpunodebind=0 --preferred=0 ./external_sort_tpch_compact ...
```

The soft binding is:
- **More robust** (won't OOM if node 0 is heavily loaded)
- **Faster at SF300+** (12.9% on this measurement; likely larger at SF500)
- **Same speed at SF50/SF100** (where the working set fits node 0
  comfortably, both modes behave identically)

## Implications for paper

This is a stronger SF300 number than was previously claimed. Update:

| Scale | Old (17.3.2.2) | New (17.3.2.7) |
|-------|----------------|------------------|
| SF300 single-GPU warm best | 7779 ms | **6838 ms** |
| SF300 throughput | 27.7 GB/s | **31.3 GB/s** |

The 31.3 GB/s number is also the highest sustained throughput we've
ever measured on a single H100 NVL — beating SF50 (25.6) and SF100
(24.4) by a comfortable margin. Counter-intuitive that the largest
workload is the fastest per-byte, but the OVC architecture amortizes
overhead across more records.

## Caveats

- This is one sweep. 17.3.2.5 showed multi-sweep variance differs.
- The --preferred policy depends on system memory state. If both
  nodes have plenty of free memory, --preferred and --membind behave
  similarly. If node 0 is full, --preferred succeeds where --membind
  fails.
- Could there be an artifact from system page-cache state? Possibly.
  Ideally would re-test on a fresh boot.
