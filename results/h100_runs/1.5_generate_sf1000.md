# 1.5 — generate TPC-H SF1000 lineitem

**Date:** 2026-05-02
**Scale:** SF1000 (720.00 GB, 5,999,989,709 × 120-byte records)
**Wall time:** 3617 s (60.3 min) end-to-end
**Status:** PASS (file size matches expected, peak RSS within limits)

## Command(s)

```bash
DATA_DIR=/mnt/data WORK_DIR=$HOME/gpu-research \
    bash ~/gpu-research/h100/gen_all_scales.sh 1000
```

## What worked

- **The chunked encoder (1.1.1, just landed in 950274d) made SF1000 possible
  on this 1 TB box.** Peak RSS during encode = 594 GB vs the projected
  1.4 TB peak with the old all-at-once `pl.read_parquet`. Without 1.1.1,
  this experiment would have OOM'd python at ~80% complete.
- Encode rate ticked up from 0.15-0.17 GB/s (pre-fix) to 0.21 GB/s — the
  per-row-group reads + immediate `del` improved cache locality in addition
  to bounding RAM.
- The combined parquet (~270 GB at SF1000) + binary (720 GB) used <1.4 TB
  on a 3.5 TB disk; comfortable margin.

## What failed

Nothing.

## Surprises

1. **RSS still drifted up to 594 GB during encoding,** despite chunked reads
   and `del`. pyarrow/polars apparently keeps internal buffer references
   that live longer than expected. Not a problem at 1 TB host RAM but worth
   noting — at SF2000 with the same drift the encoder might still need
   tuning (`gc.collect()` between row groups, or arrow buffer pinning).

2. **The encode rate went *up* by ~30% with chunked.** I would have predicted
   the chunked encoder to be slightly slower (more parquet metadata reads,
   more Python loop overhead). Instead it's faster, which says the old
   "load whole DataFrame" hit cache thrashing harder than I thought.

3. **Total disk now 40% full.** SF1500 (1.08 TB) plus SF2000 (1.44 TB) on
   top of SF1000 + others = ~3.6 TB of binaries → would exceed the 80% cap.
   Need to delete intermediate files (e.g., SF50 / SF100 once Tier 0 is
   done) before SF1500 / SF2000 generation.

## Interpretation

SF1000 generation is now feasible on this box, but it's a **60 min one-time
cost**. Combined with the 24-min SF300 and 44-min SF500 gens, we've spent
~2 hours of the week on data generation; that's reasonable amortized
across the dozen experiments that read these files. The chunked encoder
also makes the gen pipeline reusable: any future SF beyond 500 needs
1.1.1 to land.

## Follow-ups

- **1.5.1 (new) — gc.collect() between row groups in gen_tpch_fast.py.**
  Should drop the 594 GB peak RSS to a few tens of GB. Important for the
  next box that has only 256-512 GB RAM.
- **1.5.2 (new) — checksum verification.** sha256 of the binaries to make
  re-generations bit-equal. (SF10 was verified bit-equal to the original
  pre-fix encoder; SF50/SF100/SF300/SF500 weren't because they predate the
  fix. Worth re-baselining once.)
- 1.6 (next) is the SF1000 sort. Will almost certainly OOM on a single H100
  NVL (the SF500 sort already OOM'd at 360 GB), but worth confirming the
  failure mode for the paper.
