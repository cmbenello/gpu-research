# 17.3.2.3.3 — 15.5.3 distributed SF500 sort with NUMA wrap

**Date:** 2026-05-04
**GPU:** 4× NVIDIA H100 NVL (paired-concurrent: GPU 0+1, then 2+3)
**Workload:** SF500 (3.0 B records, 360 GB), partition + 4 sorted buckets
**Status:** PASS — modest 6% wall improvement, see analysis below

## Pipeline

```
Partition (CPU 64 thread, no numactl)  → 4 buckets in /dev/shm
   ↓
Pair A: GPU 0 + GPU 1, both numactl --cpunodebind=0 --preferred=0
   ↓
Pair B: GPU 2 + GPU 3, both numactl --cpunodebind=1 --preferred=1
   ↓
4 sorted bucket files (globally sorted when iterated in order)
```

`--preferred=N` (soft binding) was needed instead of `--membind=N`
(strict). With `--membind=N`, the per-pair memory pressure
(2 × 178 GB pinned + 2 × 90 GB output ≈ 536 GB) exceeded the
515 GB-per-node limit, OOM-killing 2 of 4 buckets.

## Result

| Phase | Wall | Notes |
|-------|------|-------|
| Partition (CPU) | 119 s | 3.00 GB/s end-to-end |
| Pair A (GPU 0+1, node 0) | 148 s | max wall: GPU 1 = 6601 ms |
| Pair B (GPU 2+3, node 1) | 140 s | max wall: GPU 2 = 4473 ms |
| Cleanup | 23 s | rm tmpfs files |
| **Total** | **7m10s** | vs 15.5.3 baseline **7m36s**, **1.06× faster** |

## Pair A vs Pair B asymmetry

Pair A walls (~6500 ms) are 50% slower than Pair B (~4400 ms). Same
GPU pair model, same NUMA configuration (just different node).
Why?

- **OS file cache state**: by the time Pair B runs, the partition
  output files are likely warm in node-1's page cache (or system
  cache). Pair A has cold partition files.
- **Memory state**: by Pair B's time, Pair A has freed its allocations,
  reducing memory pressure on the system.
- **No competing work on node 0**: when Pair A runs, it competes
  with the system's residual page-cache placements from the partition
  step (which ran on default NUMA before numactl took effect).

A reordered pipeline could mitigate: do partition under
`numactl --preferred=0` so output goes to node 0's page cache, then
Pair A runs faster against warm node-0 pages.

## Why only 6% improvement vs 15.5.3 baseline

The expected speedup (per single-GPU 17.3.2.1 = 1.84× warm) doesn't
fully translate to distributed because:

1. **Each pair has 2 GPUs sharing one node** — same memory channel
   contention seen in 17.3.2.3 (33% headroom lost).
2. **Cold-cache phase** in Pair A (no warm-up runs) is slower than
   the warm-cache 17.3.2.1 measurement.
3. **Partition phase** (119 sec) is unaffected by numactl — it's a
   single-process CPU job that already occupies whatever NUMA nodes
   the kernel placed it on.

## Headline numbers comparison

| Method | Wall | End-to-end GB/s |
|--------|------|------------------|
| 15.5.3 baseline (no numactl) | 7m36s | 0.79 |
| 15.5.3 + numactl --preferred | **7m10s** | **0.84** |
| Single-GPU SF300 numactl warm | 7.78 s | 27.7 (single-GPU) |

The single-GPU SF300 at 27.7 GB/s shows the H100 NVL's peak per-GPU
throughput. The 4-GPU distributed pipeline at 0.84 GB/s is 33× LESS
than per-GPU peak, because the partition phase (119 sec) dominates
and doesn't scale with GPU count.

## What this rules out for the paper

- "Distributed sort scales linearly with GPU count" — false. The
  partition phase is single-CPU bottleneck.
- "NUMA gives 1.84× speedup at all scales" — the 1.84× is for
  single-GPU; distributed gets only 1.06×.

## Key bottleneck identified

**The partition step (119 sec for 360 GB) is now the dominant cost
of distributed sort.** Optimizations that speed up the actual sort
phase have diminishing returns until partition is parallelized.

Next high-ROI work:
- Multi-process partition (4 partition workers each on its own NUMA
  node, each producing 1 bucket).
- GPU-resident partition (use GPU sample-sort for splitter
  selection, then GPU-side scan to assign).

## Caveats

- Single sweep, no statistical bounds on the 6% gain. Could be
  within noise.
- The baseline 15.5.3 was on 2026-05-03 — different system load,
  different page cache state. A like-for-like comparison would
  re-run the baseline today, but the file-cache state inevitably
  differs.
