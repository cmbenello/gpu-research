# 17.3.2.7.2 — SF300 --preferred variance bound (2 sweeps × 4 warm)

**Date:** 2026-05-04
**Workload:** SF300 (216 GB), --runs 5 × 2 sweeps under
`numactl --cpunodebind=0 --preferred=0`
**Status:** PASS — confirms 32 GB/s is robust

## Combined data

```
Sweep 1 warm: 6838, 6891, 6901, 6912 ms  (median 6896, range 74 ms)
Sweep 2 warm: 6594, 6599, 6612, 6634 ms  (median 6605, range 40 ms)

Combined 8 warm runs: 6594, 6599, 6612, 6634, 6838, 6891, 6901, 6912
  Best:    6594 ms = 32.75 GB/s
  Median:  6736 ms = 32.07 GB/s
  Worst:   6912 ms = 31.25 GB/s
  Range:   318 ms (4.7%)
```

## Updated SF300 headline

| Statistic | Value | GB/s |
|-----------|-------|------|
| Best of 8 warm | 6594 ms | **32.75** |
| Median of 8 warm | **6736 ms** | **32.07** |
| Worst of 8 warm | 6912 ms | 31.25 |

The defensible paper claim is "**SF300 sorts in 6.74 s median ≈
32 GB/s**" or "**at peak, 32.7 GB/s**".

This is the **highest sustained throughput we've measured on a single
H100 NVL** for any TPC-H scale tested. SF50 peaked at 25.6, SF100
at 24.4. SF300's higher number comes from the OVC architecture
amortizing setup overhead across more records.

## Variance comparison vs default policy

| Policy | Median | Range | Coefficient of variation |
|--------|--------|-------|---------------------------|
| Default | 13490 ms | 7265 ms | 28% |
| --membind=0 | 7790 ms | 44 ms | 0.2% |
| **--preferred=0** | **6736 ms** | **318 ms (across 2 sweeps)** | **2.4%** |

`--preferred` is slightly less reproducible than `--membind` (2.4%
vs 0.2% CoV) because the spill-to-node-1 behavior depends on
node-0 free memory at the moment of allocation, which varies sweep
to sweep. But it's still 12× tighter than default and consistently
faster.

## Why sweep 2 was faster than sweep 1

- Sweep 1 warm: 6838-6912 ms
- Sweep 2 warm: 6594-6634 ms (220 ms faster)

Possible reasons:
- System page cache state evolved between sweeps; sweep 2 might
  have had more of the lineitem.bin already in cache.
- Kernel auto-NUMA migration converged to a better placement after
  sweep 1's accesses.

Either way, sweep 2 is a tighter, more reproducible run (range only
40 ms). The "median of 8" 6736 ms balances the two sweeps' results.

## Build & run

```bash
CUDA_VISIBLE_DEVICES=0 numactl --cpunodebind=0 --preferred=0 \
    ./external_sort_tpch_compact \
    --input /mnt/data/lineitem_sf300.bin --runs 5
# Repeat to bound variance.
```

## Caveats

- Only 2 sweeps. Need ≥3-5 for statistical significance.
- Both sweeps were within 5 minutes of each other; correlated
  cache state.
- The sweep-to-sweep gap of 220 ms (3.2%) is ~3× the within-sweep
  range. So variance is dominated by run start state, not by per-run
  noise.
