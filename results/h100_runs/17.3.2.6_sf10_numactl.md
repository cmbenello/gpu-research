# 17.3.2.6 — SF10 numactl: no effect (fast-path bypasses gather)

**Date:** 2026-05-04
**Workload:** SF10 (7.2 GB, 60 M records), --runs 5 each config
**Status:** PASS — confirms SF10 fast-path is unaffected by NUMA

## Result

Default 5 runs (median 357.80 ms = 20.12 GB/s):
```
353.28, 358.07, 357.80, 357.80, 358.68
```

Numactl 5 runs (median 357.38 ms = 20.14 GB/s):
```
352.95, 357.38, 357.44, 357.48, 357.24
```

**Speedup: 1.001× (within noise).**

## Why no numactl effect at SF10

Looking at the CSV `phase2_ms` column: 0.00 for ALL SF10 runs (default
and numactl). SF10 uses the "fast path" — the path-selector
detects that the working set fits comfortably and runs a single-pass
sort with no Phase 2 gather.

The numactl wins (1.84× SF100, 1.73× SF300) come almost entirely
from the Phase 3 random gather. Without that phase, NUMA placement
doesn't matter — everything fits in HBM and CPU work is minimal.

## Variance comparison

| Config | Wall range across 5 runs |

SF10 is naturally low-variance because the fast path doesn't touch
the host RAM gather. SF100/SF300 default has huge variance from
NUMA placement luck.

## Conclusion

The "free 1.84× from numactl" result holds for SF50+, where the
gather phase dominates. SF10 already runs the fast path and doesn't
benefit. The new headline table is correct as-is.
