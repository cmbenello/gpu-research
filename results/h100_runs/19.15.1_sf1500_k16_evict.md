## Per-bucket inner sort wall

From CSV: **~5 sec / bucket** (sort itself).
Round wall: ~125 sec for 2 buckets = **~120 sec of NVMe + pin overhead per round**.

So the actual GPU sort is <10% of round time. The dominant cost is:
1. NVMe read of 67 GB (~25-30 sec at 2.4 GB/s SSD throughput for the bucket file)
2. cudaHostRegister of 67 GB (~55 sec at 1.2 GB/s NVMe-fault-bound rate)
3. Compact-key build, sort, gather (5 sec on GPU)
4. NVMe write of 67 GB (~30 sec)

K=16 roughly halves all of these vs K=8, saving ~62 sec/round. With 4 fewer rounds at K=8, savings net out.

## Why K=16 doesn't help more

The per-round overhead at K=16 (~125 sec) is dominated by file I/O and host
pinning, both of which scale linearly with bucket size. Halving bucket size
halves all overheads, but 2× the number of rounds undoes the savings.

True wins would come from changes that **don't scale with bucket size**:
- Eliminating the host pin entirely (broken at this scale — 19.7 NEGATIVE)
- Overlapping NVMe read with GPU sort (currently sequential)
- Concurrent partition + sort (would need partition + sort sharing GPUs)

## Conclusion

**K=8 is the sweet spot for SF1500.** K=16 buys 12 sec at the cost of 8
extra rounds and more partition overhead. Below K=8 (K=4) hits OOM at
this scale due to bucket size > 200 GB. Above K=8 (K=16, K=24...) gives
diminishing returns.

For the paper: report K=8 as the recommended config; document K=16 as
"linear scaling, no headroom" for completeness.
