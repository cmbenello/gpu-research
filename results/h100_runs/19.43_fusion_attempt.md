# 19.43 — Streaming fusion attempt (negative / blocked on admin)

**Goal:** Overlap pre-pin with partition's last 30% to save ~28s.

## What I tried mentally / on paper

1. **Pin buffers BEFORE partition starts.** Anonymous mmap regions are
   uncommitted; cudaHostRegister forces commit + lock. For 360 GB this
   takes ~3 min upfront — WORSE than current 28s pre-pin.

2. **Pin progressively during partition.** cudaHostRegister registers
   the whole virtual range at once. Splitting into chunks would require
   tracking offsets per chunk and unregistering separately. Fragile.

3. **Multi-thread cudaHostRegister with small chunks.** Tested in 19.30
   — the kernel acquires page-table locks per call, so parallel
   cudaHostRegister mostly serializes. 8 parallel threads on 8 buckets
   = ~28s vs ~38s sequential (only ~25% speedup).

4. **Producer/consumer: GPU sorts buckets while CPU partitions later
   input.** Each bucket's contents come from ALL input segments under
   uniform sample-sort, so no bucket is "ready" until partition fully
   ends. Hierarchical sample sort would fix this (partition into 4
   super-buckets first, then within each super-bucket re-partition + sort)
   but is a multi-day refactor.

## Conclusions

- **Current 28s pre-pin is close to optimal** without admin access.
- Going below requires:
  - **Huge pages (MAP_HUGETLB)** — needs sudo to allocate hugepage pool;
    would 5-10× pin throughput per the 19.11 measurement.
  - **GPUDirect Storage (cuFile + nvidia_fs kernel module)** — would
    skip host pin entirely, going NVMe → GPU directly.
  - **Hierarchical sample sort** — multi-day refactor, real gain unclear
    because we're already at NVMe ceiling.

## Why this isn't a critical bottleneck

Pre-pin at 28s is ~7% of our 6m23s SF1500 best. Even fully eliminating
it would save 28s. Compare to:
- NVMe input read (348s, 91% of wall) — fundamental hardware floor
- Output write (~30s) — also NVMe-bound

The pre-pin overhead is small relative to NVMe traffic. Optimizing it
further has diminishing returns.

## Where this leaves the recipe

The 19.30 streaming K=8 + pre-pin pipeline IS the practical optimum on
this hardware. To significantly improve requires hardware change or
admin access:

- Multi-NVMe striping → multiplied bandwidth ceiling
- More host RAM → SF1500 input fits in cache → matches SF1000's NVMe-peak result
- GPUDirect Storage → bypass host pin entirely

Without those, 6m23s/6m56s mean is the floor.
