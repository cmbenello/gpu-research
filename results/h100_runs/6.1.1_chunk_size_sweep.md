# 6.1.1 — COMPACT_CHUNK_RECORDS sweep: 32 M is fine, no win at larger sizes

**Date:** 2026-05-04
**Workload:** SF100 (72 GB) under numactl, --runs 5 each
**Status:** PASS — confirms 32 M chunk default is well-tuned

## Result

| Chunk | Cold (run 1) | Warm runs (2-5) | Median warm |
|-------|--------------|-----------------|--------------|
| 32 M (default) | 3087 ms | 2942, 2944, 2972, 2979 | **2944 ms** |
| 64 M | 3710 ms | 2938, 2939, 2950, 2950 | **2940 ms** |
| 128 M | 4982 ms | 2969, 2967, 2955, 2923 | **2961 ms** |

## Findings

1. **Median warm is essentially identical (2940-2961 ms) across all
   3 chunk sizes.** Within run-to-run noise. The 6.1 nsys hint that
   bigger chunks would push closer to 55 GB/s peak does not pan out
   in real wall time.

2. **Cold runs get slower with bigger chunks**: 3087 → 3710 → 4982 ms
   as chunk size doubles. This is `cudaHostAlloc` cost — pinning more
   host memory takes more time. On `--runs 5`, the cost is amortized
   away, but on a one-shot run it adds significantly.

## Why warm runs don't improve with bigger chunks

The bottleneck is not the H2D launch overhead — it's the per-chunk
CPU compact extract running concurrently with the previous chunk's
DMA. Even at 32 M, the extract is fully overlapped with H2D, so no
overlap-improvement is possible from larger chunks.

The 28 GB/s realized vs 55 GB/s peak from 6.1 nsys is the upper bound
of fragmented DMA, not a tunable. Closing that gap would require
single-monolithic-transfer instead of pipelined extract+upload —
but that loses the extract pipelining benefit (would add ~600 ms of
serial extract time).

## Conclusion

- **Keep COMPACT_CHUNK_RECORDS=32M default.** No improvement from
  larger.
- Env override is now available for future tuning if needed.
- For one-shot sorts, smaller chunks (16 M or 8 M) might be faster
  cold; not tested.

## Build & run

```bash
COMPACT_CHUNK_RECORDS=33554432 numactl --cpunodebind=0 --membind=0 \
    ./external_sort_tpch_compact --input lineitem_sf100.bin --runs 5
```

## Caveats

- Single sweep, no statistical bounds.
- Tested only at SF100. SF300 might show different pattern since
  it has 3× more chunks.
- The COMPACT_CHUNK_RECORDS env var only affects the OVC slow path
  (paths that hit `Step 1+2: Key Upload` with chunked compact).
  Single-pass paths and OVC compact-key paths use different upload
  logic.
