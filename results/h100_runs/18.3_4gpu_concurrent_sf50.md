# 18.3 — 4-GPU concurrent independent SF50: per-GPU 16.4 GB/s

**Date:** 2026-05-04
**Setup:** 4 GPUs each running independent SF50 sort concurrently
**NUMA:** GPU 0/1 → node 0 --preferred, GPU 2/3 → node 1 --preferred
**Status:** PASS

## Comparison to isolated single-GPU

| Mode | Per-GPU wall | Aggregate | Per-GPU throughput |
|------|--------------|-----------|---------------------|
| Isolated 1-GPU SF50 (18.2) | 1413 ms | 25.5 GB/s | 25.5 |
| **4-GPU concurrent SF50 (this)** | **~2200 ms** | **~64 GB/s** | **~16.4** |

Per-GPU wall is **1.55× slower** when running 4 concurrent jobs.
Per-GPU throughput drops 25.5 → 16.4 GB/s (35% reduction). This is
the node-pair contention penalty.

Aggregate: 4 × 16.4 = **64 GB/s for the 4-query batch** (144 GB
sorted in ~2200 ms).

## Same per-GPU rate as SF500 4-way

| Workload | per-GPU GB/s |
|----------|---------------|
| 4-GPU concurrent SF50 (4 × 36 GB) | 16.4 |
| 4-GPU partition SF500 (4 × 90 GB) | 16.4 |

**Identical.** Per-GPU throughput in 4-way concurrent mode is ~16
GB/s regardless of per-GPU workload size. The bound is node-pair
memory channel contention, not workload size.

## Implication for batch query workloads

If you have N independent SF50-sized sort queries to run on this box:
- Run them serially on 1 GPU: N × 1.41 s
- Run 4 in parallel on 4 GPUs: ceil(N/4) × 2.2 s

Crossover: 4-GPU parallel becomes faster when **N ≥ 2.5** (at N=2,
serial is 2.82 s vs 4-GPU parallel 2.2 s). So even small batches
benefit from 4-way concurrency.

For **paper claim**: "Throughput for batch SF50 sorting is 64 GB/s
on 4× H100 NVL = 1.6 SF50 queries per second."

## Caveats

- Single sweep. Want ≥3 sweeps for variance.
- All 4 GPUs sort the same input file (lineitem_sf50.bin). System
  page cache could over-share data, though each GPU has its own
  pinned-host copy after cudaHostRegister.
- The per-GPU 2200 ms wall is consistent within 7% across GPUs
  (2110-2251 ms). Slight bias toward node 1 GPUs being faster
  (2110 vs 2243 ms) — could be cold cache asymmetry.
