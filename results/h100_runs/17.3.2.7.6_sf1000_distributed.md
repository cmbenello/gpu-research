# 17.3.2.7.6 — SF1000 distributed sort with NUMA `--preferred` per pair

**Date:** 2026-05-04
**GPU:** 4× NVIDIA H100 NVL paired (GPU 0+1 → node 0, GPU 2+3 → node 1)
**Workload:** SF1000 (6 B records, 720 GB lineitem)
**Status:** PASS — 20m08s total, 12.6% faster than baseline

## Result

| Phase | Wall | Notes |
|-------|------|-------|
| Partition (CPU) | 7m52s | 1.52 GB/s end-to-end (NVMe-bound) |
| Pair A (GPU 0+1, node 0) | 5m49s | both GPUs ~45 sec wall |
| Pair B (GPU 2+3, node 1) | 6m04s | both GPUs ~44 sec wall |
| **Total** | **20m08s** | vs 15.5.3 baseline 22m40s, **1.13× faster** |

## Per-GPU sort walls

| GPU | Bucket size | Wall | GB/s |
|-----|-------------|------|------|
| 0 | 180.85 GB | 44.77 s | 4.04 |
| 1 | 178.39 GB | 44.90 s | 3.97 |
| 2 | 177.70 GB | 44.88 s | 3.96 |
| 3 | 183.06 GB | 42.29 s | 4.33 |

All 4 GPUs finish in 42-45 sec — extremely consistent. The `--preferred`
NUMA wrap helped each pair share its node's memory channels evenly,
avoiding the GPU 1/GPU 2 lag we saw in 17.3.2.3.3 SF500 (where pair A
was 50% slower than pair B due to cold-cache asymmetry).

End-to-end throughput improved 12.6% (0.529 → 0.596 GB/s).

## Where the time goes

```
20m08s total
├── Partition:  7m52s  (39%)  ← NVMe-bound, 1.52 GB/s
├── Pair A sort: 5m49s  (29%)
├── Pair B sort: 6m04s  (30%)
└── Other:       0m23s  ( 2%)  ← cleanup, status reporting

Sort total (Pair A + B): 11m53s = 720 GB / 713 sec = 1.01 GB/s
```

The partition phase is now 39% of total wall — solidly the bottleneck.
The combined sort time is at NVMe-write-rate for 720 GB output.

## Why pair A and pair B took similar time (5m49s vs 6m04s)

In 17.3.2.3.3 SF500, pair A took 148 sec and pair B took 140 sec —
a 5% gap. Here at SF1000 the pair A took 349 sec and pair B took
364 sec — a 4% gap. So the gap doesn't grow with workload size,
just stays small.

Per-GPU walls (44-45 sec each across all 4) show that the workload
is well-balanced. NUMA --preferred is doing its job evenly across
both nodes.

## Caveats

- Single sweep. SF1000 takes 20+ min so multi-sweep variance is
  expensive to bound; not done.
- NVMe write rate dominates (315 sec for 720 GB partition output =
  2.28 GB/s, close to NVMe write peak). Without faster storage,
  the partition phase is hard to speed up further.
- Scientific control: would need to rerun 15.5.3 baseline today
  for an apples-to-apples comparison. Hardware/OS state may have
  changed since 2026-05-03.

## Follow-ups

- 17.3.2.7.6.1 — pre-partitioned SF1000 sort: skip the 8-min
