# 18.1 — Small-N correctness + latency floor sweep

**Date:** 2026-05-04
**GPU:** 1× NVIDIA H100 NVL, numactl --preferred=0
**Workload:** SF50 input, sliced to N records via `--limit-records`
**Status:** PASS — all 6 sizes verified correct

## Result

| Records | Wall (ms) | Throughput (GB/s) | Verify |
|---------|-----------|-------------------|---------|
| 1,000 | 2.46 | 0.05 | PASS |
| 10,000 | 2.75 | 0.44 | PASS |
| 100,000 | 3.53 | 3.4 | PASS |
| 1,000,000 | 9.76 | 12.3 | PASS |
| 10,000,000 | 64.71 | 18.5 | PASS |
| 100,000,000 | 603.51 | 19.9 | PASS |

## Findings

1. **Per-process overhead floor: ~2.5 ms.** The 1k-record sort
   takes 2.46 ms — essentially the cost of CUDA initialization and
   path-selector dispatch.

2. **Throughput scales from latency-dominated (0.05 GB/s at 1k) to
   bandwidth-saturated (~20 GB/s at 100M+).** Crossover near 1M
   records (12.3 GB/s). The fast path scales linearly above this
   point.

3. **All sizes use Phase 1 only (phase2_ms = 0).** The path-selector
   keeps small inputs on the fast single-pass path; OVC/gather only
   kicks in at larger scales (>~100M records).

## Latency vs scale fit

For N records, wall ≈ 2.5 ms + N × 6 ns (above 1M):
```
N=10M:   2.5 + 60   = 62.5 ms (measured 64.7)
N=100M:  2.5 + 600  = 602 ms  (measured 603.5)
```

Tight match, ~6 ns/record at scale.

## What this rules out

- "The sort silently corrupts data at small N": no — all sizes
  PASS.
- "Hidden 100ms+ setup cost": no — 1k records run in 2.5 ms.
  Process startup is ~2.5 ms.
- "Fast path is buggy at sub-1M scale": no — 1M PASS at 12.3 GB/s.

## Useful for paper

Throughput scaling chart from 1k to 100M (and onward to SF300 at
1.8B from elsewhere) covers 6 orders of magnitude. The "kneepoint"
where throughput plateaus is at about 1M records.

## Build & run

```bash
for N in 1000 10000 100000 1000000 10000000 100000000; do
    numactl --cpunodebind=0 --preferred=0 \
        ./external_sort_tpch_compact \
        --input lineitem_sf50.bin --offset-records 0 --limit-records $N
done
```

## Caveats

- Single run per N. Variance unknown.
- All sizes use the fast path; doesn't test the OVC slow path
  edge cases.
- Used SF50 input data; other distributions (e.g., all-equal keys,
  reverse-sorted) not tested here. Tier 18.2/18.3 would cover those.
