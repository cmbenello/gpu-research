# 18.2 — Per-GPU SF50 sort: all 4 H100s give identical timing

**Date:** 2026-05-04
**GPUs:** 4× NVIDIA H100 NVL (each tested individually)
**Workload:** SF50 (36 GB), --runs 5 each, with numactl --preferred per node
**Status:** PASS — all GPUs within 0.18% of each other

## Result (median warm of 4 warm runs)

| GPU | NUMA | Cold (ms) | Warm walls (ms) | Median warm | GB/s |
|-----|------|-----------|------------------|--------------|------|
| 0 | node 0 | 1834.95 | 1411.59, 1413.54, 1412.75, 1414.52 | 1413.05 | 25.48 |
| 1 | node 0 | 1837.81 | 1415.81, 1419.24, 1412.63, 1415.40 | 1415.61 | 25.43 |
| 2 | node 1 | 1840.00 | 1417.60, 1416.62, 1410.73, 1412.86 | 1414.74 | 25.45 |
| 3 | node 1 | 1831.79 | 1413.13, 1413.71, 1413.45, 1410.80 | 1413.29 | 25.47 |

## Findings

1. **All 4 GPUs give the same SF50 wall** within 2.6 ms (0.18%
   range). Median spread: 1413.05 - 1415.61 = 2.56 ms.

2. **Cold runs are also tightly clustered:** 1832-1840 ms across
   all 4 GPUs. Even the cold-run pathology has the same shape on
   each GPU.

3. **Throughput: 25.43-25.48 GB/s** across all 4 GPUs.

## Why this matters

Combined with 6.2.1 (HBM bandwidth uniform within 0.5% across 4
GPUs), this confirms:

- **All 4 H100 NVLs in this box are interchangeable.** No degraded
  GPU. Any single-GPU benchmark applies to all 4.
- **NUMA --preferred works symmetrically.** GPU 0/1 on node 0 and
  GPU 2/3 on node 1 produce identical results.

For paper claims: any "per-GPU" number we report is representative
of all 4 GPUs, not just the one we measured.

## Implication for multi-GPU scaling

If all 4 GPUs run a single SF50-equivalent workload (each their
own 36 GB chunk), aggregate would be 4 × 25.5 = 102 GB/s.

In reality (17.3.2.3), 4 GPUs on 90 GB chunks each (SF500 split)
gave 65.7 GB/s aggregate = 16.4 GB/s/GPU. The drop from 25.5 (per-
GPU peak) to 16.4 (in 4-GPU concurrent) is the **35% loss to
node-pair contention** documented in 17.3.2.3.2.

So per-GPU peak ≠ multi-GPU aggregate / 4. Concurrent GPUs hit
memory contention.

## Cold vs warm spread

| GPU | Cold | Warm best | Cold/Warm ratio |
|-----|------|------------|------------------|
| 0 | 1834.95 | 1411.59 | 1.30× |
| 1 | 1837.81 | 1412.63 | 1.30× |
| 2 | 1840.00 | 1410.73 | 1.30× |
| 3 | 1831.79 | 1410.80 | 1.30× |

All 4 GPUs have the same cold-run penalty (~30%). This is the
"pinned host memory cost" identified in 6.1 nsys (cudaHostAlloc +
cudaHostRegister ~400 ms once per program).

## Build & run

```bash
for G in 0 1 2 3; do
    NODE=$((G / 2))
    CUDA_VISIBLE_DEVICES=$G numactl --cpunodebind=$NODE --preferred=$NODE \
        ./external_sort_tpch_compact \
        --input /mnt/data/lineitem_sf50.bin --runs 5
done
```

## Caveats

- Single sweep per GPU. Variance unknown. Want ≥3 sweeps per GPU.
- Tests run sequentially; system page-cache state evolves between
  GPUs. Earlier GPUs (0 + 1) might have warmer state for later
  GPUs (2 + 3) — but they all gave nearly identical results, so
  page-cache effect is negligible.
- Only tested at SF50. Larger workloads might surface per-GPU
  differences (HBM thermal throttle, etc.) — though 11.5 throttle
  test showed no thermal issue at SF100.
