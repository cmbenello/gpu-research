# 17.3.2.7.3 — SF500 single-GPU still OOMs (HBM limit, not NUMA)

**Date:** 2026-05-04
**Workload:** SF500 (360 GB) on 1 × H100 NVL with `numactl --preferred=0`
**Status:** FAIL — Phase 2 (GPU prefix merge) OOMs at HBM ceiling

## Result

```
Allocating 360.00 GB pinned host memory: 177342 ms (2.03 GB/s)
[ExternalSort] GPU: 98.60 GB free, budget: 64.09 GB
[ExternalSort] Triple-buffer: 3 × 21.36 GB

== Phase 1: Run Generation == [PASS]
  5 runs in 21611 ms (16.66 GB/s)

== Phase 2: GPU 16B Prefix Merge ==
  GPU free before merge alloc: 38.48 GB
  CUDA error at src/external_sort.cu:2340: out of memory   [FAIL]
```

Phase 1 (chunked LSD sort) successfully produced 5 sorted runs on
GPU. Phase 2 needed to allocate the prefix merge buffer (~30+ GB)
but only 38 GB HBM was available, and allocator returned OOM.

## Why SF500 single-GPU is still infeasible

H100 NVL has 94 GB HBM. SF500 = 3 B records × 16 B OVC key/perm
= 48 GB minimum just for the keys. Plus the prefix merge needs
double-buffered storage (~96 GB). So SF500 in single-pass on one
GPU NEVER fits in 94 GB HBM, regardless of NUMA settings.

Numactl helps host RAM placement; it doesn't help GPU memory.

## What --preferred DID accomplish

The host RAM allocations all succeeded:
- Pin 360 GB input on host
- Pin output buffer
- Stage 5 chunks of compact keys

So `--cpunodebind=0 --preferred=0` worked for the host-side parts —
no host OOM despite needing 720+ GB total host RAM. The spill-to-
node-1 behavior of `--preferred` did its job.

## Conclusion

- **SF500 single-GPU is a hard stop on H100 NVL's 94 GB HBM.**
- Numactl wrap doesn't change this; it only helps host RAM.
- The 4-GPU position-partitioned (15.4 / 17.3.2.3) and globally-
  sorted distributed (15.5.3 / 17.3.2.3.3) pipelines are the
  correct path for SF500+.

This is a useful **negative result** that confirms the multi-GPU
pipeline is required, not a workaround.

## Caveats

- Tested with `--runs 1`. With `--runs 5`, persistent buffers might
  reduce the per-run alloc footprint slightly. Untested.
- The input load took 177 sec (2.03 GB/s vs NVMe's 6.2 GB/s peak)
  — that's `cudaHostRegister` walking 360 GB of pages and pinning
  them. Most of that 177 sec is the pin step, not disk read.

## Phase 1 sub-result (interesting)

Phase 1 alone produced 5 sorted GPU runs (~80 GB output) in 21.6 sec
= **16.66 GB/s**. That's solid — shows the chunked LSD path is
working. The OOM is in the merge step, not the sort step.

A Phase-1-only mode (output 5 sorted slices to host, skip merge,
return them as 5 sorted streams) would let single-GPU produce
SF500-output for downstream processing. Could be useful for stream
processing where you don't need globally sorted output.

## Follow-ups

- 17.3.2.7.3.1 — implement Phase-1-only mode that writes the 5
  intermediate runs to host without trying to GPU-merge. Useful as
  a single-GPU "streaming sort" primitive.
- 17.3.2.7.3.2 — measure exactly what allocation at line 2340 OOMs.
  Maybe a smaller buffer size would let SF500 squeak through.
