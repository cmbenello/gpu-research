# 19.1 — SF1500 (1.08 TB > 1 TB host RAM): boundary found

**Date:** 2026-05-04
**Workload:** SF1500 (9 B records, 1.08 TB lineitem)
**Status:** PARTIAL — 2/4 buckets sorted; 2/4 OOM-killed under paired-concurrent

This is the **out-of-host-RAM regime test** the user asked for.

## Pipeline (paired-concurrent, identical to SF1000 17.3.2.7.6)

```
00:00 — start
       partition 1.08 TB → 4 buckets on NVMe
18:00 — partition done (1.00 GB/s end-to-end, 18m partition wall)
18:00 — pair A start (GPU 0 + GPU 1 on node 0)
26:09 — pair A done (8m09s)
        - GPU 0: PASS, 38.8 sec wall, 6.88 GB/s on 267 GB bucket
        - GPU 1: OOM-killed (anon_rss 482 GB; node 0 ran out)
26:09 — pair B start (GPU 2 + GPU 3 on node 1)
34:27 — pair B done (8m18s)
        - GPU 2: PASS, 60.6 sec wall, 4.51 GB/s on 273 GB bucket
        - GPU 3: OOM-killed
34:48 — wall total
```

## What we learned

### The architecture's hard ceiling

| Workload | Per-bucket size | Per-pair RAM (input+output) | Result |
|----------|------------------|------------------------------|--------|
| SF1000 (180 GB/bucket) | 180 GB | ~360 GB pinned + lazy outputs | PASS in 14m20s |
| **SF1500 (270 GB/bucket)** | 270 GB | ~540 GB pinned + outputs | **OOM on 2/4** |

Per-node RAM = 515 GB. Pair on same node:
- SF1000: 2 × 180 GB pinned + slow lazy outputs ≈ fits in 515 GB
- **SF1500: 2 × 270 GB pinned + outputs ≈ 1.08 TB > 1 TB host RAM**

OOM dump shows **anon_rss 482 GB per process** at SF1500; pair = 964 GB > available.

### Boundary: above 1 TB host RAM, paired-concurrent fails

The SF1500 is **unsafe to run with paired-concurrent sort on this hardware**.

## What still works at SF1500

The 2 buckets that did succeed sorted correctly:
- GPU 0: 267 GB bucket in 38.8 sec = 6.88 GB/s
- GPU 2: 273 GB bucket in 60.6 sec = 4.51 GB/s

These are **single-process** times (the other GPU on the same node
got OOM-killed, freeing memory for the survivor). So one bucket
per node DOES work at SF1500 size.

## The fix: 2-GPU one-per-node sequential rounds

Like 18.5c at SF1000, run only 1 GPU per node at a time:
- Round 1: GPU 0 (node 0) + GPU 2 (node 1) sort buckets 0, 2
- Round 2: GPU 0 (node 0) + GPU 2 (node 1) sort buckets 1, 3

Each round uses 270 GB pinned per node. Fits in 515 GB.
Predicted total: ~5 min × 2 rounds = 10 min sort phase.
Plus 18 min partition = 28 min total for SF1500.

## Comparison: scaling boundary

| Scale | Records | Bytes | Architecture status |
|-------|---------|-------|----------------------|
| SF50-300 | 0.3-1.8B | 36-216 GB | Single-GPU works |
| SF500 | 3B | 360 GB | Single-GPU OOMs HBM; 4-GPU works |
| SF1000 | 6B | 720 GB | 4-GPU paired works (14m20s) |
| **SF1500** | **9B** | **1080 GB** | **paired OOMs; need one-per-node sequential** |
| SF2000 | 12B | 1440 GB | exceeds NVMe partition output (>1.7 TB free) |

So **SF1500 is the practical limit** for this hardware with the
current architecture, and even then requires the careful one-per-node
sequential strategy.

## Above-RAM is a real architectural concern

The current gpu_crocsort architecture **mmaps input** and **pins
output**. Both require physical RAM. For data > host RAM, we'd need:
- True streaming sort (input read incrementally from NVMe to GPU)
- Or out-of-core merge with disk-resident runs
- Or NCCL all-to-all (each GPU only has its share, never holds all)

These are **non-trivial code changes** — paper-2 territory.

## Implication for paper

Two new claims:
1. **SF1500 partition completes** (1.08 TB → 4 sorted buckets, 18 min)
2. **Sort phase boundary is per-node RAM**, not GPU HBM. SF1500
   requires one-GPU-per-node sequential strategy. With that, we
   can sort SF1500 globally in an estimated ~30 min total.

Recommended retry: implement the one-per-node sequential SF1500.
