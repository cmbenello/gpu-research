## Findings

1. **All 4 GPUs are within 0.5%** of each other at every size — no
   degraded GPU on this box. Useful for the paper: results from any
   one GPU generalize to all four.

2. **HBM bandwidth is flat across working-set size** above 1 GB.
   Tiny dip at 16 GB read-only (~4% lower) is statistical noise from
   ITERS=5 — not a real cliff.

3. **2191 GB/s is the headline read+write peak.** That's 65% of the
   spec sheet 3.35 TB/s. To approach the spec peak you'd need
   multi-stream concurrent kernels filling each HBM stack independently
   (out of scope for the sort path, which has serial dependency
   between LSD passes).

4. **1521 GB/s is the read-only peak.** Useful when characterizing
   reduction-style passes (sample sort splitter selection, statistics
   passes).

## Why this matters for the paper

When we report "single-GPU sort throughput X" we need to know that
the GPU we measured on isn't an outlier. This sweep confirms uniformity
across the 4-GPU box, so any number reported for "1× H100 NVL" is
representative.

## Build & run

```bash
nvcc -O3 -arch=sm_90 experiments/hbm_saturation_sweep.cu \
     -o experiments/hbm_saturation_sweep
./experiments/hbm_saturation_sweep > 6.2.1_hbm_sweep.csv
```

Total wall time: ~3 minutes (5 sizes × 4 GPUs × ~5 s per measurement).
