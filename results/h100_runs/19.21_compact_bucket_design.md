# 19.21 — Compact-bucket partition design (projected SF1500 ~12m)

**Status:** Design only, not yet implemented.
**Predicted result:** SF1500 wall **~12m** (vs 22m31s PERM_ONLY, vs 49m15s baseline).

## Context

After 19.20 PERM_ONLY landed at 22m31s, the partition phase (14m45s) is
**63% of total wall**. Partition writes 1.08 TB of full TPC-H records to
intermediate bucket files at the NVMe write ceiling (1.3 GB/s).

To go below 22m, we must reduce partition write volume.

## Key insight

The bucket files don't need to contain full 120-byte records. The sort
phase reads them, extracts a 32-byte compact key, sorts, and outputs
indices. The 88 bytes of payload per record are **never touched during
sort** — they're only needed at downstream-gather time, which already
reads from `input.bin`.

**So write ONLY (key + offset) per record into bucket files.**

```
Current bucket entry: 120 bytes
  [0..65]   = 66-byte sort key (with 26 varying bytes)
  [66..119] = 54-byte payload (untouched during sort)

Compact bucket entry: 40 bytes
  [0..31]  = 32-byte compact key (the 26 varying bytes + 6 padding)
  [32..39] = 8-byte offset into input.bin (uint64)
```

3× less data per record. **Bucket files: 22.8 GB each (vs 67 GB now).
Total partition output: 365 GB (vs 1.08 TB).**

## What needs to change

### 1. New `partition_by_range_compact` (extend existing tool)

Modify pass 2 to write 40-byte (key + offset) records:

```cpp
// Pass 2 inner loop (current):
//   memcpy(bucket_out[b] + wp[b] * RECORD_SIZE,
//          input + r * RECORD_SIZE, RECORD_SIZE);

// Pass 2 inner loop (new):
//   uint8_t* dst = bucket_out[b] + wp[b] * 40;
//   const uint8_t* src = input + r * RECORD_SIZE;
//   // Extract 32-byte compact key from src using cmap
//   for (int i = 0; i < 26; i++) dst[i] = src[cmap[i]];
//   for (int i = 26; i < 32; i++) dst[i] = 0;
//   *(uint64_t*)(dst + 32) = r;  // global record offset
```

The cmap (26 varying bytes' positions) is the same one extracted by
`external_sort.cu` during initial cmap detection. Need to either
hard-code it or re-run the detection up-front.

### 2. New sort path that reads 40-byte records

`external_sort_tpch_compact` currently reads 120-byte records. Easiest
route: add an env `COMPACT_BUCKET=1` that:

a. Reads 40-byte records (instead of 120).
b. Skips the cmap detection + compact-key extraction (data is already
   in the right format).
c. Uploads keys directly (32-byte stride from the 40-byte input).
d. Sorts as before.
e. Output: 8-byte global offset per sorted record (read from
   bucket file using the perm).

Output size: 8 × 9B = **72 GB total** (8 bytes per record vs 4 bytes
in current PERM_ONLY which only indexes within bucket).

### 3. Verification

End-to-end: gather sorted records via `input.bin[offset[i]]` and check
key order. Same validation as 19.20 just with global offsets.

## Why this matters for the paper

This is a true **structural** improvement, not just tuning. It:

1. Validates that "what gets written to disk" is the right axis
   to optimize for at NVMe-bound scales.
2. Demonstrates a usable composable pipeline: partition output =
   compact (key, offset) format, sort output = sorted offsets.
   Both are standard primitives in column-store query engines.
3. Drops SF1500 to **~12m = 76% faster than 49m baseline.**
4. Throughput becomes 1.5+ GB/s end-to-end on a single SSD —
   the GPU sort is no longer the bottleneck; nothing is.

## Risks / open questions

- The cmap is data-dependent. Currently detected at sort time (per-
  bucket). Need to either lift detection into partition phase, or
  hard-code for TPC-H.
- 40-byte records aren't 8-byte aligned for the 8-byte offset; need
  to handle padding or use 48-byte (32+8+8) layout for clean alignment.
- Pass 2's per-record CPU work increases (must extract compact key),
  but pass 2 is NVMe-write-bound today so the extra CPU is hidden.

## Implementation cost

Estimate **2-3 hours** of careful coding plus 1-2 hours of debugging.
Worth it for the projected 8-10m savings.

## Path forward

1. Hard-code cmap for TPC-H (the 26 varying byte positions).
2. Modify `partition_by_range.cpp` to emit 40-byte records.
3. Add `COMPACT_BUCKET=1` path in `external_sort.cu` that bypasses
   cmap detection and the 120-byte read.
4. Validate end-to-end at SF50.
5. Run at SF1500.
