|--------|------|--------------|------|-------------|

## Why `--preferred` beats `--interleave` and `--membind`

**The asymmetric memory access pattern is the key:**

- Random reads (input → gather threads): need LOW LATENCY → benefit
  from staying on node 0 (local to the worker threads).
- Sequential writes (gather output): need HIGH BANDWIDTH → benefit
  from using BOTH nodes' DDR5 channels.

| Policy | Read placement | Write placement | Write bandwidth |
|--------|----------------|------------------|-----------------|
| `--membind=0` | node 0 (local) | node 0 only | 1× DDR5 |
| `--interleave=0,1` | 50% remote | 50% remote | 1× DDR5 each |
| `--preferred=0` | node 0 (local) | node 0 first then node 1 | up to 2× DDR5 |

`--preferred` uniquely gives:
- Local reads (no UPI penalty for random access)
- Spread writes (both nodes' channels for sequential bandwidth)
- Fallback safety (doesn't OOM if node 0 fills up)

`--interleave` gives 50/50 split for everything → both reads and
writes pay UPI penalty.

`--membind` doesn't spill → caps writes at 1× DDR5 bandwidth and
risks OOM.

## Updated production deployment recipe

```bash
numactl --cpunodebind=0 --preferred=0 ./external_sort_tpch_compact \
    --input lineitem_sf300.bin --runs 5
```

This is the **best-tested** wrap for single-GPU sort. Same speed as
other policies at SF50/SF100 (where node 0 fits comfortably) and
fastest at SF300+ where it matters most.

## Cold runs

All 3 numactl modes have similar cold-run wall (~20-22 sec). Default
cold is 60 sec — 3× slower because of pathological first-touch
NUMA placement. Numactl wrap (any mode) avoids the cold pathology.

This is achieved with a single-line wrapper:

No code change required. Highest single-GPU throughput we've ever

## Caveats

- Single sweep per policy (except `--preferred` which has 2). Need
  ≥3 sweeps each for statistical bounding.
- Tested only at SF300. SF100 didn't show difference between
  policies (17.3.2.7.1) because node 0 has plenty of headroom.
- `--preferred` win depends on system memory state. If node 1 is
  full, --preferred falls back to using node 0 only (similar to
  --membind).
