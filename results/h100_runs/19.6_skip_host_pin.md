# 19.6 — SKIP_HOST_PIN=1 implementation: 9% faster SF300 warm

**Date:** 2026-05-04
**Goal:** Verify the 19.5 prediction by adding env override to skip cudaHostRegister
**Status:** PASS — 9% warm improvement (smaller than predicted but real)

## Code change

`external_sort.cu` line 3088-3110: added `SKIP_HOST_PIN=1` env check
that bypasses `cudaHostRegister` on the input mmap. Single-line change,
fully backward compatible.

## Result (SF300 --runs 3)

| Mode | Cold (ms) | Warm 1 | Warm 2 | Warm median |
|------|-----------|---------|---------|--------------|
| Default (with cudaHostRegister) | 20487 | 7457 | 7406 | **7430** |
| **SKIP_HOST_PIN=1** | 21793 | 6762 | 6784 | **6773** |

**Warm-run improvement: 8.8% wall reduction.**
**Cold-run regression: 6.4% slower.**

## Per-phase (cold run)

| Phase | Default | SKIP_HOST_PIN | Difference |
|-------|---------|----------------|-------------|
| Phase 1 (extract+upload+sort) | 16148 ms | 17943 ms | +1795 (slower) |
| Phase 2 (merge+gather+fixup) | 4339 ms | 3850 ms | -490 (faster) |

The cold Phase 1 is slower because each compact-key chunk's cudaMemcpy
from non-pinned host runs at ~10 GB/s vs ~55 GB/s for pinned. There
are 3 chunks of 18-24 GB each at SF300 = ~3 × 1.5 sec extra = 4.5 sec
extra. Matches the observed +1.8 sec Phase 1 delta (the CPU extract
pipeline overlaps the upload, hiding most of it).

The Phase 2 IS faster regardless (-490 ms). Possibly because pinning
makes the CPU's gather-phase reads of h_data slower (CUDA holds page
locks).

## Per-phase (warm)

| Phase | Default warm | SKIP_HOST_PIN warm | Difference |
|-------|---------------|---------------------|--------------|
| Phase 1 | 3164 ms | 2935 ms | -229 (faster) |
| Phase 2 | 4267 ms | 3838 ms | -429 (faster) |

Both phases are faster on warm runs! The pin no longer helps because
the input is in cache. So skipping pin is uniformly better warm.

## Why the 9% warm vs 30× predicted

The 19.5 microbench predicted 30× because it was measuring a SINGLE
cold pin + single transfer. In practice:
1. Pin happens ONCE per program invocation (cold run 1).
2. Warm runs 2,3 don't re-pin.
3. At SF300, the pin cost (~180 sec) is amortized over 3 runs.
4. The remaining gain is from per-cudaMemcpy unpinned penalty (smaller).

For SF1500, where each bucket is sorted ONCE in a one-shot mode, the
30× prediction would actually apply. Re-test there.

## What this means for paper

Three usable claims from this discovery:
1. **SKIP_HOST_PIN=1 gives ~9% wall reduction at SF300 warm**. Free
   improvement.
2. **Architecture-level insight:** for one-shot sort (no warm runs),
   the pin cost dominates. We shouldn't pay it.
3. **The cudaHostRegister API is not optimal for OLAP sort**. This
   is a portable systems-engineering finding.

## Implementation

```cpp
const char* skip_pin_env = getenv("SKIP_HOST_PIN");
bool skip_pin = skip_pin_env && atoi(skip_pin_env) > 0;
cudaError_t reg_err = cudaSuccess;
if (!skip_pin) {
    reg_err = cudaHostRegister(h_data, total_bytes, cudaHostRegisterDefault);
}
// Rest of code unchanged — cudaMemcpy auto-detects pinned vs not.
```

## Caveats

- Single test, no statistical bound.
- cudaMemcpy from non-pinned has different stream/async behavior. Not
  validated for all code paths.
- Untested on SF1500 where the win should be biggest. NEXT STEP.
