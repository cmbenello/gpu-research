# 17.2 — Scaling law fit: linear in N, 3.5 ns/record

**Date:** 2026-05-04
**Method:** scipy.optimize.curve_fit on warm-best wall times
**Status:** PASS — gpu_crocsort sort time scales LINEARLY with N (radix sort)

## Data points (single-GPU, numactl --preferred, warm best)

| Scale | Records | Wall (s) |
|-------|---------|----------|
| SF10 | 60 M | 0.353 |
| SF50 | 300 M | 1.408 |
| SF100 | 600 M | 2.952 |
| SF300 | 1.8 B | 6.594 |

## Linear model fit

```
t = 3.51e-9 · N + 0.404 s
```

Where:
- α = 3.51 ns/record (per-record cost; matches expected cost of LSD pass + gather)
- β = 0.404 s (fixed setup overhead per call: cudaInit + alloc + path-selector)

### Fit quality

| Scale | Measured | Predicted | Error |
|-------|----------|------------|-------|
| SF10 | 0.353 | 0.610 | +74% (fast path skips Phase 2) |
| SF50 | 1.408 | 1.457 | +3.5% |
| SF100 | 2.952 | 2.511 | −15% |
| SF300 | 6.594 | 6.720 | +2.0% |

The SF10 fit is poor because SF10 takes the fast path (no Phase 3
gather). For SF50 and SF300 the linear model is within 4%; SF100
is within 15%. Reasonable fit for a radix sort.

## N log N model fit

```
t = -0.79e-9 · N·log2(N) + 27.9e-9 · N − 0.166 s
```

The N·log(N) coefficient comes out *negative*, meaning the data
prefers a sub-N·log(N) growth — consistent with radix sort being
O(N) not O(N log N).

The N log N model fits SF300 to 0.2% (perfect) but the negative
coefficient confirms O(N) is the correct asymptotic.

## Predictions for unmeasured scales

If single-GPU could hold the working set in HBM (it can't above
SF300):

| Scale | Records | Predicted wall | Predicted GB/s |
|-------|---------|-----------------|-----------------|
| SF500 | 3 B | 10.9 s | 33.0 |
| SF1000 | 6 B | 21.5 s | 33.5 |
| SF10000 | 60 B | 211 s (3.5 min) | 34.1 |

The predicted throughput plateaus at ~34 GB/s — close to the
measured SF300 peak of 32.7 GB/s. The fixed overhead becomes
negligible at scale.

## Why linear (not N log N)

CUB radix sort is O(N) per LSD pass, with k passes for k-byte
keys. So total = k · O(N) = O(N) since k is constant for our
fixed 32 B compact key.

The data confirms this empirically: doubling N doubles wall time
within 3-15%.

## Caveats

- 4 data points is very few for fitting. Want SF200 / SF400 to
  bracket SF300 better.
- SF10 is fast-path (no Phase 2 gather), so it's a different
  algorithm. Excluding SF10 might give a tighter fit.
- Predictions for SF500+ are extrapolation; the actual constraint
  is HBM capacity (single-GPU OOMs at SF500).
- For the actual SF1000 globally-sorted measurement (20m08s with
  partition + distributed sort), the linear extrapolation
  underestimates by 56× because the partition step + distributed
  overhead doesn't scale linearly with N.

## Comparison: distributed vs predicted single-GPU

| Scale | Predicted single-GPU (if HBM unlimited) | Actual distributed | Distributed overhead |
|-------|-----------------------------------------|---------------------|------------------------|
| SF500 | 10.9 s | 7m10s = 430 s | 39× slower |
| SF1000 | 21.5 s | 20m08s = 1208 s | 56× slower |

If HBM scaled proportionally to GPU count, we'd expect 4-GPU
distributed at SF1000 to be ~5 s ((6e9 records / 4 GPUs at 21.5s
single = 5.4s per GPU). Actual: 1208 s.

The 56× distributed overhead is from:
- 472 s partition (NVMe-bound at 1.5 GB/s)
- 5m49s + 6m04s pair sorts (NVMe write at 2 GB/s)
- All largely NVMe-bound, not GPU-bound

So for the paper: **distributed sort is NOT GPU-bound at SF1000;
it's NVMe-bound on the partition + write phases.**

## Implications

1. **The sort algorithm itself is well-engineered** (linear scaling,
   34 GB/s asymptotic).
2. **HBM capacity is the single-GPU ceiling**, not algorithm cost.
3. **Distributed sort is dominated by I/O**, not compute.
4. **Pre-partitioned input** would skip 39% of distributed wall.
