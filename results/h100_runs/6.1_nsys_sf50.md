# 6.1 — nsys profile of SF50 sort under numactl

**Date:** 2026-05-04
**Workload:** SF50 (36 GB) under `numactl --cpunodebind=0 --membind=0`,
`--runs 2 --no-verify`
**Status:** PASS — confirms gather is the dominant phase

## Aggregate timing (across both runs in the program)

### CUDA API calls (top 5)

| API | Total | Instances | Avg | Notes |
|-----|-------|-----------|-----|-------|
| cudaHostRegister | 1455 ms | 1 | 1455 ms | Once per program (mmap input pin) |
| cudaHostAlloc | 1268 ms | 4 | 317 ms | Pinned compact + perm + stage buffers |
| cudaHostUnregister | 743 ms | 1 | 743 ms | Once per program (cleanup) |
| cudaEventSynchronize | 467 ms | 26 | 18 ms | Stream syncs |
| cudaFreeHost | 429 ms | 4 | 107 ms | Cleanup |

The 1.5 + 1.3 + 0.7 ≈ **3.5 sec of CUDA host-memory ops are paid
once per program**, regardless of `--runs N`. With `--runs 5`, they
amortize to 0.7 sec per run; with `--runs 1`, they consume 100% of
the cold-run wall.

### GPU compute time (kernels, top 3)

| Kernel | Total | Instances | Avg/instance |
|--------|-------|-----------|---------------|
| cub DeviceRadixSortOnesweep | 244 ms | 128 | 1.9 ms |
| extract_uint64_from_compact | 94 ms | 8 | 11.8 ms |
| cub RadixSortHistogram | 9 ms | 8 | 1.1 ms |

Per-run GPU compute: ~175 ms (~12% of 1.4 s warm wall).

### PCIe transfers

| Direction | Total | Bytes | Effective GB/s |
|-----------|-------|-------|-----------------|
| H2D | 685 ms | 19.2 GB | 28 GB/s |
| D2H | 43 ms | 2.4 GB | 56 GB/s |

H2D effective is 28 GB/s — much lower than the 55 GB/s peak measured
in 6.3.1 because these are 20 fragmented transfers (chunked compact
upload at 960 MB each) instead of one large 8 GB transfer.

D2H is 56 GB/s — perm download is small (1.2 GB) and gets close
to peak.

## Per-warm-run breakdown (back-of-envelope)

For a 1414 ms warm SF50 run (median from 17.3.2.2):

| Phase | Time | % of wall |
|-------|------|-----------|
| GPU compute (CUB sort) | 175 ms | 12% |
| H2D (compact key upload) | 340 ms | 24% |
| D2H (perm download) | 22 ms | 2% |
| CPU compact extract | ~250 ms | 18% |
| CPU gather (random) | ~600 ms | 42% |
| Other (sync/alloc) | ~30 ms | 2% |

Gather is **42% of warm wall**, the largest single phase.

## Two big API costs that don't show up in warm runs

`cudaHostAlloc` + `cudaHostRegister` total **2.7 sec** for first run.
On `--runs 2`, this is amortized once. On `--runs 5`, it's ~0.5 sec
per run. The persistent buffers we already implemented (B1, 0.3.1)
are working — the alloc happens once.

For a one-shot production sort (`--runs 1`), this 2.7 sec is paid
on top of the 1.4 sec actual sort work, making the cold sort
**~4 sec** end-to-end. Numactl helps with the actual sort but
not with these one-time setup costs.

## Conclusions for next optimization

1. **Gather is still the bottleneck at 42% of warm wall.** The 17.3
   NUMA gain captured part of this; further gains need different
   techniques (e.g., GPU-resident gather, see follow-up work).

2. **PCIe is at 28 GB/s realized for chunked uploads** vs 55 GB/s
   for monolithic. Reducing chunk count (currently 20 transfers)
   or merging chunks could push the upload phase closer to peak.

3. **Cold-start setup is 2.7 sec** of cudaHost*. For one-shot
   sorts, this is the dominant cost. Persistent process or
   pre-allocated buffer pools would help.

## Build & run

```bash
nsys profile -t cuda,nvtx,osrt --output sf50_nsys --force-overwrite=true \
    --stats=true \
    numactl --cpunodebind=0 --membind=0 \
    ./external_sort_tpch_compact --input lineitem_sf50.bin --runs 2
```

Report at `/tmp/sf50_nsys.nsys-rep` (open with `nsys-ui` or analyze
the sqlite DB at `/tmp/sf50_nsys.sqlite`).

## Caveats

- This is a profiled run; nsys overhead may slightly perturb timings.
- `--runs 2` so values are aggregate across 1 cold + 1 warm. Per-warm
  run extraction needs nvtx ranges or the sqlite DB.
- Results are for SF50 (36 GB). SF100/300 should have similar shape
  but more time in gather (since gather scales linearly while CUB
  sort scales sub-linearly with data size).
