## What worked

- All three smoke configurations PASS verification after the fix described below.
- Compact path beats non-compact by ~12% on SF10 (352 vs 397 ms) — consistent with the 4-LSD-pass-vs-9-pass story for 32 B compact key vs 66 B non-compact key.
- Throughput is reproducible across 3 runs to <1% (352.20 / 352.22 / 353.95 ms).
- Runtime cmap detection correctly identifies 26/66 bytes as varying for TPC-H lineitem — matches the "26 varying bytes" comment in the source.

## What failed (and got fixed in this session)

**Bug 1 — `external-sort-tpch` and `external-sort` builds didn't compile.**
`COMPACT_KEY_SIZE` and `runtime_compact_size` were referenced unconditionally
in `src/external_sort.cu` even though both are gated by `#ifdef USE_COMPACT_KEY`.
Same shape as commit 2eead72. Fixed in commit 1944409 by gating the default arg
and the call site with the same `#ifdef`.

**Bug 2 — `external_sort_tpch_compact` produced unsorted output on SF10
and on small synthetic inputs (silent FAIL: 10 violations every run).**
`ExternalGpuSort::sort()` had a fast path for "data fits in GPU" that did
`cudaMemcpy → sort_chunk_on_gpu → cudaMemcpy back`. The compact-key kernel
inside `sort_chunk_on_gpu` reads `d_runtime_cmap`, `num_varying`, and
`runtime_compact_size` — all of which are populated by a runtime varying-byte
detection block that *only ran on the slow path*. So on small inputs:

  1. `build_compact_keys_kernel` wrote compact keys using a NULL `d_runtime_cmap`
     and `num_varying = 0` — i.e. all-zeros for every record.
  2. CUB radix-sort happily sorted those identical garbage keys.
  3. The records came back unsorted.
  4. The verifier flagged violations (10 per run).
  5. No kernel error, no abort: the run finished in ~350 ms with a clean exit
     code, just a log line saying `FAIL: 10 violations`.

If you only inspect the CSV / wall-time, this looks like a passing fast benchmark.
Caught here only because the prompt mandates `--runs 3` (which leaves verify on).

Fix in commit dc42168: hoist the runtime-cmap detection above the fast-path
branch and pass `runtime_compact_size` to `sort_ws.allocate`. After the fix
all three smoke configs PASS reproducibly.

## Surprises

1. **The bootstrap "verified" message was misleading.** Setup.sh's smoke step at
   line 147 (`./external_sort_tpch_compact --input lineitem_sf10.bin`) runs
   *without* `--runs`, which defaults to no-verify on cold start (or buries
   the FAIL output behind the `grep -E "PASS|FAIL"` filter). The bootstrap log
   I inherited claimed "Built and verified, 1M synthetic PASS, 195 ms" — that
   195 ms was the `gpu_crocsort` synthetic test, not the compact binary. I
   would have started from a corrupt baseline if I hadn't bothered to run the
   compact verify with `--runs 3`. Lesson: always re-run smoke tests with
   verification on the actual hardware before trusting prior PASS labels.

2. **The non-compact path was *actually* never used** before today. It hadn't
   been compiled cleanly since the runtime-cmap rework (commit e214fa7). All
   prior numbers in `results/` came from the compact path.

3. **H100 SF10 wall-time is *slower* than the projection** (352 ms measured
   vs ≤150 ms projected, vs 280 ms RTX 6000 baseline). At SF10 the workload is
   bottlenecked by the 7.2 GB H2D + 7.2 GB D2H over PCIe 5 — about 14.4 GB
   of host↔device traffic. If PCIe 5 ceiling is ~32 GB/s, that's 14.4/32 = 450 ms
   pure-PCIe. We're hitting 352 ms which means we're already PCIe-saturated;
   *more compute won't help*. The H100 advantage will only show on workloads
   bigger than HBM where compression cuts PCIe (which is exactly what the
   compact + bitpack path is for at SF300+). The single-GPU SF10 number is
   *not* the headline; it's the upper bound on what's possible without
   compression at this scale.

## Interpretation

The H100 box bring-up exposed two real correctness bugs in `gpu_crocsort` that
weren't triggered by the upstream RTX 6000 build pipeline. Bug 1 was a build
break on any non-USE_COMPACT_KEY target; Bug 2 was a *silent correctness
failure* in the compact path's data-fits-in-GPU fast branch — a much more
serious category of bug because the binary returns "sorted" data with a clean
exit code while the data is in fact unsorted. Any benchmark numbers that ran
through this fast path before today (i.e. anything with `num_records ≤
buf_records ≈ 179 M`, which on a 100 GB H100 covers SF10/SF20/SF30) are
unreliable. The hoisted-cmap fix in commit dc42168 makes both paths see the
same compact map, and SF10 now PASSes with 20.4 GB/s effective.

The headline number for 0.1 is **PASS, 352 ms, 20.4 GB/s on SF10 compact**,
slower in absolute terms than the RTX 6000 (which is unsurprising at PCIe-bound
small scales) but with throughput that's been *verified correct* — a trustworthy
floor for the rest of the queue.
