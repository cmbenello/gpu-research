# 19.19 — SF1500 K=16 + 4-GPU, no output write: 25m20s [upper bound for perm-only]

**Date:** 2026-05-05
**Result:** **25m20s, 16/16 PASS** — 19% faster than 31m07s with full output.
**Status:** Confirmed sort phase = 6m05s (vs 12m04s with full output).

## Why this matters

The output write of 1.08 TB of sorted records is **44% of every round's
wall time** (per 19.18 attribution: 64 sec / 145 sec). Eliminating the
write drops sort phase from 12m04s → 6m05s.

**This is the upper bound for any output-reducing strategy** (perm-only,
projection, compressed). A real perm-only output (36 GB total) would land
slightly above this — projected 25m30s to 26m total.

## Pipeline

```
03:44:47 — start
       partition K=16 — 16 buckets × ~67 GB
04:01:52 — partitioned (17m05s)
       evict + setup ~2m
04:04:02 — sort phase begins (4-GPU concurrent × 4 rounds, no --output-file)
04:05:34 — round 1 done (1m32s, vs 2m59s with write)
04:07:04 — round 2 done (1m30s)
04:08:35 — round 3 done (1m31s)
04:10:07 — round 4 done (1m32s) + verification + cleanup
========
TOTAL: 25m20s, 16/16 PASS
Sort phase: 6m05s (365 sec)
```

## Round-time comparison

| Variant            | R1   | R2   | R3   | R4   | Total sort | Total wall |
|--------------------|------|------|------|------|------------|------------|
| 19.16 with write   | 2m59 | 3m03 | 3m01 | 3m01 | 12m04s     | 31m07s     |
| **19.19 no write** | **1m32** | **1m30** | **1m31** | **1m32** | **6m05s** | **25m20s** |

Each round saves ~88 sec by skipping output write. Matches 19.18's per-
round attribution (64 sec write cost in single-round test) — the slightly
higher savings here reflect that with writes, the NVMe drive is also
contending with reads of the next round's bucket file.

## Implications for the design

**The 31m07s ceiling was: NVMe at 1.3 GB/s write × 2.16 TB written = 28m
sequential floor.**

Eliminating the output write reduces NVMe traffic from 2.16 TB to 1.08 TB
(only intermediate buckets written), giving:
- Partition: 1.08 TB at 1.3 GB/s write = 14m floor
- Sort: 1.08 TB read at 3.1 GB/s = 5.8m floor (write is now negligible)
- **Total: ~20m theoretical floor.** We measure 25m, so 5m of overhead.

## What "no output write" produces

Each per-bucket sort STILL produces sorted records in pinned RAM (the
internal h_output buffer). They're just discarded at end-of-process. So
this is a measurement run, not a usable sort.

A **usable** sort with similar speedup would output the **sorted
permutation** (4-byte indices into input.bin) instead of the full
records. 36 GB perm vs 1.08 TB records — write cost negligible.

## Caveats / what perm-only would lose

1. Downstream consumer must do the gather (random reads from input.bin)
   to materialize sorted records. For TPC-H Q1-style aggregations that
   already touch every record, the gather is "free."

2. Verification must be modified to compare via perm (read keys at
   `input[perm[i]]` and check ordered).

## Next step

**Implement PERM_ONLY=1 env in external_sort.cu** to skip the CPU gather
phase and return sorted_output = h_perm. Then re-run 19.19 with PERM_ONLY=1
to confirm the ~25-26m projection.

## Caveat to the headline

The 31m07s is still the headline for "full sorted output, single SSD."
But with this attribution, the paper now has TWO results:
- **Full sorted output: 31m07s** — write-bound (1.3 GB/s SSD limit)
- **Perm-only output: ~25m** — read-bound (validates the SSD ceiling argument)

Both are useful: perm-only matches what a database engine internally uses;
full output matches batch pipeline output.
