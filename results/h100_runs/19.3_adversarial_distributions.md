# 19.3 — Adversarial distribution stress (--runs 5 stable medians)

**Date:** 2026-05-04
**Workload:** SF50-equivalent (300 M synthetic 120 B records, ~36 GB)
**Status:** PASS — 1.65× spread, all distributions correct

This extends 18.6 with proper 5-run statistics + adversarial distributions.

## Result (median of 4 warm runs each)

| Distribution | Median warm (ms) | GB/s | Phase 1 | Phase 2 | StdDev |
|--------------|-------------------|------|----------|----------|---------|
| **sorted** (ascending) | **1170.8** | **30.7** | 681 | 489 | 2.4 |
| **reversed** (descending) | 1184.6 | 30.4 | 680 | 504 | 1.0 |
| all_equal (single key) | 1521.6 | 23.6 | 707 | 814 | 7.4 |
| zipfian (skewed) | 1925.6 | 18.7 | 1211 | 715 | 1.7 |
| **random** (uniform 64-bit) | **1929.4** | **18.7** | 1212 | 715 | 4.4 |

For comparison, TPC-H SF50 lineitem (natural distribution):
**1413 ms / 25.5 GB/s** — falls between sorted and random.

## Spread analysis

- Best/worst ratio: 30.7/18.7 = **1.65×**
- Variance is sub-1% within each distribution (StdDev / median 0.1-0.5%)
- The spread is driven by the runtime adaptive compact-key encoding:
  sorted/reversed yield 9.6 GB H2D, random/zipfian yield 14.4 GB H2D
  (50% more PCIe traffic).

## All-equal: a "soft adversarial" case

All-equal records (all keys identical) trigger the Phase 4 fixup
on every record. Phase 2 wall = 814 ms vs ~490 for sorted.
Net: 23.6 GB/s vs 30.7 GB/s — 23% slower, not catastrophic.

The OVC merge architecture handles ties correctly via the explicit
fixup pass. No silent corruption; verified output is correctly
sorted (unchanged input is the sort).

## Why our zipfian ≈ random

The current zipfian generator picks 50% of records from a "popular"
set of 1M keys uniformly. The compact map sees full 64-bit entropy
across both the popular and tail sets, so the compact key is the
same size as random. A true Zipf-α distribution would have less
entropy in the popular set; not currently tested.

## Implications for paper

### Strong robustness claim

gpu_crocsort handles all 5 distributions correctly. **No
distribution-induced failure.** The ~1.65× spread between best and
worst is from *adaptive PCIe compression*, not algorithmic instability.

### Why this matters for SOTA comparison

RMG Sort (BTW 2023) tested only Zipf 1.0 (pop=1, decay=1) and reported
slowdowns under skew. We observe distribution-dependent throughput
but it's a clean 1.65× range — no "skew collapse" failure mode.

### TPC-H is not worst-case

Natural TPC-H lineitem (25.5 GB/s) is closer to sorted than to random
because TPC-H has structured low-entropy fields (l_returnflag has 3
values, l_linestatus has 2). Real-world database workloads with
similar structure will fall in this range.

Worst-case for our algorithm: **uniform random 64-bit keys = 18.7
GB/s.** Best-case: **sorted = 30.7 GB/s.**

## What real-world distributions look like

Typical database tables have:
- Low-cardinality columns (status, category): ~sorted-like behavior
- Timestamp columns: nearly sorted
- Random IDs (UUIDs): random-like behavior
- Foreign keys: typically Zipfian

A composite key (TPC-H lineitem sort key = 13 columns) blends these.
The runtime compact map finds the bytes that matter and ships only
those. **This is the contribution.**

## Build & run

```bash
cd ~/gpu-research/gpu_crocsort
g++ -O3 -std=c++17 -pthread experiments/gen_synthetic.cpp \
    -o experiments/gen_synthetic

for D in random sorted reversed all_equal zipfian; do
    ./experiments/gen_synthetic --records 300000000 \
        --output /mnt/data/synth_${D}_300M.bin --distribution $D
done

for D in random sorted reversed all_equal zipfian; do
    CUDA_VISIBLE_DEVICES=0 numactl --cpunodebind=0 --preferred=0 \
        ./external_sort_tpch_compact \
        --input /mnt/data/synth_${D}_300M.bin --runs 5
done
```

## Caveats

- Synthetic data only at SF50-equivalent scale.
- Zipfian generator uses 50/50 split, not true Zipf-α weighted.
- Did not test column-collapsed distributions (e.g., single shipdate);
  would need a real TPC-H generator with config knobs.
