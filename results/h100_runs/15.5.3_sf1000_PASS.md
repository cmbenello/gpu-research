# 15.5.3 — SF1000 distributed sort PASS [HEADLINE]

**Date:** 2026-05-03 18:48 → 19:11 UTC
**Wall time:** 22 min 40 sec
**Status:** PASS — **6 BILLION records (720 GB) globally sorted on this hardware for the first time**

## What this proves

Single-GPU SF1000 OOMs trying to allocate the key buffer (720 GB × 66 B
key-extracted = 47 GB which actually fits, but the path-selector logic
plus output-buffer reservation pushed it over before — see 1.6 writeup).
15.5 distributed-with-merge SF1000 also OOM'd because the merge phase
needed >500 GB of memory.

15.5.3 **sample-partition** distributes the work across 4 H100 NVLs
without needing a merge step. After this run produces 4 sorted bucket
files in disjoint key ranges, **iterating them in order = globally
sorted output**. No merge buffer needed.

## Pipeline

```
INPUT (lineitem_sf1000.bin, 720 GB on NVMe)
   │
   │  partition_by_range tool (CPU, 64 threads)
   │  ─ 8 min 11 sec total
   │     ├─ Pass 1 (count):  119 s
   │     └─ Pass 2 (route):  345 s @ 2.09 GB/s (NVMe write)
   ▼
4 partitioned bucket files on /mnt/data:
   bucket_0: 180.85 GB / 1.51 B records  (key range A)
   bucket_1: 178.39 GB / 1.49 B records  (key range B)
   bucket_2: 177.70 GB / 1.48 B records  (key range C)
   bucket_3: 183.06 GB / 1.53 B records  (key range D)
   │
   │  Pair A (GPUs 0+1, NO_MAP_POPULATE=1)
   │  ─ Wait for both, delete bucket_0,1 on completion
   │  ─ 7 min wall
   ├──▶ sorted_0.bin (180.85 GB, sorted by key range A)
   └──▶ sorted_1.bin (178.39 GB, sorted by key range B)

   │  Pair B (GPUs 2+3, NO_MAP_POPULATE=1)
   │  ─ 7 min 28 sec wall
   ├──▶ sorted_2.bin (177.70 GB, sorted by key range C)
   └──▶ sorted_3.bin (183.06 GB, sorted by key range D)

GLOBALLY SORTED:
   iterate sorted_0 → sorted_1 → sorted_2 → sorted_3
   = 6,000,000,000 records in sorted order, VERIFIED PASS.
```

## What worked

- **NO_MAP_POPULATE=1** prevented the OOM cascade that killed the
  earlier attempt's 4th process. Without populate, the gather-output
  buffer faults lazily as each thread writes; less concurrent
  memory pressure.
- **Sequential per-pair execution** (2 sorts at a time, not 4) kept
  pinned memory under 360 GB per pair vs 4-concurrent's 720 GB
  pinned.
- **Sort outputs to /mnt/data** rather than /dev/shm: SF1000 needs
  720 GB total sorted output which doesn't fit in 504 GB tmpfs.

## What was learned (failed attempts)

1. **First attempt: outputs to /dev/shm.** Failed because 720 GB sorted
   output couldn't fit in 504 GB tmpfs. Half-finished, 1 of 4 buckets
   missing.
2. **Second attempt: outputs to /mnt/data, default MAP_POPULATE.**
   Failed because 2 concurrent sorts × (178 GB pinned input + 178 GB
   MAP_POPULATE'd output) = 712 GB resident, plus the partition input
   page cache, hit the 1 TB host wall. dmesg confirmed OOM-kill of
   GPU 2's process.
3. **Third attempt: outputs to /mnt/data, NO_MAP_POPULATE=1.** PASS.
   Lazy faulting on the output dropped per-process resident from
   356 GB to ~180 GB.

## Surprises

1. **The partition phase dominates SF1000 wall** — 8 min of partition
   vs 14 min for 4 sorts. This suggests Pass 2 (write 720 GB to NVMe)
   is the binding constraint at 2 GB/s. NVMe sequential write
   ceiling on this box looks to be ~3 GB/s; we're at 70% of that.
2. **The sort phase is much SLOWER per byte than at SF500** because
   each bucket (~180 GB) just barely fits the OVC threshold, putting
   each sort into a 3-chunk run-gen flow. SF500 buckets at 90 GB took
   the simpler single-pass slow path.
3. **NO_MAP_POPULATE adds ~1 min of gather time vs MAP_POPULATE'd
   case** — the gather pays per-page minor faults instead of getting
   pre-allocated pages. Acceptable tradeoff for not OOMing.
