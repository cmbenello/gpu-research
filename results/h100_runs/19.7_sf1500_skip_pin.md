**Date:** 2026-05-04
**Goal:** Validate the 19.5 prediction (31% SF1500 win from skipping pin)
**Status:** NEGATIVE — pin and unpinned are within 1.5% at SF1500

## Why the prediction failed

The 19.5 microbench measured cudaHostRegister at 1.2 GB/s on warm-cache
data and predicted ~225 sec savings per bucket at SF1500. Reality:

**The pin cost includes the NVMe page-fault cost.**

When `cudaHostRegister` is called on a freshly-mmap'd file, it walks
each page; pages not in OS page cache get faulted from NVMe at
~6.2 GB/s. So the 1.2 GB/s "pin rate" includes both:
- Actual page-locking syscall overhead (~30 GB/s if pages are warm)
- NVMe page faults (~6.2 GB/s for cold pages)

Effective combined: ~1.2 GB/s.

When `SKIP_HOST_PIN=1`, the explicit pin step is skipped, but the
data still has to come from NVMe — just deferred to the per-cudaMemcpy
step. cudaMemcpy from non-pinned mmap reads via OS page cache, which
must also fault from NVMe.

Net: same total NVMe I/O, just shifted between "pin" and "transfer"
columns.

## Where the 19.5 prediction was right

The 19.5 microbench correctly showed:
- WARM-cache: cudaMemcpy from non-pinned is ~10 GB/s vs pin+transfer
  at 55 GB/s
- For repeated transfers of the same data, pin amortizes

But it MISSED that:
- Cold cache pin includes the NVMe read implicit in the page-fault
- Cold cache unpinned ALSO includes the NVMe read in cudaMemcpy
- The two converge at NVMe peak

So **the 19.6 SF300 finding (2.9% warm wall improvement)** is the
real, robust SKIP_HOST_PIN benefit:
- Warm runs amortize cold pin
- SKIP_HOST_PIN avoids per-warm-run page lock overhead

Cold runs are NOT improved by SKIP_HOST_PIN at SF1500 scale.

## Useful negative finding for paper

This is a clean negative result that adds rigor:
1. The cudaHostRegister cost is NOT a pure software overhead.
2. It includes NVMe-bound page-fault cost that you can't skip.
3. The architectural finding from 19.5 applies only to:
   - Warm-cache regimes (multiple sorts of same data)
   - Pre-faulted memory (pre-touch input before timing)

For SF1500 one-shot cold-cache: pin and unpinned are ~equivalent.

## What would actually improve SF1500 cold

To beat the 49m15s baseline:
- **Faster NVMe** (RAID0, NVMe-of-Fabric)
- **GPUDirect Storage** — GPU reads directly, no host RAM intermediate
- **Pre-touch input** before timing, then SKIP_HOST_PIN
- **Streaming sort** that overlaps NVMe I/O with GPU compute

None of these are 1-line changes; all are paper-2 territory.

## Updated SKIP_HOST_PIN deployment recipe

| Workload | SKIP_HOST_PIN benefit |
|----------|------------------------|
| Single warm sort (the SF300 case) | +3% wall reduction |
| Multiple warm sorts (--runs 5+) | Same +3% |
| **Cold one-shot (SF1500-class)** | **0% (within noise)** |

## Build & run

```bash
SKIP_HOST_PIN=1 ./external_sort_tpch_compact ...
```

Use it. It either helps or is neutral; never hurts (within noise).

## Caveats

- Single sweep at SF1500. ~50 min runs make multi-sweep expensive.
- The 1.5% slowdown is probably within run-to-run variance.
- Bucket 2 was particularly slow (54s vs 43s baseline) — possibly
  the OS scheduler hit different state. Other 3 buckets were
  comparable.

## Conclusion for paper

**Two-claim story:**
1. SKIP_HOST_PIN is a real ~3% wall improvement on warm sorts (SF300).
2. SKIP_HOST_PIN is **neutral** on cold one-shot sorts (SF1500) —
   the cudaHostRegister cost is dominated by NVMe page-fault
   I/O which can't be skipped.

The 19.5 microbench prediction was right about the algorithm
(unpinned cudaMemcpy < pin amortization at low transfer count) but
missed that NVMe I/O dominates at cold scale.
