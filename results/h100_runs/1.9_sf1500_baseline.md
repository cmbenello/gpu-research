# 1.9 — TPC-H SF1500 sort (FAIL: kernel-killed for host OOM)

**Date:** 2026-05-03 01:02 UTC
**GPU(s):** 1× NVIDIA H100 NVL (CUDA_VISIBLE_DEVICES=0)
**Scale:** TPC-H SF1500 (1080 GB, 9,000,021,272 × 120-byte records)
**Wall time:** ~3 min before SIGKILL during input-pin phase
**Status:** **FAIL** — Linux OOM killer SIGKILLs the process at 1.05 TB
resident memory (process tried 1067 GB virtual; 1024 GB host RAM total).

## Command(s)

```bash
source ~/gpu-research/h100/env.sh
cd ~/gpu-research/gpu_crocsort
CUDA_VISIBLE_DEVICES=0 ./external_sort_tpch_compact \
    --input /mnt/data/lineitem_sf1500.bin --runs 1
```

## Raw numbers

```
$ tail -25 /tmp/1.9_sf1500.log
(empty — no stdout produced before SIGKILL)
$ exit 137  ← SIGKILL

$ dmesg | grep "Out of memory"
[22815.984163] Out of memory: Killed process 41940 (external_sort_t)
    total-vm:1067738568kB, anon-rss:1049343016kB,
    file-rss:2240kB, shmem-rss:0kB
```

So the binary touched **1067 GB of virtual address space** and reached
**1049 GB of resident memory** before the kernel killed it. The host has
1024 GB total — the process couldn't possibly fit. The "anon-rss" 1049 GB
is the pin-mapped input (1080 GB) being faulted in faster than the page
cache can be reclaimed.

## What worked

The OOM-kill was clean — no zombie, GPU freed within seconds, the file
on disk wasn't corrupted. Subsequent sort attempts on smaller scales
work normally.

## What failed

Mapping the 1080 GB input via mmap + cudaHostRegister couldn't even
finish, let alone the sort. The *input alone* exceeds host RAM if you
count the registered pinned pages plus any temporary allocations.

## Surprises

1. **The OOM killed the process during *input pin*, not during the sort
   itself.** Earlier I expected the sort to OOM at the GPU side (like
   SF500). Reality: the binary never even reached the GPU — the host pin
   for a 1080 GB input on a 1024 GB box is impossible.

2. **The 1.6.1 lazy-h_pin fix didn't help here.** That fix avoided one
   triple-buffer pin in the constructor; it doesn't change that the
   *input itself* needs 1080 GB pinned. SF1500 is out of single-host
   envelope, period.

3. **SF1500 is the inflection where multi-host becomes necessary, not
   multi-GPU.** Even if all 4 H100 NVLs aggregate to 376 GB HBM, the
   *host* still has to hold the 1080 GB input. Either need a >1 TB host
   or stream the input from NVMe in chunks (which kills throughput at
   2-3 GB/s read).

## Interpretation

SF1500 single-host is **infeasible** at 1024 GB host RAM with the current
"pin entire input then sort" architecture. To sort SF1500 on this box
we need either:

- (a) **Streaming input**: read input chunks from NVMe rather than pinning
  the whole thing. Bounded by NVMe (~3 GB/s) so adds ~360 s to wall time.
- (b) **A >1.5 TB host**: not available here.
- (c) **Multi-host clusters**: out of scope.

(a) is the natural path forward but requires a substantial code change
to the run_generation.cu pipeline. Filed as 1.9.1 below.

## Follow-ups

- **1.9.1 (new) — streaming-input variant of `external_sort_tpch_compact`.**
  Read SF1500 input in 200-300 GB chunks from NVMe + sort each + merge.
  Wall time floor at NVMe read: 1080 / 3 = 360 s (vs current 9 s for SF300).
  Throughput drops dramatically but it's the only single-host path.
- **1.9.2 (new) — partial-pin mode.** mmap 1080 GB but only pin what's
  needed for the active chunk. Trade some PCIe overhead for not OOMing.
- 1.10 (next) is gen SF2000, which will hit the disk cap (need 1.44 TB
  binary + ~530 GB parquet on top of the 2.3 TB already used; won't fit
  in 3.5 TB without deleting smaller scales).
