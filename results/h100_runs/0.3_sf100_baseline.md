## What worked

- Verification PASS on all 10 runs (compact path post-fix).
- Wall time stable to ~3% across warm runs.
- `Runtime compact map: 27/66 bytes vary` — one *more* varying byte than SF50
  (which had 26/66). Probably an `l_orderkey` byte that didn't vary across
  the smaller SF50 sample but does at SF100 — confirms the cmap detection
  is doing something useful instead of using a static map.

## What failed

Nothing failed verification. But the *throughput* is slower than expected
(see Surprises).

## Surprises

1. **H100 SF100 (4.50 s best) is *slower* than RTX 6000 SF100 (3.74 s).**
   This is not the result the H100 prep doc was predicting. Three plausible
   explanations:

     a. **The slow-path upload sends full 66-byte keys, not 32-byte compact
     keys.** PCIe H2D traffic = 39.60 GB = 66 B × 600M records exactly. The
     "compact key" optimization only kicks in for the *sort* on the GPU
     (4 LSD passes on 32 B compact key) but the upload doesn't take advantage.
     Bitpack reports 24 B compact keys but H2D is still 39.60 GB. So bitpack
     can't help wall time on this code path.

     b. The RTX 6000 baseline of 3.74 s might have been cached / inflated.
     Need to re-run on the original RTX 6000 to confirm before claiming H100
     is "slower".

     c. The PCIe 5 of H100 is being used at ~25 GB/s, not the 32 GB/s ceiling.
     Could be due to triple-buffer chunking or non-pinned-memory transfers
     on this build.

2. **USE_BITPACK does nothing useful at SF100 either.** Same story as 0.2 —
   the merge phase savings (one fewer LSD pass) are cancelled by the CPU
   prepack overhead. At SF100, neither term grows fast enough to break the
   tie. The prepack is also a serial CPU pass over 600 M records.

3. **27 varying bytes at SF100 vs 26 at SF50.** The runtime cmap is
   scale-aware in the sense that more data exposes more variation. For a
   "stable" production system this could be a problem (a small sample might
   miss a varying byte and produce wrong sort order). The fix in 0.1 scans
   all records, not a sample, so we're safe — but worth flagging that the
   cmap can change between datasets even from the same TPC-H scale family.

4. **Baseline run 2 was the slowest of all 5 (5379 ms).** Doesn't fit the
   "first warm run is consistent with rest" story. Hard to explain without
   profiling. Could be NUMA migration of pinned memory mid-run, or page
   cache eviction since the input is 72 GB and we have ~1 TB RAM but with
   another process running gen jobs in the background. Worth a 6.x profile.

## Interpretation

The headline SF100 number on a single H100 NVL is **4.5-4.8 s, 15.0 GB/s
effective wall-time throughput**. At this configuration, the H100 does not
beat the RTX 6000 — *not* because the H100 is slow, but because the current
slow-path code uploads the full 66-byte key per record, defeating the entire
point of the compact-key encoding for PCIe. The actual GPU sort phase
(merge) is well under 2 seconds; the upload is 2.8 s. Compress the upload
to 32 B (or 24 B with bitpack) and we should see ~30-50% wall-time
improvement, putting SF100 at ~3.0 s and finally beating RTX 6000 — and
opening room for the bitpack optimization to actually pay off.

This is the most actionable finding so far this week: **the h100/queue
will not produce a "win" against RTX 6000 unless we fix the slow-path
upload to send compact keys, not full keys.** Filed as 0.3.1 below.

## Follow-ups

- **0.3.1 (new) — slow-path upload should send compact keys, not 66-byte
  raw keys.** Currently `external_sort.cu` slow path extracts compact keys
  on CPU into a host buffer but uploads the *full* key bytes. Trace where
  39.60 GB of H2D comes from and switch to uploading the 19.2-32 GB compact
  buffer instead. Should give 1.5-2× wall-time improvement at SF50/SF100/SF300.
- **0.3.2 (new) — re-run RTX 6000 SF100 to confirm 3.74 s baseline.** If it
  was warm-cached or single-run, the comparison is unfair to H100.
- **0.3.3 (new) — profile run-2 anomaly.** Use `nvprof` / Nsight Systems on
  a 5-run loop and inspect what's different in the second iteration. Could
  be allocator behavior, CUDA context warmup, NUMA, or page cache.

  ≈ 23 min). Should fit comfortably (53 GB used, 3.3 TB free).
