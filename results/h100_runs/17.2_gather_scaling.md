# 17.2 — Gather scaling: thread count vs working-set size

**Date:** 2026-05-03
**Hardware:** Xeon Platinum 8468, 96 cores × 2 NUMA nodes (192 logical)
**Workload:** Random permutation gather of 120 B records, anonymous
mmap with `MADV_HUGEPAGE`, prefetch BLOCK=256
**Status:** PASS — production code's 64-thread default is suboptimal at SF100+

## The non-monotonic curve

Three different working-set scales produce three different optimal
thread counts:

- **12 GB fits partially in L3** (110 MB shared per socket): more
  threads helps because cache hits dominate. Peak at **96 threads**.
- **72 GB busts L3 entirely**: TLB pressure + memory channel
  contention cap us at **32 threads**. More threads regress.
- **216 GB is fully DRAM-bound**: enough memory pressure that
  having lots of threads in flight hides DRAM latency. Peak at
  **128 threads**.

The reversal between SF100 (32 peak) and SF300 (128 peak) is the
key finding. At 72 GB, threads compete for L3 capacity and TLB
slots, so contention dominates. At 216 GB, no thread is in cache
anyway, so concurrency lets the memory controller queue more
in-flight reads. Throughput becomes purely "how many random reads
can I keep pending."

→ **Production guidance: pick the thread count by working-set size**:
- < 16 GB: 96 threads
- 16-128 GB: 32 threads
- > 128 GB: 96-128 threads

## Production validation: SF50 sweep with `GATHER_THREADS`

Single warm run (run 2 of 2) at SF50 with the real sort:

| GATHER_THREADS | Wall (ms) | Throughput (GB/s) |
|----------------|-----------|-------------------|
| 32  | 2914 | 12.4 |
| 48  | **1333 ← fastest single run** | 27.0 |
| 64 (default) | 2300 | 15.6 |
| 96  | 2283 | 15.8 |
| 128 | 1927 | 18.7 |

**Variance is huge** (32-thread: 2914 ms, 48-thread: 1333 ms). The
prior 1.51 s headline was *best-of-5 warm*; my single-warm runs span
1333-2914 ms, so I can't conclude a robust default change from this
data.

→ **Decision: keep default at 64, keep env override for tuning.**
The synthetic gather_bench scaling curve is not directly predictive
of production wall-time variance. Real sort gather has additional
factors:
- Permutation correlation (post-CUB perm is partially-sorted)
- Cache state from prior phases (compact extract pre-warms input)
- Concurrent allocation/free traffic during gather

## Production-code action items

The current production gather defaults are:
- `external_sort.cu:1403` → 48 (multi-run OVC gather path)
- `external_sort.cu:1922` → 48 (sample-sort gather path)
- `external_sort.cu:2930` → 64 (compact-key path with `GATHER_THREADS` env override)

Across SF100 and SF300:
| Threads | SF100 | SF300 | average |
|---------|-------|-------|---------|
| 32      | 41.1  | 35.7  | 38.4 |
| 48      | 39.3  | 46.1  | 42.7 |
| 64 (default) | 36.4 | 44.9 | 40.7 |
| 96      | 37.2  | 54.1  | 45.6 ← best avg |
| 128     | 28.8  | 55.9  | 42.4 |

→ **Best single static default across SF100/SF300: 96 threads** (avg
46 GB/s vs 64-thread default 41 GB/s, a +12% gain at large scales).

→ Better still: adaptive — pick by working-set size:
- < 16 GB: 96 threads
- 16-128 GB: 32 threads
- > 128 GB: 96-128 threads

Adaptive picks would yield 65 / 41 / 56 GB/s = avg 54 GB/s, vs
static 64-thread = 41 GB/s = +32% on average. But production sorts
are mostly in the > 128 GB regime, so the static-96 default
captures most of the win without code complexity.

## Why production gather measures 44 GB/s (real) but bench shows 41 GB/s peak

In production, the gather pulls from a **already partially-sorted**
permutation (post-CUB), which has more spatial locality than my
**fully random** permutation. The benchmark is a worst-case lower
bound. The real gather is faster because the perm correlates with
sort key, not entirely random.

The point still stands: 32 threads is the right peak shape. The
*absolute* number changes with the perm distribution.

## Caveats

- This is a single-pass gather. A two-pass gather (sort perm by source
  position first) was tried earlier and was **2-4× SLOWER** due to TLB
  thrash on the 216 GB write target — see prior session notes.
- The 12 GB curve uses anonymous mmap, not file-backed. Production
  uses file-backed mmap of the input lineitem.bin which has different
  page-cache behavior.
- All measurements are with `MADV_HUGEPAGE` advisory; actual huge-page
  promotion depends on kernel state (not always granted).

