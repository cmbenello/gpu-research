# 17.2 — Gather scaling: thread count vs working-set size

**Date:** 2026-05-03
**Hardware:** Xeon Platinum 8468, 96 cores × 2 NUMA nodes (192 logical)
**Workload:** Random permutation gather of 120 B records, anonymous
mmap with `MADV_HUGEPAGE`, prefetch BLOCK=256
**Status:** PASS — production code's 64-thread default is suboptimal at SF100+

## Findings

| Threads | 12 GB (SF12) | 72 GB (SF100) |
|---------|--------------|----------------|
| 1       | 1.1 GB/s | 1.4 GB/s |
| 2       | 2.7      | 2.3      |
| 4       | 5.8      | 4.1      |
| 8       | 11.9     | 9.3      |
| 16      | 25.3     | 21.6     |
| **32**  | 42.3     | **41.1 ← peak** |
| 48      | 40.5     | 39.3     |
| 64      | 51.7 (default) | 36.4     |
| **96**  | **65.4 ← peak** | 37.2     |
| 128     | 55.8     | 28.8     |

## The cliff: small vs large working set

- **12 GB fits in L3 (× 96 cores ÷ 192 logical = 1 MB per logical CPU
  L3 + 110 MB shared)**: more threads helps because cache hits dominate.
  Peak at 96 threads.
- **72 GB busts L3 entirely**: TLB pressure + memory channel contention
  cap us at 32 threads. More threads compete for the same DDR5
  channels. Peak at 32 threads, then *regressing* by 30% at 128.

This is the first-principles reason: **random gather is bandwidth +
TLB-bound, not compute-bound**. Throwing more threads beyond what
saturates the memory system just adds contention.

## Production-code action items

The current production gather defaults are:
- `external_sort.cu:1403` → 48 (multi-run OVC gather path)
- `external_sort.cu:1922` → 48 (sample-sort gather path)
- `external_sort.cu:2930` → 64 (compact-key path with `GATHER_THREADS` env override)

The SF100 sweep shows:
- 64 threads: 36.4 GB/s
- 32 threads: 41.1 GB/s (+13%)
- 48 threads: 39.3 GB/s (+8%)

→ **Recommendation: change the default to 32 threads at SF100+, keep
the env override.** Smaller workloads like SF12 still benefit from
more threads, but production sorts are SF100+.

→ Better still: pick the thread count adaptively based on input
size. Threshold: working set < 16 GB → 96 threads; ≥ 16 GB → 32
threads.

## Why production gather measures 44 GB/s (real) but bench shows 41 GB/s peak

In production, the gather pulls from a **already partially-sorted**
permutation (post-CUB), which has more spatial locality than my
**fully random** permutation. The benchmark is a worst-case lower
bound. The real gather is faster because the perm correlates with
sort key, not entirely random.

The point still stands: 32 threads is the right peak shape. The
*absolute* number changes with the perm distribution.

## Caveats

- This is a single-pass gather. A two-pass gather (sort perm by source
  position first) was tried earlier and was **2-4× SLOWER** due to TLB
  thrash on the 216 GB write target — see prior session notes.
- The 12 GB curve uses anonymous mmap, not file-backed. Production
  uses file-backed mmap of the input lineitem.bin which has different
  page-cache behavior.
- All measurements are with `MADV_HUGEPAGE` advisory; actual huge-page
  promotion depends on kernel state (not always granted).

## Build & run

```bash
g++ -O3 -std=c++17 -pthread -fopenmp \
    experiments/gather_bench.cpp -o experiments/gather_bench
./experiments/gather_bench --records 600000000 > /tmp/sf100.csv
./experiments/gather_bench --records 100000000 > /tmp/12gb.csv
./experiments/gather_bench --records 1800000000 > /tmp/sf300.csv  # 216 GB
```
