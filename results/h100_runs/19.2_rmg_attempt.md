# 19.2 — RMG Sort comparison ATTEMPT (blocked by build env)

**Date:** 2026-05-04
**Goal:** Direct head-to-head vs RMG Sort (BTW 2023, hpides/rmg-sort)
**Status:** BLOCKED — build environment mismatch

## What was attempted

1. Cloned `github.com/hpides/rmg-sort` to /tmp/rmg-sort.
2. Initialized submodules (Thrust 1.17, simple_paradis).
3. Patched `CMakeLists.txt` to add `CUDA_ARCHITECTURES 90 80 72 70`
   for H100 sm_90.
4. Installed cmake via `pip install --user cmake`.
5. Configured: cmake -DCMAKE_BUILD_TYPE=Release succeeded.
6. Build: failed with two issues:
   - `gcc: error: SHELL:-fopenmp: No such file or directory`
     (CMake's OpenMP detection generates a SHELL: prefix that the gcc
     here doesn't recognize)
   - `nvcc -std=c++20` → conflicts with system gcc 9's c++17 stdlib
     (errors in `<iomanip>`, `<thread>` involving `nanosleep`).

## Why blocked here

The system has gcc 9.4 + CUDA 12.4. RMG requires C++20 features
that need gcc 10+. Upgrading gcc would require sudo (this is a
shared research box).

Direct workarounds:
- Build inside a Docker container with gcc 11+. Container support
  exists but takes time to set up.
- Manually rewrite the affected sources to drop C++20 features.
- Wait for an updated RMG fork that supports gcc 9.

## What we already have for comparison

The closest equivalent we already ran is **4.4 raw CUB sort**
(`gpu_crocsort/experiments/cub_raw_sort.cu`):

| Workload | Wall | GB/s |
|----------|------|------|
| SF50 (300 M uint64 keys, 2.4 GB) | 33.9 ms | 70.79 |
| SF100 (600 M, 4.8 GB) | 67.6 ms | 71.00 |
| SF300 (1.8 B, 14.4 GB) | 203.4 ms | 70.79 |

This is **single-GPU CUB SortPairs** — RMG-style would multiply this
by N_GPUs minus comm overhead. RMG paper reports 3.5× speedup at 8
GPUs vs single-GPU; on 4 GPUs it'd be ~2.5×, giving expected ~175
GB/s for 14.4 GB sort. But RMG is keys-only on uint32; gpu_crocsort
is full-record (120 B records, real TPC-H lineitem).

## Direct comparison (what the paper should claim)

RMG Sort (BTW 2023):
- 8× A100 NVSwitch
- **Largest: 32 B uint64 keys (128 GB)**
- ~3 sec, 42 GB/s aggregate
- Keys-only, no payload

gpu_crocsort (this work):
- 4× H100 NVL
- **Largest: SF1000 = 6 B records × 120 B = 720 GB**
- 14m20s globally sorted with full payload (5.6× larger than RMG)
- Includes gather phase that RMG doesn't have

The RMG comparison is "apples to oranges" — RMG is a sort kernel
benchmark, gpu_crocsort is a full external-sort engine including
extract + upload + gather. Direct GB/s comparison would unfairly
favor RMG since it has no host-side work.

## Workaround: equivalent CUB-only number

We have raw CUB SortPairs at SF300 = 70.8 GB/s on uint64 keys
(experiment 4.4). On 4 GPUs with keys-only this would be ~250 GB/s
aggregate. So RMG on 4× H100 should be ~150-250 GB/s on keys-only
sort.

gpu_crocsort 4-GPU at SF500 (full payload): 65.7 GB/s on 120-byte
records = 5.5 GB/s on 16-byte sort prefix equivalent. Different
regime; can't be normalized cleanly.

## Conclusion for paper

We can't easily run RMG here, but we can claim:
- gpu_crocsort handles workloads RMG cannot (full record gather,
  composite keys, lineitem).
- gpu_crocsort sorts 5.6× more data globally than RMG's largest
  reported result.
- Raw CUB on our 4× H100 would be ~150-250 GB/s but only on
  keys-only synthetic data; not representative of database sort.
