# 15.3 — 4-GPU concurrent SF100 (weak-scaling baseline)

**Date:** 2026-05-03 07:19 UTC
**GPU(s):** 4× NVIDIA H100 NVL (all 4 active)
**Workload:** Each GPU independently sorts the same SF100 file (no
partitioning, no merge — embarrassingly parallel)
**Wall time:** 2 min 44 s for 4 × (3 runs of SF100)
**Status:** PASS (all 12 sorts verified)

## Command(s)

`/tmp/run_15.3.sh`:
```bash
source ~/gpu-research/h100/env.sh
BIN=/home/cc/gpu-research/gpu_crocsort/external_sort_tpch_compact
INPUT=/mnt/data/lineitem_sf100.bin

CUDA_VISIBLE_DEVICES=0 $BIN --input $INPUT --runs 3 > /tmp/15.3_gpu0.log 2>&1 &
CUDA_VISIBLE_DEVICES=1 $BIN --input $INPUT --runs 3 > /tmp/15.3_gpu1.log 2>&1 &
CUDA_VISIBLE_DEVICES=2 $BIN --input $INPUT --runs 3 > /tmp/15.3_gpu2.log 2>&1 &
CUDA_VISIBLE_DEVICES=3 $BIN --input $INPUT --runs 3 > /tmp/15.3_gpu3.log 2>&1 &
wait
```

Single-GPU SF100 baseline (from 0.3): **4.50 s warm best** at 16.0 GB/s.

**Aggregate 4-GPU effective throughput (warm):**
- 4 × 72 GB sorted concurrently in ~4.6 s = **62.6 GB/s of aggregate sort throughput**.
- Cold first run: ~21 s due to 4 procs simultaneously reading 4 × 72 = 288 GB from NVMe at 3 GB/s aggregate = ~96 s floor; we ran serially through NVMe.

## What worked

- **Linear weak-scaling** at the workload level. Each GPU reaches its
  single-GPU warm-best with no degradation when 3 other GPUs sort
  concurrently. Per-GPU PCIe5 is independent, HBM is independent,
  CUDA contexts are independent.
- The 4 processes share **one copy** of the 72 GB SF100 file in page
  cache (read-only mmap), so host RAM is *not* 4× — it's ~72 GB for
  the input plus 4 × 72 GB output (mmap, lazy-faulted now post-1.12.1)
  ≈ 360 GB peak. Comfortable under 1 TB host RAM.
- All 12 sorts PASS verification — confirms there's no GPU/CUDA-stream
  cross-contamination when 4 sorts run concurrently on the same box.

## What failed

Initial attempt with bash `( cmd ) & PID=$! ; ( cmd2 )` syntax confused
the parser — only the first command spawned, the rest got "command not
found". Fixed by writing the launcher as a shell script with each
`cmd & ; PID0=$!` on separate lines. Documented for future use.

Cold-cache run had heavy NVMe contention — 4 procs hit NVMe at 3 GB/s
aggregate, so each cold run was ~21 s instead of the single-proc
~12 s cold. After page cache warms, no contention.

## Surprises

1. **Warm-best per GPU is identical to single-GPU baseline** (4.50 s).
   Zero degradation at 4-way concurrency. This is a strong signal that
   the per-GPU sort is *not* host-bandwidth bound — host can absorb
   all 4 PCIe streams + all 4 OVC merges + all 4 gathers concurrently.
2. **GPU 3's first run was *fastest cold*** (5.3 s vs 21 s on GPUs
   0/1/2). Likely scheduling — GPU 3 was last to start so by the
   time it loaded, 75% of the file was already cached by GPUs 0-2.
3. **The 4 mmap'd output buffers (4 × 72 GB = 288 GB) didn't OOM-kill**
   anything — confirms 1.12.1's lazy faulting + B1's persistent buffer
   reuse handle multi-process concurrency correctly.

## Interpretation

The H100 box is a clean weak-scaling target up to 4 concurrent sorts
at SF100 each (288 GB total data sorted in ~5 s warm at 62 GB/s
aggregate). The PCIe5, HBM3, host RAM, and CPU are all independent
enough across the 4 GPUs to give linear scaling for this size class.

This is the ceiling for the simplest multi-GPU sort architecture
("4 independent sorts, no inter-GPU communication"). The next step
(15.4) is a TRUE distributed sort: partition input by RANGE so the 4
GPU outputs concatenate to a single globally-sorted result. NVLink
shuffle (15.2 measured at 533 GB/s aggregate) makes the partition
phase cheap.
