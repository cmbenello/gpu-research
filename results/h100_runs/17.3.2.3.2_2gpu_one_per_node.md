# 17.3.2.3.2 — 2-GPU one-per-node test: confirms node-pair contention

**Date:** 2026-05-04
**Setup:** GPU 0 → node 0, GPU 2 → node 1 (one GPU per NUMA node).
**Workload:** SF300 split into 2 contiguous halves (108 GB each).
**Goal:** Test whether the 33% headroom in 17.3.2.3 (4-GPU 2-per-node)
is due to 2 GPUs sharing a single NUMA node's memory channels.

## Result (108 GB per GPU, 3 runs each)

| GPU | NUMA | run 1 | run 2 | run 3 |
|-----|------|-------|-------|-------|
| 0   | 0    | 4327  | 4253  | 6215 (outlier) |
| 2   | 1    | 4368  | 4250  | 4311  |

Per-run aggregate (max wall):

| Run | max wall | aggregate GB/s |
|-----|----------|-----------------|
| 1   | 4368     | 49.5            |
| 2   | **4253** | **50.8 ← best** |
| 3   | 6215 (GPU 0 outlier) | 34.8 |

## Scaling efficiency picture

| Config | aggregate GB/s | per-GPU avg | scaling vs single |
|--------|-----------------|--------------|---------------------|
| 1 GPU SF300 numactl | 27.7 | 27.7 | 1.0× |
| **2 GPU one-per-node, SF300/2** | **50.8** | **25.4** | **1.83×** (92% efficient) |
| 4 GPU two-per-node, SF500/4 | 65.7 | 16.4 | 2.37× (59% efficient) |

When each GPU has its own NUMA node, **scaling efficiency is 92%** —
nearly perfect. The 8% loss is probably I/O serialization on the
shared input file (both GPUs read the same lineitem.bin).

When 2 GPUs share a node, **scaling drops to 59%** — confirming
the 33% headroom in 17.3.2.3 is real and is caused by memory channel
contention within a NUMA node.

## What this tells us

- **2 GPUs per box (one per node)** should be the configuration for
  maximum per-GPU throughput. ~25 GB/s × 2 = 50 GB/s.
- **4 GPUs per box** sacrifices 33% of per-GPU throughput to memory
  contention. Net aggregate is 65.7 GB/s — better than 2-GPU's 50.8
  but only 1.29× more for 2× the GPUs.
- For paper claims: report both. "Best 2-GPU: 50 GB/s, best 4-GPU:
  66 GB/s." The reader should pick based on their workload's
  scaling target.

## The run-3 GPU 0 anomaly

GPU 0 run 3 came in at 6215 ms vs run 2's 4253 ms — a 46% slowdown.
Probable cause: kernel auto-NUMA migration triggered between runs
(`khugepaged` or `numa_balancing` daemons may reshuffle pages they
think are imbalanced). Numactl's strict `--membind=0` is supposed
to prevent this, but the kernel may still try to migrate THP-promoted
pages.

Disabling `kernel.numa_balancing` (kernel sysctl) might fix the
sporadic outlier. Out of scope for this experiment; flagged as
17.3.2.3.2.1.

## Caveats

- Single sweep. Need ≥3 sweeps to bound variance.
- The run-3 GPU 0 outlier (6215 ms vs 4253 ms) hints at kernel
  NUMA-migration noise even with strict membind. Need to investigate.
- 2-GPU and 4-GPU partition results are NOT globally sorted — they
  produce N sorted slices that need a final merge step or callers
  iterate them in order (15.5.3 model uses key-range partition for
  globally sorted outputs).
