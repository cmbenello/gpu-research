# 19.8 — Per-byte entropy of TPC-H lineitem: theoretical limit 19.5 B

**Date:** 2026-05-04
**Workload:** SF300 lineitem, 1 M records sampled
**Status:** PASS — quantifies the compact-map opportunity

This data backs the **adaptive compact-key encoding** claim with hard
numbers. Per the per-byte distinct-value count of TPC-H lineitem's
66-byte sort key:

## Headline result

```
Total entropy: 156 bits per record (vs 528 if all bytes full)
Theoretical compact key size: 19.5 bytes
Our actual compact map output: 32 bytes (1.6× theoretical limit)
Raw uncompacted key size: 66 bytes (3.4× theoretical limit)
```

## Distribution of distinct values per byte

| Bytes | Count | Description |
|-------|-------|-------------|
| ≤16 distinct values | 48 / 66 | Mostly constant (e.g. l_returnflag, l_linestatus, padding) |
| 17-64 distinct values | 4 / 66 | Low-cardinality (e.g. l_tax bytes) |
| >64 distinct values | 14 / 66 | High-cardinality (l_orderkey, l_partkey, l_quantity, l_extendedprice low bytes) |

## Byte-by-byte entropy map

Bytes with **1 distinct value** (constant): positions 2, 3, 6, 7, 10,

These 35 constant bytes are TPC-H tax/discount/extendedprice high

spec).

## Implications for adaptive compact-map

### Why the compact-map works

A naive radix sort on the full 66-byte key needs to ship 66 bytes per
record across PCIe = 119 GB at SF300 (1.8 B records). Our compact-map
detects which bytes vary and ships only those = 32 bytes × 1.8 B =
57.6 GB = **52% reduction** over raw.

A theoretically-perfect Huffman-coded key would be 19.5 bytes ×
1.8 B = 35 GB = **71% reduction**. We're 30% above optimal because
we round to byte boundaries (radix sort needs byte-aligned digits).

### Why 32 B not 20 B

Our compact map picks bytes whole — if a byte has 2 distinct values
(1 bit of entropy), we still ship the whole byte. This wastes ~7
bits per such byte. For the 48 low-entropy bytes, that's ~7 bits
× 48 = 336 bits = 42 B of waste. We keep only ~half of those bytes
(the ones the radix sort actually uses for ordering), so net waste
is ~20 B.

A bit-packed compact map would shave to ~20 bytes but would require
non-byte-aligned radix passes, which CUB doesn't support efficiently.

### Quantified: PCIe traffic reduction

For SF300 (1.8 B records):
- Raw 66-byte sort key: 119 GB to PCIe
- Our 32-byte compact: **57.6 GB** (-52%)
- Optimal 19.5-byte: 35 GB (-71%, theoretical)

Our 32-byte choice trades ~20 GB extra PCIe for radix-sort
efficiency. Worth it: the H2D bandwidth at 55 GB/s × 21 GB extra
= 0.4 sec, vs gaining O(N) speedup from byte-aligned radix.

## What this proves for the paper

1. **TPC-H lineitem has 30% of bytes carrying real entropy and 70%
   are dead.** A naive sort wastes 70% of PCIe.

2. **Our adaptive compact-map captures the 32-byte working subset
   automatically** at runtime, no manual schema specification needed.

3. **Other databases / workloads should benefit similarly.** Customer
   tables, financial data, log timestamps all have similar
   low-cardinality columns that get padded into wide rows.

## Comparison: synthetic workloads

| Distribution | Expected entropy | Compact-map output |
|--------------|------------------|--------------------|
| TPC-H lineitem | 156 bits, 19.5 B theoretical | 32 B actual |
| sorted (SF50, 18.6) | ~8 bits (only top byte varies) | ~9.6 GB compact (= ~32 B/record × 0.3) |
| reversed | similar to sorted | ~9.6 GB |
| all_equal | 0 bits (1 distinct key) | 9.6 GB (compact still has constant bytes) |
| **uniform random 64-bit** | **64 bits per record** | **14.4 GB (= ~32 B × 0.5)** |

Wait — the random has ~50% entropy of compact key but all_equal also
has ~30%. The compact-map struggles with all_equal because it
includes constant bytes that don't help discrimination but radix
sort processes them anyway.

## Caveats

- Sample size = 1 M records (out of 1.8 B). Larger sample might find
  more distinct values in some bytes.
- Per-byte distinct count is a crude entropy metric; doesn't capture
  inter-byte correlations.
- TPC-H is synthetic data with deliberate low cardinality on certain
  columns. Real production data may have higher entropy.
