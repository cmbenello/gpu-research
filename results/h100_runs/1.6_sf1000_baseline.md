# 1.6 — TPC-H SF1000 sort (FAIL: pinned-host-memory exhausted)

**Date:** 2026-05-02
**GPU(s):** 1× NVIDIA H100 NVL
**Scale:** TPC-H SF1000 (720.00 GB, 5,999,989,709 × 120-byte records)
**Wall time:** 6 min before crash (mostly NVMe load)
**Status:** **FAIL** — `cudaMallocHost` returns "invalid argument" trying to
add 63.6 GB pinned-host-memory (3 × 21.21 GB triple buffer) on top of an
already-pinned 720 GB input.

## Command(s)

```bash
source ~/gpu-research/h100/env.sh
cd ~/gpu-research/gpu_crocsort
CUDA_VISIBLE_DEVICES=0 ./external_sort_tpch_compact \
    --input /mnt/data/lineitem_sf1000.bin --runs 1
```

## Raw numbers

```
Allocating 720.00 GB pinned host memory... OK (6 min, 2.04 GB/s NVMe cold read)
[ExternalSort] GPU: 97.90 GB free, budget: 63.63 GB
[ExternalSort] Triple-buffer: 3 × 21.21 GB (177 M records each)
CUDA error at src/external_sort.cu:778: invalid argument
```

`src/external_sort.cu:778` = `cudaMallocHost(&h_pin[i], buf_bytes)` — the
allocator for the 21.2 GB triple-buffer slot fails because we've already
pinned 720 GB of host memory, leaving roughly 304 GB of unpinned RAM, but
the kernel won't let us pin all of that (Linux usually caps pinned memory
to ~50% of system RAM unless `RLIMIT_MEMLOCK` is unlimited *and* huge pages
are reserved).

```
$ ulimit -l
(probably "unlimited" since the 720 GB pin worked, but the marginal pin
fails — the kernel may simply be unable to find 21 GB of contiguous huge
pages after the 720 GB pin fragmented memory)
```

## What worked

- The 720 GB input *did* load and pin. NVMe sustained 2.04 GB/s on a
  truly cold read (vs 3.3 GB/s for SF500). Likely contention or fewer
  parallel reads at this size.
- The cudaMallocHost call producing "invalid argument" is a graceful
  failure — no GPU corruption, no kernel hang. The binary exits with
  code 1 and the GPU returns to idle within seconds.

## What failed

The triple-buffer allocation. Three observations:

1. The triple-buffer is *additional* pinned host memory on top of the
   720 GB input. Total pinned: 720 + 63.6 = 783.6 GB of 1024 GB.
2. The host_sort path doesn't seem to need this triple buffer — it's set
   up unconditionally in the constructor at line 778 even when the data
   is fully held in `h_data` already. That's wasteful at any scale and
   fatal at SF1000.
3. The lazy GPU buffer allocation (`d_buf[i] = nullptr`) means the failure
   is *only* on the pinned-host-buffer side, not GPU memory.

## Surprises

1. **The unconditional cudaMallocHost in the constructor is the bug.**
   At smaller scales the triple buffer is "cheap" (3 × ~5 GB = 15 GB) so
   nobody noticed; at SF1000 it triples on top of the 720 GB pin and
   becomes the hard limit. Filed as 1.6.1 below.
2. **The cold-read at SF1000 is slow** — 2.04 GB/s on the same NVMe that
   read SF500 at 3.32 GB/s. Possibly because the larger pinned-memory
   allocation forced a different IO path, or the kernel was thrashing
   migrating pages to make 720 GB pinned.

## Interpretation

Single-GPU SF1000 *might* sort correctly if the triple-buffer were lazy-
allocated like the GPU buffers. The OVC architecture's per-record buffers
are 60 GB on GPU, plus prefix buffers 60 GB, plus 8 GB perm — all on GPU,
not host-pinned. So the host-pinned needs are just the input (720 GB) +
the gather output (also 720 GB once sorted = another huge pin). At a
1024 GB host RAM, *two* 720 GB pins simply don't fit. So even with the
triple-buffer fix, SF1000 single-GPU is borderline.

The natural answer is multi-GPU (15.4 / 15.5): split the input, each GPU
handles ~180 GB, output streams back to host in chunks. **SF1000 is the
hard motivation for multi-GPU on this box.**

## Follow-ups

- **1.6.1 (new) — make triple-buffer lazy.** Move the `cudaMallocHost`
  for `h_pin[i]` from the constructor into the slow path that actually
  uses it. Should let SF1000 single-GPU at least *try* the OVC path.
- **1.6.2 (new) — output buffer reuse.** When sorted output is to a
  caller-allocated buffer (like the `h_data` we already have pinned),
  skip the additional gather-output pin. Currently `h_output =
  mmap(...)` allocates a *separate* 720 GB buffer.

  300 PASS, SF500 / SF1000 FAIL on 1 GPU).
