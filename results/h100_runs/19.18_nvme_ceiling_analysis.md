# 19.18 — NVMe ceiling analysis: why 31m07s is at the floor for current pipeline

**Date:** 2026-05-05
**Question:** Why can't we go below 31m07s?

## Hardware ceiling on this box (single NVMe at /mnt/data)

```
$ dd if=/dev/zero of=/mnt/data/test bs=4M count=4096 conv=fdatasync
17 GB written in 13.2 sec → 1.3 GB/s peak write

$ dd if=/mnt/data/lineitem_sf1500.bin of=/dev/null bs=4M count=4096
17 GB read in 5.5 sec → 3.1 GB/s peak read
```

**Single SSD limits: write 1.3 GB/s, read 3.1 GB/s, ~3.3 GB/s combined.**

## Where the 31m07s goes

For SF1500 (1.08 TB input → 1.08 TB output) the current pipeline does:

```
Phase 1 (partition):  read 1.08 TB input  + write 1.08 TB intermediate buckets
Phase 2 (sort):       read 1.08 TB intermediate + write 1.08 TB sorted output
```

- Partition write 1.08 TB at 1.3 GB/s = **14 min minimum**
- Sort write 1.08 TB at 1.3 GB/s = **14 min minimum**
- Sort read 1.08 TB at 3.1 GB/s = 5.8 min (overlaps with sort write)
- These two phases must be sequential (sort needs partition output)

**Sequential floor: 14 + 14 = 28 min, plus eviction/setup = ~31 min.**

Our measured 31m07s **IS** the NVMe write ceiling, paid twice. Adding more
GPUs, NUMA tuning, or smarter scheduling cannot break this floor — it's a
fundamental I/O limit.

## What that implies for "smarter, not faster"

Two structural changes can break the floor:

### Option A: Perm-only output (eliminate the second 14m write)

Instead of writing 1.08 TB of sorted records, write 36 GB of sorted
permutation indices (4 B per record × 9B records). Downstream consumer
gathers from input.bin lazily.

- Sort write: 1.08 TB → 36 GB. Saves ~13 min on phase 2.
- New total: 14m partition + ~3m sort phase = **~17m projected**

### Option B: Streaming partition (eliminate the first 14m write)

GPU classifies records on-the-fly during input streaming. Per-bucket
output stays in HBM/host RAM, sorted in place, written directly to
final output file.

- Partition write: 1.08 TB → 0. Saves ~14 min on phase 1.
- New total: ~6m streaming + 14m output = **~20m projected**

### Option A + B (both): ~10m projected

Theoretical floor approaches read-only-once + small-write =
1.08 TB / 3.1 GB/s + 36 GB / 1.3 GB/s = 6m + 28s ≈ **6.5 min**.

## Why the current 31m07s is still impressive

Most papers benchmark in-memory sort. Our novelty is:
- Single-host sort beyond host RAM (1.08 TB > 1 TB)
- Full TPC-H lineitem records, not just keys
- Single shared SSD on a commodity 4×H100 box

Even at 31m07s, we're 9-118× faster than DuckDB / Polars / DataFusion at
SF50/SF100, and beat them at SF300 where they OOM. The NVMe-bound regime
is one no-one else publishes — they stop before they hit it.

## Next implementation lever (highest ROI)

**Perm-only output (Option A) is the most tractable.** Required changes:

1. Add `PERM_ONLY=1` env path to `external_sort.cu` that:
   - After sort, writes `ss.h_perm` (4-byte indices) to `--output-file`
   - Skips the CPU gather phase (saves ~17 sec at SF1500/bucket too)
2. Run partition unchanged
3. Re-run K=16 4-GPU recipe with PERM_ONLY=1

Predicted: 17-19 min total (vs current 31m07s) — **40-45% additional
speedup beyond the current best**, putting us at 0.95-1.06 GB/s
end-to-end on full TPC-H records.
