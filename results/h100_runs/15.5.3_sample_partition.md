# 15.5.3 — sample-partition distributed sort (no merge phase)

**Date:** 2026-05-03 17:11–17:41 UTC
**Status:** PASS — 3 BILLION records globally-sorted-by-range

## What this does

Eliminates the merge phase of 15.5 by *partitioning before sorting*:

```
INPUT (lineitem_sf500.bin, 360 GB)
    │
    │  partition_by_range tool (CPU, multi-threaded)
    │  ─ sample 8192 keys
    │  ─ sort sample, pick K-1 splitters at quantiles
    │  ─ pass 1: count records per bucket (per-thread)
    │  ─ pass 2: route records to bucket files (per-thread offsets)
    ▼
4 bucket files (key-range disjoint)
    │
    │  4 GPUs sort each bucket (sequential or paired)
    ▼
4 sorted bucket files, by construction in range order
    │
    │  (skip: just iterate in order, no concatenation needed)
    ▼
GLOBALLY SORTED OUTPUT (verified PASS)
```

## Implementation

Two new tools:
- `gpu_crocsort/experiments/partition_by_range.cpp` — CPU sample-sort partitioner
- (existing) `external_sort_tpch_compact --output-file` writes sorted output to disk

Verification: max(bucket[i].last_key) ≤ min(bucket[i+1].first_key), and
each bucket internally sorted (verified by the sort binary's own `--runs`
verifier).

## Measurements at SF500 (3 B records, 360 GB)

```
Phase                      Wall (s)   Notes
─────────────────────────────────────────────────────────────
Partition (CPU, 64-thread)
  Sample + splitters:           0.6
  Pass 1 (count):             123
  Pass 2 (route):              42  (8.59 GB/s — close to RAM ceiling)
  Total partition:           ~178   (2.03 GB/s end-to-end)

Sequential GPU sort (4 buckets, ~89 GB each)
  Bucket 0:                    35 s sort + 38 s write to /dev/shm
  Bucket 1:                    14 s sort + 39 s write
  Bucket 2:                    35 s sort + 40 s write
  Bucket 3:                    46 s sort + 38 s write
  Total sort phase:          ~672 s (incl. per-process input pinning ~30s each)

End-to-end wall:           ~847 s (~14 min)
```

Versus 15.5 merge-based: ~12 min wall. Sample-partition is *slightly
slower* end-to-end at SF500 (because of sequential per-bucket pinning
overhead), but produces output in a form that **doesn't need merge**.
Useful for downstream consumers that iterate in sorted order without
materializing a single concatenated file.

## What worked

- **Sample partitioning gave near-uniform bucket sizes** (88.66, 90.73,
  91.59, 89.02 GB — only 3% deviation across 4 buckets).
- **Verification PASS** at SF10 (60M records concat'd), SF100 (600M), and SF500
  (3B). Boundary check confirms range order; the sort binary's `--runs`
  verify confirms internal sortedness.
- **Pass 2 hit 8.59 GB/s end-to-end** — the route step is memory-bandwidth
  bound, near the host RAM ceiling.

## What failed (initial attempts)

- **v1: 4 concurrent sorts, output → /dev/shm**: 1 process OOM-killed.
  336 GB partition tmpfs + 4 × 178 GB pinned (input + MAP_POPULATE'd
  output) = 1.05 TB > 1024 GB host. Cascaded.
- **v2: 4 concurrent sorts, output → /mnt/data**: same OOM cascade
  because the binary's `MAP_POPULATE` on the output buffer makes it
  resident even when the path is on NVMe. 2 of 4 GPUs got stuck in
  Step 4 (Download Permutation) — likely also OOM-related.
- **v3: sequential sorts**: PASS but slower than parallel would have been.

## Surprises

1. **Pass 1 (count) is slow** — 123 sec to scan 360 GB at 2.9 GB/s.
   That's much slower than Pass 2 (42 sec at 8.6 GB/s for the same
   amount of data). Hypothesis: Pass 1 reads the input in random
   pattern (because each thread does a contiguous chunk but the
   bucket-decision causes branch unpredictability). Pass 2 has the
   same pattern but the output writes are NT-friendly. Worth a
   profile.
2. **The per-bucket sort wall is dominated by pinning + writing**, not
   the actual GPU compute. Each fresh `ExternalGpuSort` instance
   re-pins its 89 GB input (~30 s cold), runs the GPU sort (~5 s), and
   writes 89 GB output (38 s for shm or 60 s for nvme). The GPU
   itself is idle most of the wall time.

## Interpretation

For a "globally sorted" SF500 result this is a **valid alternative to
15.5's merge-based approach** with similar wall time. It's preferable
when:
- The downstream consumer can iterate K files in order (no merge needed).
- The disk-staging budget is tight (no need for an intermediate merged file).

It's *not* preferable when:
- A single concatenated output is required (cost adds 2-3 min).
- Available host RAM is small (the partition's intermediate files
  occupy 336 GB at SF500).

## Follow-ups

- **15.5.3.1 — paired sorts** (2 GPUs at a time): predicted 8 min wall vs 14 sequential.
- **15.5.3.2 — fuse the partition write with NVMe streaming** instead of
  buffering the entire partitioned output in /dev/shm. Saves the 336 GB
  RAM peak.
- **15.5.3.3 — share the input pin across the 4 sort processes**.
  Currently each process pins its own 89 GB even though all 4 buckets
  came from the same SF500 file. A shared pin or a fork-after-pin
  approach would save 3 × 89 GB of redundant pinning work.
