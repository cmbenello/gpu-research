**Date:** 2026-05-02
**GPU(s):** 1× NVIDIA H100 NVL (CUDA_VISIBLE_DEVICES=0)
**Scale:** TPC-H SF300 (216.00 GB, 1,799,989,091 × 120-byte records)
**Wall time:** ~3 min/config
**Status:** PASS (all 6 runs)

## Command(s)

```bash
source ~/gpu-research/h100/env.sh
cd ~/gpu-research/gpu_crocsort

CUDA_VISIBLE_DEVICES=0 ./external_sort_tpch_compact \
    --input /mnt/data/lineitem_sf300.bin --runs 3
CUDA_VISIBLE_DEVICES=0 USE_BITPACK=1 ./external_sort_tpch_compact \

```

## What worked

- Both configurations PASS verification on all 3 runs.
- The OVC ARCHITECTURE auto-engaged at SF300 because the compact-key buffer
  (118.8 GB total = 1.8B records × 66B key) exceeds the budget; it falls
  back to run-gen + GPU OVC merge + CPU gather. *This is the path SF100 should
  also have used* — see Surprises.
- **USE_BITPACK delivers a real 11-13% wall-time improvement at SF300.**
  This is the first scale where bitpack pays off; consistent with the 0.2/0.3
  hypothesis that bitpack only wins when the merge phase dominates.
- PCIe compression on the wire: baseline 3.75×, bitpack 5.0× — the bitpack
  is what closes the last 22 GB of PCIe traffic at SF300.

## What failed

Nothing failed verification. The only unhappy thing is the cold-run penalty
(see Surprises).

## Surprises

1. **Cold-run penalty is enormous at SF300.** Baseline cold = 93 s, warm = 10 s
   (9× slower cold). Bitpack cold = 73 s. The 216 GB input has to be paged
   into RAM from NVMe; at NVMe sustained ~3 GB/s, that's ~72 s of disk time
   alone, matching the cold-cold penalty. After the file is in page cache
   (~/proc/meminfo MemAvailable should drop by 216 GB after run 1), warm
   runs hit RAM bandwidth, not NVMe.

2. **Bitpack cold run is *worse* than baseline cold (73 vs 93 s).** Wait —
   actually the bitpack cold *total* is 72 s vs baseline cold 93 s, so
   bitpack is 21 s *faster* cold. But the wall_ms field shows 58 s vs 14 s —
   bitpack's run-gen took longer because it does the per-position min/max
   scan on cold pages. Net: still faster overall because the merge gain
   outweighs the prepack cost. Worth profiling.

3. **The OVC architecture changes the scaling story between SF100 and SF300.**
   At SF100 (72 GB), data fits the single-pass threshold so the binary runs
   the "Step 1+2 Key Upload → key-only sort → CPU gather" path. That path
   uploads *full 66-byte keys*, killing throughput (see 0.3). At SF300, the
   key buffer would be 118.8 GB which exceeds the budget, so it falls back
   to OVC architecture which uses 32-byte (or 24 B with bitpack) compact
   uploads. **SF300 is therefore *more efficient than SF100* per byte sorted**
   (24.2 GB/s vs 16.0 GB/s). This is the opposite of the typical "throughput
   degrades with scale" story — and it's a code-path-selection artifact, not
   a hardware effect. Filed under 0.3.1 already; this strongly motivates
   fixing the SF100 path.

4. **The merge phase at SF300 is 5.1-6.0 s** — that's ~80% of the warm wall
   time. Predicted by 0.2/0.3 (bitpack helps when merge dominates), confirmed
   here. At SF500 / SF1000 the merge will dominate even more, so bitpack's
   relative win should grow.

## Interpretation

SF300 is the **first scale where the H100 / compact-key pipeline meaningfully
beats what the RTX 6000 can do at all** — the RTX 6000 OOMs at SF300, the
H100 sorts it in 8.9 s warm. Bitpack delivers a clean 11-13% wall-time win
here, validating the hypothesis from 0.2/0.3 that bitpack only pays when
merge dominates total wall time. The headline number for the H100 paper at
this point is **24 GB/s sustained on a 216 GB sort with a 32B compact key
+ bitpack on a single H100 NVL**, with PCIe traffic compressed 5× from
the raw record bytes.

What's still wrong: the path-selection between SF100 (single-pass key
upload, slow) and SF300 (OVC, fast) is a code-quality issue, not a hardware
limit. Fixing 0.3.1 (compact uploads in the single-pass path) should make
SF100 also reach ~20 GB/s and give the curve a clean monotonic story.

## Follow-ups

- **1.2.1 (new) — repeat SF300 with `vmtouch -e lineitem_sf300.bin` between
  runs** to force a *true* cold-cold run for each iteration. Current cold
  is "first run after gen", the next 2 are warm. We don't know what a
  re-cold-run looks like (NVMe + same kernel cache eviction patterns).
- **1.2.2 (new) — measure how much of the 5-6 s merge is OVC sort vs CPU
  gather.** The OVC buffer is 36 GB on GPU (4B OVC + 8B prefix + 4B perm
  per record) — gather then reads from host the original records via the
  global perm. CPU gather at SF300 = 1.8 B records × 120 B / N threads is
  the next bottleneck candidate.
- **1.2.3 (new) — try reducing OVC buffer.** 8B prefix + 4B OVC + 4B perm
  could potentially be fused or made implicit. Saves HBM that could go
  toward larger run sizes (fewer merge passes).
- 1.3 (next) is generating SF500 (~360 GB). Disk has 3.0 TB free → fine.
  Encode time at observed 0.16 GB/s = ~37 min. Likely exceed the 60-min
  experiment cap if we wait to write the result; gen-only experiments
  should probably be excluded from that cap.
