2. **93.2% wall-time reduction** at SF100 cold (first-touch heavy).
3. Achieves **24 GB/s effective throughput** for SF100 warm — the

This is the strongest single-experiment finding of the day. It comes

## Why default Linux is so much worse

The kernel's auto-NUMA placement decisions are conservative and
historically dominated by "first-touch" — the page goes to whichever
node first wrote to it. For our sort:

- **Input file mmap:** pages go to the node where the master thread
  reads them first. Master thread is wherever the kernel placed it.
- **Pinned host CUDA buffers:** allocated by the master thread, so
  on master's node.
- **Worker threads:** the kernel may schedule them on EITHER node
  depending on load balancing. Half land on the wrong node.
- **Random gather:** every read has 50% chance of being remote
  (~140-180 ns vs ~80 ns local). Half the gather time is spent
  waiting on UPI traffic.

Numactl forces *everything* — process, memory, CUDA host pinning — to
node 0. All accesses are local. UPI is idle.

## Cold-run pathology (default 45 sec vs numactl 3 sec)

The 14.7× cold speedup tells a worse story for the default policy:
during the first sort run, the input file is being paged in from
NVMe, the gather output buffer is being faulted in zero-pages, and
the GPU pinned buffers are being registered. Default policy spreads
all these allocations across both nodes, then 50% of the workers'
first reads hit the wrong node, plus the kernel may migrate pages
back and forth as access patterns change. Numactl pins everything
up front, so first-touch is local-only.

This means: **the prior "headline 3.02 s SF100 first-warm-after-cold"
was actually leaving the easier 1.84× win on the table for the warm
case, plus a 14.7× cold-run pathology that matters for short-lived
processes.**

```bash
CUDA_VISIBLE_DEVICES=0 numactl --cpunodebind=0 --membind=0 \
    ./external_sort_tpch_compact \
    --input /mnt/data/lineitem_sf100.bin --runs 5
```

## Caveats

- Single sweep, not run multiple times. Variance unknown.
- The default policy's high variance (5042-6157 ms range across 4
  warm) suggests the kernel is doing something different each run
  (different page placements? migration?). Numactl variance is
  much tighter (2952-2987 = 1.2% spread), confirming "bind everything
  removes the noise."
- This is a single-GPU result. Multi-GPU sorts (15.5.3) would need
  per-GPU NUMA binding.
