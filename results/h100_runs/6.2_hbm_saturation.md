# 6.2 — HBM3 bandwidth saturation on H100 NVL (PEAK MEASURED)

**Date:** 2026-05-03
**GPU:** 1× NVIDIA H100 NVL (94 GB HBM3, sm_90, CUDA 12.4)
**Workload:** Synthetic streaming kernels on 32 GB device buffers
**Status:** PASS — sets the roofline ceiling for sort throughput

## Result

```
GPU: NVIDIA H100, 99.9 GB HBM

32.2 GB working set, 4,026,531,840 uint64 elements

  pure_memcpy          1-read 1-write : 147 ms / 5 iters → 2191 GB/s
  modify_2x_p1         1-read 1-write : 147 ms / 5 iters → 2191 GB/s
  streaming_sum        1-read 0-write :  106 ms / 5 iters → 1521 GB/s
```

## Interpretation

- **2191 GB/s** for read+write streaming = effective HBM3 bandwidth
  with both directions in flight. Counts each byte once for read and
  once for write, so the total bus traffic is 2× this number ≈ 4382
  GB/s of bus activity on a roofline that nvidia advertises as
  3.35 TB/s. We hit ~65% of the marketing peak — typical for
  user-mode kernels (vs. nvbandwidth which can hit ~80%).
- **1521 GB/s** for read-only streaming reduction = the upper bound
  on any sort phase that only needs to read.
- The fact that `pure_memcpy` and `modify_2x_p1` produce identical
  numbers shows the compute (mul + add) is fully hidden — both are
  100% memory-bound. Good.

## Roofline implication for sort

A radix-sort LSD pass over 32 B compact keys does:
- 1 read of in[N×32] = 32N B
- 1 write of out[N×32] = 32N B
- Total bus = 64N B per pass

For SF300 = 1.8 G records × 32 B = 57.6 GB per pass. At **2191 GB/s
peak**, one LSD pass should take **57.6 / 2191 = 26.3 ms**. CUB does
~3 LSD passes for 32 B keys (8-bit digits × 4 = 32 bits at a time
across ~half the key bytes that vary), so theoretical minimum =
**~80 ms** for the GPU sort kernel alone.

Actual measured CUB sort time on SF300 compact keys: **150-180 ms**
(from prior 1.5 traces). So CUB hits **~50% of HBM peak** — that's
the canonical "good radix sort" number. There is no big gain to be
had by replacing CUB with a custom radix; the HBM ceiling is the
ceiling.

## Where the rest of wall time goes

SF300 wall = ~8.6 s. GPU sort = ~0.2 s. So **2.3% of wall is
GPU-bound**. The other 97.7% is host-side: compact extract (~1.5 s),
PCIe upload (~0.5 s), gather (~5.0 s), fixup (~1.0 s). The roofline
lesson: **stop optimizing the GPU**, the GPU is already at half-peak
HBM. The remaining work is on the host gather and the PCIe lane.

## What this rules out for the paper

- "We could 2× faster sort throughput on a future GPU" — only if
  HBM bandwidth scales. H100 → H200 = 4.8 TB/s vs 3.35 TB/s = 1.43×.
  So a GPU-only roofline projection for H200 is **+43% on the GPU
  sort phase** but the host-bound gather phase doesn't move. Total
  end-to-end speedup expected: ~5%.
- "We could 10× by going to FP8/INT4 keys" — irrelevant, our key
  is bytes.
- "We could parallelize across HBM stacks" — already happening at
  the hardware level inside one GPU.


## Caveat

This is a 32 GB working set. Real sort working sets at SF300 are
~70-90 GB on device. Larger sets exhibit similar bandwidth (HBM is
not capacity-throttled for streaming), so the 2191 GB/s number is
representative.
