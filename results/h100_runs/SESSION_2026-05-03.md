### Tier A — small wins
- **Critical regression fix** (commit `4741407`): 1.6.1 lazy-h_pin
  used clobbered `buf_bytes` → SF300 segfault on every run since 1.6.1
  landed. Fixed.
- **NUMA-pin gather threads — REVERTED** (negative result documented
  inline). Both single-core and node-level pinning made gather 3-4×
  slower. Real fix needs explicit buffer NUMA partitioning.
- **Conditional MAP_POPULATE** (commit `76e1e8d`): restores eager
  output-buffer faulting at scales where it fits in host RAM. SF300
  gather went 17.2 s → 4.8 s (post-1.12.1 was 17.2 s; this restored
  the original 5 s and slightly beats it).
- **B1 — reuse h_output across `--runs`** (commit `0b763a1`): persist
  output buffer in `ExternalGpuSort` member; avoid per-run mmap +
  MAP_POPULATE. **SF300 wall: 10.10 → 8.65 s = 15% faster than
  original baseline.**
- **0.3.1 — slow-path compact upload** (commit `20f8058`): pipelined
  CPU compact-extract + H2D with persistent ping-pong stage buffers.
  Replaces 66 B/record strided DMA with 32 B/record compact. 9 LSD
  passes → 4. **SF50: 2.62 → 1.51 s (1.74×). SF100: 4.50 → 3.02 s
  (1.49×). First SF100 win vs RTX 6000.**

### Tier C — multi-GPU
- **15.2 NVLink p2p bench** (commit `0e74aa0`): pair-wise NVLink
  uniformly **133 GB/s/direction** across all 4 H100 NVLs (89% of
  theoretical NV6 = 6 × 25 GB/s ceiling). Aggregate 533 GB/s, 17×
  PCIe5 ceiling. Intra-GPU memcpy 994 GB/s.
- **15.3 4-GPU concurrent SF100** (commit `01a186e`): each GPU sorts
  the same SF100 file independently. Per-GPU warm best 4.50 s
  (identical to single-GPU). **62.6 GB/s aggregate** sort throughput.
  Linear weak scaling at this scale.
- **15.4 4-GPU SF500 partition** (commit `5d3bf7f`): added
  `--offset-records` / `--limit-records` flags. Each of 4 GPUs sorts
  90 GB chunk by position. **360 GB sorted in 6.77 s warm at 53.2
  GB/s aggregate.** Single-GPU SF500 OOMs entirely → "infinite
  speedup" headline.
- **B3-light** (commit `bb7cf3d`, partial): relaxed OVC path-selector
  to use compact-upload memory footprint. SF500 single-GPU now
  reaches the OVC path (gets through Phase 1) but still OOMs in Phase 2
  merge (60 GB OVC + 49 GB merge bufs = 109 GB > 88 GB). True chunked
  OVC needed to fully unblock.
- **15.5 distributed sort with merge** (commit `a9c7d50`): added
  `--output-file` flag + `merge_sorted_runs.cu` k-way merge tool.
  **3 BILLION records (SF500, 360 GB) globally sorted on 4 H100 NVLs
  in 11 minutes**, single-GPU OOMs. Verified globally sorted output.
  Bottleneck identified: 528 s host k-way merge.
- **15.5.2 multi-threaded merge** (commit `6e9af6e`): splitter-based
  parallel merge across N threads. **5.6× speedup** at 64 threads
  (1.35 → 7.60 GB/s on SF100 4-way). Projects SF500 distributed wall
  from 11 min to ~3 min.

### Tier 4 — cross-library baseline

## Remaining bottlenecks (post Session 2)

1. **CPU gather phase** still dominates single-GPU wall at SF300+
   (~5 s of ~9 s). Two-pass gather + NUMA-partitioned buffers projected
   2-3× win. **Filed as deferred Tier B work.**
2. **Distributed sort merge phase** post-15.5.2 is at ~7.6 GB/s.
   Could push to 10+ GB/s with NUMA-aware merge worker placement.
   **Filed as 15.5.2.1.**
3. **Single-GPU SF500/SF1000 still OOM.** Need true chunked OVC
   (B3 proper) or sample-partition+sort+concat (15.5.3). Both
   substantial code.
4. **Distributed sort I/O staging.** Phase 1 writes 360 GB to NVMe
   then phase 2 reads it back. Saves ~138 s if streamed via
   shared memory (15.5.1 demonstrated for SF100 in /dev/shm).
