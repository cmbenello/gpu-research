# 17.3.2 — `numactl` wrapped sort: SF100 wall improves 21% with single command

**Date:** 2026-05-03
**GPU:** 1× NVIDIA H100 NVL
**Workload:** TPC-H SF100 (72 GB, 600 M records)
**Status:** PASS — drop-in production deployment knob, no code change

## Result (single warm run, --runs 2)

| Policy | Phase 1 (ms) | Phase 2 (ms) | Total wall (ms) | Throughput | vs default |
|--------|---|---|---|---|---|
| default | 1884 | 1881 | **3765** | 19.1 GB/s | 1.00× |
| **`numactl --cpunodebind=0 --membind=0`** | 1495 | 1482 | **2977** | 24.2 GB/s | **1.27×** |
| `numactl --interleave=all` | 1544 | 1547 | 3091 | 23.3 GB/s | 1.22× |

Phase 1 = compact key extract + H2D + GPU sort + D2H. Phase 2 = CPU
gather + fixup. Both phases improve roughly equally with NUMA binding
(both involve large host-RAM streaming + random access traffic).

## Why this works without code changes

`numactl --cpunodebind=0 --membind=0` is a wrapper that:
1. Sets the process's CPU affinity to all logical CPUs on node 0
   (the 96 even-numbered CPUs on this dual-socket box).
2. Sets the memory policy to "allocate from node 0 only" (`mbind`
   `MPOL_BIND`).

Subsequent allocations — including the CUDA pinned host buffers, the
gather output (`malloc`), and the mmap'd input file — all land on node
0 memory, where the threads also run. **No cross-socket UPI traffic
during the entire sort.**

## Production deployment recommendation

Wrap the sort binary in a one-liner:

```bash
numactl --cpunodebind=0 --membind=0 \
    ./external_sort_tpch_compact \
    --input /mnt/data/lineitem_sf100.bin \
    --runs 2
```

For multi-GPU workflows where each GPU pair maps to a NUMA node:
- GPU 0/1 → node 0
- GPU 2/3 → node 1

```bash
numactl --cpunodebind=0 --membind=0 ./sort --gpu 0 ... &
numactl --cpunodebind=0 --membind=0 ./sort --gpu 1 ... &
numactl --cpunodebind=1 --membind=1 ./sort --gpu 2 ... &
numactl --cpunodebind=1 --membind=1 ./sort --gpu 3 ... &
wait
```

## Implication for headline numbers

If we re-run the SF50/100/300 baselines under `numactl --cpunodebind=0
--membind=0`, headline wall times should drop by ~20%:
- SF50: 1.51 s → ~1.20 s
- SF100: 3.02 s → ~2.40 s
- SF300: 8.65 s → ~6.90 s

This needs verification with `--runs 5` for stable medians. The
numbers above are single warm runs.

## Caveats

- Single warm run, no statistics. Variance can hide a 5-10% true
  effect inside larger 1-run noise. The 21% number is robust because
  it dwarfs typical single-run variance, but the precise multiplier
  needs `--runs 5` median.
- Multi-GPU workflows that *intentionally* span nodes (e.g., NCCL
  all-reduce across all 4 GPUs) will see different optima. This
  result is for single-GPU sort only.
- The bind-node-0 vs interleave gap (24.2 vs 23.3 GB/s = 4%) is
  small. If for some reason single-node binding is impossible (e.g.,
  bumping into 504 GB / 1 TB host RAM split per node), interleave is a
  decent second choice.
