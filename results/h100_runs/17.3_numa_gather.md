# 17.3 — NUMA-aware gather: 3.9× speedup from single-node binding

**Date:** 2026-05-03
**Hardware:** Dual-socket Xeon Platinum 8468, 2 NUMA nodes (96 cores each)
**Workload:** Gather 600 M × 120 B records via random permutation (SF100 size = 72 GB)
**Threads:** 32 (the SF100 peak from 17.2)
**Status:** PASS — strongest single-experiment finding of the day

## Result

| Policy | Wall (ms) | GB/s | vs default |
|--------|-----------|------|------------|
| default (kernel auto) | 14138 | 5.09 | 1.0× |
| **bind-node-0** (`numactl --cpunodebind=0 --membind=0`) | **3629** | **19.84** | **3.9×** |
| bind-node-1 (`numactl --cpunodebind=1 --membind=1`) | 3993 | 18.03 | 3.5× |
| interleave (`numactl --interleave=all`) | 5047 | 14.27 | 2.8× |

## Why this is huge

Each invocation is a fresh process — no warm cache, no warmup. So the
table reads "cold-cache gather throughput as a function of NUMA policy."
The default policy gives **5 GB/s on first touch**; binding to a single
node gives **20 GB/s on first touch**, a **3.9× win**.

Production-code implication: in the real sort, gather happens once after
CUB and then nothing else. There is no "warm" — the input file just got
mmap'd, the perm just came off the GPU, and the gather output buffer is
fresh. We are *always* in the cold-cache regime. That means **the
production gather phase has been leaving a 3-4× speedup on the table**.

## Why bind-node-0 wins

- **Local memory latency:** ~80 ns. **Cross-node latency:** ~140-180 ns
  (UPI hop). Random gather with 50% remote pages adds ~50 ns per access
  × 600 M reads = ~30 sec just for cross-node penalty. Matches the
  observed 14138 - 3629 = 10509 ms difference.
- **No coherence ping-pong on output:** when threads spread across both
  nodes write to the (interleaved) output, modified cache lines bounce
  between L3 caches. Single-node confines coherence to one socket.
- **Faster page faults:** the input buffer touch step (single-threaded
  OpenMP loop in the bench) populates page tables. Default policy
  rounds the pages to "local node of the touching CPU." If subsequent
  worker threads run on a different node, all reads are remote. Bind
  pins both to the same node.

## interleave is OK, but not as good

Interleave round-robins pages across both nodes. Threads on each node
see 50% local + 50% remote on average, so latency = (80 + 160)/2 = 120
ns instead of pure-local 80 ns. Matches: interleave 14.3 GB/s vs
bind-node-0 19.8 GB/s = 1.39× ratio = 1 / (80/120).

## What this means for the past "NUMA pinning was slower" finding

Earlier attempts in this session (per BOTTLENECKS.md) reported that
NUMA pinning the gather threads made the warm gather 3-4× SLOWER.
Two possible explanations for the contradiction:

1. **Earlier test pinned threads to a specific CPU (single core),
   not a node.** With 32 threads pinned to core 0, only 1 logical CPU
   does work; rest serialize on the same core. That's a 32× slowdown.
2. **Earlier test pinned threads to wrong node.** If memory was
   allocated on node 1 (where the bench process started) but threads
   were pinned to node 0, every read is remote. Could explain 3-4×.

The right binding policy is "co-locate threads + memory on **the same**
node." `numactl --cpunodebind=0 --membind=0` does exactly that. The
`bind-node-0` and `bind-node-1` results are nearly identical (19.8 vs
18.0 GB/s) — confirming the policy works regardless of which node, as
long as they match.

## Production action items

1. **Implement NUMA binding inside `external_sort.cu`** using
   `numa.h` (`numa_alloc_onnode`, `numa_run_on_node`, `numa_set_localalloc`).
   Pin gather threads + the gather output buffer to one node.
   Expected: 3-4× cold gather speedup → SF100 wall drops from
   3.02 s → ~2.0 s, SF300 wall from 8.65 s → ~5.5 s.

2. **Pre-fault output buffer on the right node before gather**
   using a parallel touch loop pinned to the same node.

3. **Don't migrate.** `numa_set_localalloc()` once at process start is
   often enough; don't sprinkle bindings through the code.

4. **Caveat: GPU/PCIe placement.** GPU 0 is on node 0's PCIe lane
   (verified earlier in 15.1). For a single-GPU sort, binding to node 0
   is also right for PCIe locality. For multi-GPU, would want one
   "gather worker pool" per GPU+node pair.

## Caveats

- Only one cold run per policy. Variance unknown. Should be repeated
  ~5× and median taken before committing to the paper as a result.
- The measurement is "first touch behavior of the kernel." Kernel may
  be smarter about THP / migration in newer versions; we're on Linux
  5.4.0-198-generic which is older.
- Effect at smaller working sets (SF50 = 36 GB) may be less dramatic
  if a larger fraction of the input fits in node-local L3 already.
