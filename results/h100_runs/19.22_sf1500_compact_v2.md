## What changed from 19.21 v1

19.22 v2 eliminates the partition pass-1 count step. Instead of:
1. Pass 1: read input, count records per bucket (184s)
2. Pass 2: read input, route + write compact records (233s)

It does ONE pass with per-thread per-bucket RAM buffers and atomic offset
counters in the bucket files. Each thread fills its 256K-record buffer,
then atomically reserves a region in the destination bucket file via
`__sync_fetch_and_add` and `pwrite`s the buffer.

```cpp
auto flush_bucket = [&](int b) {
    uint64_t bytes = buf_count[b] * COMPACT_REC_SIZE;
    uint64_t offset = bucket_offsets[b].fetch_add(bytes, std::memory_order_relaxed);
    pwrite(bucket_fds[b], buf[b].data(), bytes, offset);
    buf_count[b] = 0;
};
```

Total RAM for buffers: 64 threads × 16 buckets × 10 MB = 10 GB
(small enough; 1 TB host RAM available).

## Per-phase NVMe analysis

NVMe ceilings on this box (single SSD): write 1.3 GB/s, read 3.1 GB/s.

Theoretical minimum for SF1500 with compact pipeline:
- Read input once: 1.08 TB / 3.1 GB/s = 348s = 5.8m (read-bound)
- Write 360 GB intermediate: 360/1.3 = 277s (write-bound)
- Read 360 GB intermediate: 360/3.1 = 116s
- Write 68 GB sorted offsets: 68/1.3 = 52s
- Theoretical floor: max(read input, write intermediate) = ~6m

Our 8m07s is within 35% of theoretical SSD floor. The remaining gap:
- Per-bucket pin overhead in sort phase: ~30s × 4 rounds (overlapped with NVMe)
- Eviction + setup: ~1m12s
- CUDA context + GPU init: small but real

## What it costs to get below 8m

The remaining bottleneck is the partition's read+write phase, which does
1.08 TB read + 360 GB write at NVMe rates. To reduce:
- Skip the intermediate write entirely (streaming partition → sort).
  Multi-day refactor.
- Use multiple NVMe drives (none available on this box).
- Reduce intermediate write further (uint5 offset = 5 bytes vs 8). Saves
  ~3 sec.

The 8m07s is essentially the hardware floor for this single-SSD config.
