# 1.12 — host RAM staging SF1000 (FAIL: same kernel OOM as 1.9)

**Date:** 2026-05-03 01:08 UTC
**GPU(s):** 1× NVIDIA H100 NVL (CUDA_VISIBLE_DEVICES=0)
**Scale:** TPC-H SF1000 (720 GB input, target was 720 GB output)
**Wall time:** ~6 min before SIGKILL
**Status:** **FAIL** — different root cause than 1.6, *exposed by* the
1.6.1 fix.

## Command(s)

```bash
source ~/gpu-research/h100/env.sh
cd ~/gpu-research/gpu_crocsort
CUDA_VISIBLE_DEVICES=0 ./external_sort_tpch_compact \
    --input /mnt/data/lineitem_sf1000.bin --runs 2
```

## Raw numbers

```
$ exit 137 (SIGKILL)

$ dmesg | grep "Out of memory"
[23590.594005] Out of memory: Killed process 42139 (external_sort_t)
    total-vm:1505814688kB    ← 1.51 TB virtual
    anon-rss:1012033684kB    ← 1.01 TB resident
    shmem-rss:23445652kB     ← 23 GB triple-buffer
    file-rss:97428kB
```

So the binary mapped 1.51 TB of address space and faulted in 1.01 TB of
resident memory before the kernel killed it.

## What worked

- **The 1.6.1 lazy-h_pin fix is good** — the OOM happened later than
  before. Previously cudaMallocHost failed at the constructor with
  "invalid argument"; now it actually got into the sort path.

## What failed

The binary wants to hold *two* 720 GB host buffers concurrently: the
pinned input and the gather-output `mmap(... MAP_POPULATE ...)`. Source:
`src/external_sort.cu:1534-1535`:

```c
uint8_t* h_output = (uint8_t*)mmap(nullptr, total_bytes, PROT_READ|PROT_WRITE,
                                    MAP_PRIVATE|MAP_ANONYMOUS|MAP_POPULATE, -1, 0);
```

`MAP_POPULATE` faults the pages eagerly. So 720 GB mmap *immediately*
costs 720 GB resident. Combined with 720 GB pinned input, the binary
needs 1.44 TB of resident memory — 40 % over the 1024 GB host limit.

Total breakdown of the 1.51 TB virtual:
- 720 GB pinned input (`cudaHostRegister`)
- 720 GB MAP_POPULATE'd output
- ~50 GB triple-buffer + GPU scratch + Python interpreter etc.

The kernel kills it as soon as anon-rss approaches the host RAM size.

## Surprises

1. **The `MAP_POPULATE` flag is the smoking gun.** Without it, the output
   buffer would be lazily faulted; pages would only be paid for as the
   gather phase actually writes them, and the kernel could evict input
   pages it no longer needs (they're file-backed via cudaHostRegister
   on an mmap of the input file). With MAP_POPULATE, both 720 GB
   buffers are forced resident from the start. Filed under 1.6.2 (output
   buffer reuse) — but a simpler fix is to drop MAP_POPULATE and let
   the gather phase fault as it writes.

2. **The "host RAM staging" approach can't actually work as the queue
   item described.** The whole premise — "load input into pinned RAM,
   then sort from RAM" — fails because the sort *also* needs an output
   buffer of equal size, doubling the RAM footprint. We need either
   (a) sort-in-place into the input buffer, or (b) stream the gather
   output to disk in chunks rather than holding it all in RAM.

## Interpretation

SF1000 single-GPU is *technically* feasible on this hardware — there's
enough HBM (94 GB) and PCIe bandwidth — but the *output staging* is what
exhausts host RAM, not the input or GPU compute. The fix is a code
change (1.6.2 / 1.12.1) to either reuse the input buffer or stream the
output. Until that lands, **1.12 is mechanically blocked, not
science-blocked**.

## Follow-ups

- **1.12.1 (new) — drop MAP_POPULATE on the gather output.** Lets the
  pages fault lazily as the gather kernel writes them. Should let
  SF1000 single-GPU reach the sort phase. Combined with 1.6.2
  (in-place gather), would unlock SF1000 cleanly.
- **1.12.2 (new) — measure the resident-mem fault curve.** With
  MAP_POPULATE off, the gather phase will progressively fault output
  pages. We can measure how that interacts with the still-pinned
  input pages (does the kernel migrate? does it swap? does it OOM
  later in the run?).
- 1.13 (next) is the same staging exercise at SF1500 — but SF1500 input
  alone (1080 GB) is bigger than host RAM so 1.13 would fail at input
  pin. Skip with reason "SF1500 input does not fit in host RAM".
