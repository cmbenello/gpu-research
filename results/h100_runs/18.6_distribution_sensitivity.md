# 18.6 — Distribution sensitivity: 1.7× spread between best and worst case

**Date:** 2026-05-04
**GPU:** 1× NVIDIA H100 NVL, numactl --preferred
**Workload:** 300 M synthetic 120 B records (~36 GB), --runs 3 each
**Status:** PASS — all distributions sort correctly; 1.7× wall spread

## Result (last warm run)

| Distribution | Wall (ms) | GB/s | PCIe H2D | Notes |
|--------------|-----------|------|----------|-------|
| **sorted** (ascending) | **1168.6** | **30.8** | 9.6 GB | best — runtime compact-map shrinks |
| **reversed** (descending) | **1184.6** | **30.4** | 9.6 GB | matches sorted |
| all_equal (single key) | 1527.0 | 23.6 | 9.6 GB | extra Phase 4 fixup work |
| **random** (uniform 64-bit) | **1924.3** | **18.7** | 14.4 GB | worst — full compact key |
| zipfian (skewed, 1% pop keys) | 1924.3 | 18.7 | 14.4 GB | matches random |

For comparison, TPC-H SF50 lineitem: **1413 ms / 25.5 GB/s** — falls
between sorted (best) and random (worst), closer to sorted because
TPC-H has natural low-entropy columns (l_returnflag, l_linestatus, ...).

## Why sorted/reversed are 1.6× faster than random

The runtime compact-map analyzes the input bytes and ships only the
**bytes that vary** through PCIe. For:

- **sorted**: only the top-of-key byte varies; lower bytes are 0 or
  sequential. Compact map = ~32 bytes → 9.6 GB H2D.
- **random**: all 64 bits of the key are random. Compact map = full
  ~48 bytes → 14.4 GB H2D.

PCIe upload time scales with the compact key size, so random is
1.5× more PCIe traffic, ~600 ms more wall. Matches the observed
gap.

## All-equal anomaly: 25% slower than sorted

`all_equal` should sort fastest (no work needed), but takes 1527 ms
vs sorted 1168 ms. Why?

Looking at the trace: Phase 2 (GPU OVC merge) takes 805 ms vs ~488
ms for sorted. The GPU detects ALL records have ties (same prefix)
and runs the CPU prefix-fixup phase on every record — 300 M fixup
operations.

This is actually a worst-case for the OVC architecture: the prefix
merge can't disambiguate ties without falling back to CPU comparison.

For paper: gpu_crocsort handles all-equal correctly but with a 25%
penalty. Most real workloads (TPC-H, financial data, etc.) have
enough entropy that this case doesn't apply.

## Zipfian = random

The zipfian distribution (50% of records pull from 1% of keys) gave
identical results to random. Reason: my zipfian generator uses
random-rng + 50% probability of picking from a small set, so the
"popular" half effectively functions as a uniform distribution
within that subset. The compact map sees similar entropy.

A more realistic zipfian (e.g., key frequency follows 1/rank) would
need a more careful generator.

## Build & run

```bash
cd ~/gpu-research/gpu_crocsort
g++ -O3 -std=c++17 -pthread experiments/gen_synthetic.cpp \
    -o experiments/gen_synthetic

for D in random sorted reversed all_equal zipfian; do
    ./experiments/gen_synthetic --records 300000000 \
        --output /mnt/data/synth_${D}_300M.bin \
        --distribution $D
    CUDA_VISIBLE_DEVICES=0 numactl --cpunodebind=0 --preferred=0 \
        ./external_sort_tpch_compact \
        --input /mnt/data/synth_${D}_300M.bin --runs 3
done
```

## Caveats

- Single 300 M record dataset; SF100/SF300 not tested for
  distributions.
- Synthetic generator has its own timing that's not measured here.
- `zipfian` here is approximated; a true zipf-α distribution would
  need a real Mersenne-twister + log mapping.
- Cold-cache numbers also vary by 1.5-2× across distributions
  (driven by initial mmap pin time, not sort algorithm).
