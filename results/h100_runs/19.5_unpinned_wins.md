# 19.5 — Unpinned cudaMemcpy beats pinned by 30× at large-scale one-shot reads

**Date:** 2026-05-04
**GPU:** 1× H100 NVL
**Status:** PASS — major architectural finding

This is a **direct architectural recommendation** for the paper:
the existing pin-the-input model loses to streaming-without-pin for
the SF1500 regime.

## Microbench result (4.29 GB warm cache)

```
UNPINNED memcpy: 443 ms (avg 3 iters) → 9.69 GB/s
PINNED memcpy:   78 ms (avg 3 iters) → 55.28 GB/s
                 [pin took 1723 ms = 2.49 GB/s]
  Net pinned (pin + 1 transfer): 1800 ms → 2.39 GB/s
```

## Extrapolated to SF1500 (per-bucket, 4× 18 GB compact-key transfers)

| Approach | Cost | Per-bucket cost |
|----------|------|-------------------|
| Pin 270 GB + 4× pinned 18 GB transfer | 225 s pin + 1.3 s transfer | **226 s** |
| 4× unpinned 18 GB transfer | 4 × 1.86 s | **7.4 s** |

**Unpinned is 30× faster** for the one-shot SF1500 case.

## Per-transfer crossover analysis

When does pin amortize?

For 18 GB chunks at SF1500 (4 chunks per bucket):
- N = 1 transfer: pin loses (1.86 s vs 1.86 s)
- Wait — pin's per-transfer is 0.33 s (55 GB/s), unpinned is 1.86 s.
  Pin wins per-transfer.
- But pin pays 225 s up front.
- Crossover: 225 / (1.86 - 0.33) = **148 transfers** before pin pays off.

Our SF1500 bucket only does 4 transfers. We're 37× below crossover.

## Why this matters for the paper

The current gpu_crocsort architecture:
1. mmap input file
2. cudaHostRegister entire mmap (the 1.2 GB/s ceiling from 19.4)
3. cudaMemcpy from pinned to GPU (peak 55 GB/s)

For SF1500's 4-bucket sort = 4 × 4 transfers = 16 transfers from pinned
buffer. Doesn't amortize the pin cost.

**Better architecture:**
1. mmap input file (or use direct read syscall)
2. Skip cudaHostRegister entirely
3. cudaMemcpy from non-pinned at ~10 GB/s

This changes SF1500 from 49m15s → predicted **34 min** by saving the
4 × 4-min pin steps (16 min off wall).

## What the user gets if we implement this

Concrete prediction: SF1500 dropping from 49m15s → ~34 min, with no
correctness change.

That's a paper claim: **"we identified that cudaHostRegister is a
1.2 GB/s ceiling that doesn't amortize for one-shot OLAP sorts.
Skipping it improves SF1500 wall by 31%."**

## Caveats

- Microbench was warm-cache (file in OS page cache). Cold-cache
  unpinned would be NVMe-bound at 6.2 GB/s peak — still much faster
  per-byte than pin's 1.2 GB/s.
- With small transfers, pin amortizes fast. For SF50 (4 small chunks)
  the crossover is much lower.
- Pinned has lower CPU overhead per transfer (no DMA marshaling).
  Unpinned uses CUDA's bounce-buffer path which uses some CPU.
- This experiment didn't test concurrent multi-stream pinned vs
  unpinned; might shift the analysis.

## Recommendation

Add an env override `SKIP_HOST_PIN=1` to gpu_crocsort that:
- Skips cudaHostRegister on input mmap
- Uses cudaMemcpy from non-pinned for compact extract uploads
- Falls back to pinned for output-buffer interactions

Predicted wins:
- SF1500 wall: 49m → 34 min (31% off)
- SF1000 wall: 14m20s → 12m (15% off)
- Smaller scales: minimal difference (pin cost is small)

## Build & run

```bash
nvcc -O3 -arch=sm_90 experiments/pinned_vs_unpinned.cu \
     -o experiments/pinned_vs_unpinned

CUDA_VISIBLE_DEVICES=0 ./experiments/pinned_vs_unpinned \
    /mnt/data/lineitem_sf50.bin
```
