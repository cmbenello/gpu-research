# 6.3.1 — PCIe5 transfer rate vs NUMA policy: no effect

**Date:** 2026-05-04
**GPU:** 1× NVIDIA H100 NVL (tested GPU 0 and GPU 2)
**Workload:** 8 GB cudaMemcpy H2D and D2H, 5 iterations each
**Status:** PASS — NUMA policy does NOT affect PCIe transfer rate

## Result

GPU 0 (PCIe root on NUMA node 0):

| Policy | H2D (GB/s) | D2H (GB/s) |
|--------|-------------|-------------|
| default | 55.10 | 54.31 |
| `--cpunodebind=0 --membind=0` (correct) | 55.11 | 55.06 |
| `--cpunodebind=1 --membind=1` (wrong) | 55.12 | 54.45 |

GPU 2 (PCIe root on NUMA node 1):

| Policy | H2D (GB/s) | D2H (GB/s) |
|--------|-------------|-------------|
| default | 55.11 | 54.51 |
| `--cpunodebind=1 --membind=1` (correct) | 55.13 | 55.11 |
| `--cpunodebind=0 --membind=0` (wrong) | 55.10 | 54.47 |

## Interpretation

PCIe transfer is **55 GB/s in both directions** (~86% of theoretical
peak) regardless of NUMA policy. The 0-1.5% spread between policies
is run-to-run noise. **NUMA binding does NOT affect PCIe**.

Why: `cudaMemcpy` from pinned host memory uses GPU-managed DMA. The
GPU's DMA engine pulls from host RAM through its PCIe root port
directly. The path is GPU → PCIe root port → IOMMU → pinned page;
NUMA-bound CPUs don't sit in that path.

## Why H100 NVL gets 55 GB/s vs PCIe 5 x16's 32 GB/s spec

H100 NVL has **two PCIe 5 x16 links per GPU** (a "double connector"
configuration). Effective unidirectional bandwidth is 2 × 32 = 64
GB/s peak, of which we measure 55 GB/s = 86% efficiency. That
matches typical PCIe efficiency ceilings.

(For comparison, PCIe 4 x16 would peak at 32 GB/s, half this.)

## Useful negative result

Earlier results in this session (17.3 / 17.3.2) showed that NUMA
policy has a **massive effect on host-RAM-bound CPU phases** (gather
3.9× cold, sort 1.84× warm). This 6.3.1 result shows that NUMA does
NOT help the PCIe upload phase. So:

- The 1.84× SF100 wall improvement from numactl in 17.3.2.1 is
  entirely from the host-RAM phases (extract + gather), NOT from PCIe.
- If the sort were PCIe-bound, numactl wouldn't help.
- Future "make PCIe faster" work needs different techniques
  (zero-copy, GPUDirect Storage, larger transfer chunks, etc.).

## Build & run

```bash
cd ~/gpu-research/gpu_crocsort
nvcc -O3 -arch=sm_90 experiments/pcie_numa.cu -o experiments/pcie_numa

./experiments/pcie_numa 0                                      # default
numactl --cpunodebind=0 --membind=0 ./experiments/pcie_numa 0  # correct
numactl --cpunodebind=1 --membind=1 ./experiments/pcie_numa 0  # wrong
```

## Caveats

- Single test, no statistical bounding. Variance ~1%.
- cudaMemcpy DMA path; results may differ for cudaMemcpyAsync,
  unified memory, or GPUDirect Storage paths.
- 8 GB transfer size; smaller transfers may show different
  per-transfer overhead breakdowns.
