## Comparison: where pre-partitioning helps

| Method | SF300 wall | Globally sorted? |
|--------|-------------|-------------------|
| 1-GPU SF300 numactl | **6.59 s** | yes (single output) |
| 4-GPU distributed (with partition, 18.5) | 207 s = 3.5 min | yes (4 sorted buckets) |
| 4-GPU distributed (sort only, partition pre-done) | **144 s = 2.4 min** | yes (4 sorted buckets) |

**At SF300, single-GPU is ALWAYS faster** (6.59 s vs 144 s minimum
for distributed). Pre-partitioning saves 30% of the distributed
wall, but distributed still loses to single-GPU because the sort
fits comfortably in HBM at SF300.

## Why distributed loses at SF300

Per-GPU in 4-way concurrent:
- Each GPU sorts ~54 GB at ~10-12 GB/s = ~4-5 s wall
- But pair coordination + NVMe write of bucket output adds overhead
- Aggregate per-GPU concurrent throughput: ~22 GB/s × 4 = 88 GB/s
  theoretical
- Realized: 216 GB / 144 s = 1.5 GB/s end-to-end

Compared to single-GPU SF300 at 32.7 GB/s, distributed is 22× slower.

The reason: the distributed pipeline writes 216 GB to tmpfs/NVMe
and reads back, plus pair coordination adds latency. Single-GPU
keeps everything in HBM.

## When pre-partitioning DOES win

For SF500+ where single-GPU OOMs in HBM, distributed is the only
option. Pre-partitioning would save the 39% partition cost from
the 17.3.2.3.3 / 17.3.2.7.6 numbers:
- SF500 distributed (with partition): 7m10s → without partition: ~5m
- SF1000 distributed (with partition): 20m08s → without partition: ~12m

**Pre-partitioning is a useful workflow optimization when the input
is going to be sorted multiple times** (e.g., interactive analytics
on the same dataset).

## Build & run

```bash
bash /tmp/run_18.5b_prepartitioned_sf300.sh
```

Buckets persist in /dev/shm/sf300_buckets between runs (until cleanup).

## Caveats

- Single sweep, no statistical bound.
- Distributed sort writes output to /dev/shm; system page-cache
  state could affect 2nd+ runs.
- The 54 GB / GPU is at the "fits comfortably in 94 GB HBM" regime,
  unlike SF1000 where each GPU's 180 GB chunk doesn't fit.

## Conclusion for paper

For the paper's headline claims:
- **Single-GPU SF300 (6.59 s) is the right number to lead with.**
- For SF500+, distributed is mandatory; pre-partitioning is a
  ~30% optimization on top.
- The story is: "GPU sort wins when data fits in HBM. Above HBM
  capacity, it's a NVMe-bound workflow problem, not a sort problem."
