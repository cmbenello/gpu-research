# 19.4 — cudaHostRegister cost: 1.2 GB/s constant

**Date:** 2026-05-04
**GPU:** 1× H100 NVL
**Workload:** mmap + cudaHostRegister + cudaHostUnregister of various lineitem files
**Status:** PASS — characterizes the SF1500 cold-load bottleneck

## Result

| File | Size (GB) | mmap (ms) | cudaHostRegister (ms) | unregister (ms) | munmap (ms) | total (ms) | GB/s |
|------|-----------|-----------|------------------------|-----------------|-------------|------------|------|
| sf10 | 7.2 | 0 | 5776 | 160 | 187 | 6123 | 1.18 |
| sf50 | 36 | 0 | 28352 | 753 | 762 | 29867 | 1.20 |
| sf100 | 72 | 0 | 56452 | 1507 | 1591 | 59550 | 1.21 |
| sf300 | 216 | 0 | 173628 | 5100 | 4659 | 183386 | 1.18 |
| sf500 | 360 | 0 | 314136 | 8702 | 8480 | 331318 | 1.09 |

## Findings

1. **`cudaHostRegister` is at ~1.2 GB/s constant** across all sizes
   from 7 GB to 360 GB. Linear in size; perfectly predictable.

2. **`cudaHostUnregister` is ~22-46× faster** (23 GB/s typical):
   it just decrements ref-counts on already-pinned pages.

3. **`mmap` is essentially free** (under 1 ms) because Linux maps
   lazily — actual page faults happen later via cudaHostRegister
   walking and pinning each page.

4. **The unregister + munmap overhead is ~5% of total** even at
   SF500 size. The register step is the dominant cost.

## Why this matters

**For SF1500 sequential sort, the cudaHostRegister of 270 GB per
bucket = 225 sec = 3.75 min**. Times 4 buckets = 15 min just for
pinning. That matches the observed 7-8 min/bucket overhead in
19.1.3 (the rest is sort + write).

For PER-WORKLOAD scaling:
- SF50: pin = 28 sec, sort kernel = 0.45 sec → pin is 60× the sort
- SF100: pin = 56 sec, sort = 1 sec → pin is 56× sort
- SF300: pin = 174 sec, sort = 6.6 sec → pin is 26× sort
- SF1000: pin = ~600 sec, sort = ~50 sec → pin is 12× sort
- SF1500: pin = ~900 sec/bucket × 4 = 3600 sec, sort total ~3 min

**As scale grows, the relative pin cost decreases (closer to sort
work), but absolute pin cost dominates.**

## Why is `cudaHostRegister` so slow?

Per CUDA documentation, `cudaHostRegister` performs:
- Page table walk for the entire region
- Pin each page (cannot be swapped out)
- Optionally GPU-mapping (we don't ask for it explicitly here)

On Linux, page-locking 1.2 TB of memory (for SF1500's 4 buckets)
requires updating ~280 M PTE entries. At 200 ns per entry, that's
~56 sec — about right.

The hard limit is **Linux page table operations**, not CUDA itself.

## Routes around this bottleneck

### Option A — Skip the pin entirely
Use `cudaMemcpy` directly from regular (non-pinned) host memory.
Slower per-transfer (12-20 GB/s vs 55 GB/s pinned) but **no per-byte
pin cost**.

For SF1500: 4 × 270 GB compact uploads (32 B/record × 2.25 B records ≈
72 GB) at 15 GB/s unpinned = 4 × 5 sec = 20 sec instead of 4 min
of cudaHostRegister.

### Option B — Pin once, reuse across buckets
Allocate one large pinned buffer (~94 GB). Read each bucket into
that buffer in chunks. Pin cost amortized to **single 78 sec call**
instead of 4 × 4 min.

Net SF1500 savings: ~12 min off the 49 min wall.

### Option C — GPUDirect Storage
GPU reads NVMe directly via cuFile API. No host pinning ever. But
needs a compatible NVMe + driver setup.

## Implications for paper

This 1.2 GB/s pin rate is a **hardware-architectural ceiling** that's
worth calling out. It's not a sort-specific limitation — it applies
to any GPU workload that pins its input.

For the SOTA comparison: RMG Sort and Vortex sidestep this by either
(a) using small in-VRAM data (RMG) or (b) using GPU-only access via
PCIe routing (Vortex). Our SF1500 pays the cost; our **headline
should include "first SF1500 globally sorted under 1.2 GB/s host-pin
ceiling" or similar.**

## Caveats

- Single sweep, no sweep-over-sweep variance.
- The slight throughput drop at SF500 (1.09 vs 1.18 GB/s for smaller)
  may be NUMA-related; not investigated.
- Did not test cudaHostAlloc directly — different code path.
- Did not test the cudaHostRegisterReadOnly flag carefully; we
  used Read+Mapped which may be slower than just Read.
