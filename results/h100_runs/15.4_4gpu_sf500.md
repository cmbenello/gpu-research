# 15.4 — 4-GPU SF500 partition sort (each GPU handles 1/4 by position)

**Date:** 2026-05-03 07:26 UTC
**GPU(s):** 4× NVIDIA H100 NVL (all 4 active concurrently)
**Workload:** SF500 (3.0 B records, 360 GB) split into 4 contiguous chunks
of 750 M records (~90 GB each); each GPU sorts its chunk independently.
**Wall time:** 6 min 4 s for 4 × 3 sorts (cold + 2 warm each)
**Status:** PASS (all 12 sorts verified correct on their slice)

## Command(s)

Required new CLI flags `--offset-records` and `--limit-records` on
`external_sort_tpch_compact` (commit XXXX) so each binary instance sorts
a different position-range of the input file without re-generating data.

`/tmp/run_15.4.sh`:
```bash
TOTAL=3000028242
PER_GPU=$(( TOTAL / 4 ))
LAST=$(( TOTAL - 3 * PER_GPU ))

CUDA_VISIBLE_DEVICES=0 $BIN --input $INPUT --offset-records 0              --limit-records $PER_GPU  --runs 3 &
CUDA_VISIBLE_DEVICES=1 $BIN --input $INPUT --offset-records $PER_GPU       --limit-records $PER_GPU  --runs 3 &
CUDA_VISIBLE_DEVICES=2 $BIN --input $INPUT --offset-records $((2*PER_GPU)) --limit-records $PER_GPU  --runs 3 &
CUDA_VISIBLE_DEVICES=3 $BIN --input $INPUT --offset-records $((3*PER_GPU)) --limit-records $LAST     --runs 3 &
wait
```

CSV: [`15.4_4gpu_sf500.csv`](15.4_4gpu_sf500.csv).

|-----|-------|-------|-------|-----------|-----------|
| 0   | 7.56  | 7.56  | (...) | 7.56 s    | 11.9      |
| 1   | 44.72 | 7.16  | 5.65  | 5.65 s    | 15.9      |
| 2   | 29.74 | 8.11  | 6.34  | 6.34 s    | 14.2      |
| 3   | 5.67  | 18.57 | 6.77  | 5.67 s    | 15.9      |

**Aggregate single-pass time** = max(per-GPU) for the warm pass:
- Run 2 ("warm" but with NVMe contention): max = 8.11 s
- Run 3 (truly warm): max = 6.77 s

So **360 GB sorted in 6.77 s warm best across 4 GPUs = 53.2 GB/s aggregate**.

For comparison:
- Single-GPU SF500: **OOM** — d_keys_10byte allocation needs 198 GB > 94 GB HBM. Cannot run.
- Single-GPU SF300 (the largest that works): 8.94 s warm @ 24.2 GB/s.
- Extrapolated single-GPU SF500 if it worked: ~15 s @ 24 GB/s. 4-GPU is **2.2× faster** than the hypothetical single-GPU.

## What worked

- **All 12 sorts PASS verification on their slice.** No GPU-to-GPU
  cross-contamination, no kernel state pollution between concurrent
  sorts on different GPUs.
- **Per-GPU performance is consistent** with single-GPU SF100 throughput
  (15.4 GB/s warm best matches solo SF100). The 4-way concurrency at
  this larger working set still doesn't degrade per-GPU performance
  meaningfully — confirms the box's PCIe / HBM / host RAM are
  independent enough.
- The new `--offset-records` / `--limit-records` flags work cleanly with
  the file mmap path. fseeko + sliced fread loads only the needed
  90 GB per process.

## What failed

Cold first runs were brutally slow (29-45 s on GPUs 1 + 2). Cause: 4
processes each trying to read 90 GB of UNIQUE positions in the SF500
file from NVMe. NVMe sustained read is 3 GB/s shared; reading 360 GB
non-overlapping at 3 GB/s = 120 s shared between 4 readers means each
sees ~30 s. Matches.

Run 2 was inconsistent across GPUs (some 5 s, some 18 s) — suggests
NVMe + page cache eviction is still happening across runs. Run 3 is
the cleanest "fully warm" measurement.

## Surprises

1. **4-GPU SF500 is the first scale where multi-GPU genuinely beats
   single-GPU**, not just at *throughput* but at *capability*. Single-GPU
   literally cannot sort SF500 — it OOMs. So the comparison isn't 4× vs
   1× speedup; it's "infinite speedup" because 1× doesn't exist.

2. **Per-GPU SF125 throughput is ~15 GB/s**, slightly *less* than
   single-GPU SF300's 24 GB/s. Why? At SF125 (90 GB) the data fits
   below the OVC threshold so each GPU's binary takes the slow-path
   single-pass full-key upload (the 0.3.1 bug — uploads 66 B per
   record instead of 32 B). PCIe is the limit. After 0.3.1 lands,
   per-GPU should jump to ~24 GB/s, aggregate to ~95 GB/s.

3. **GPU 0 was *slowest* but didn't see contention drops** — it ran
   at consistent ~7.5 s every run. Likely because GPU 0 always has
   the leading 90 GB, which gets cached first and stays cached.
   The other GPUs evict it to load their slices.

4. **Total wall (cold start to all done) = 6 min 4 s** — but this
   includes the brutal cold phase and 3 runs each. The per-iteration
   warm cost is ~7 s.

## Interpretation

This is the 4-GPU sort **capability headline**: the H100 NVL box can
sort **360 GB in 6.77 s warm best at 53.2 GB/s aggregate** when
position-sliced across 4 GPUs. **Single-GPU SF500 doesn't exist** —
the binary OOMs trying to allocate the key buffer. So this isn't a
"scaling curve" point; it's a capability that single-GPU cannot match.

Important caveat: **this is *not* a globally-sorted output.** Each
GPU's chunk is internally sorted but the chunks are partitioned by
record position, not by key value. To produce a single
globally-sorted SF500 result, we'd need either:
- (a) Sample-partition before sort (range partition → each GPU's
  chunk is a key-range, output concatenates), or
- (b) Multi-way merge after sort (4 sorted runs → tournament merge).

(a) is filed as 15.4.1 below. (b) is also possible — would add ~3 s
host-side merge to the 7 s sort, giving SF500 globally sorted in ~10 s.
