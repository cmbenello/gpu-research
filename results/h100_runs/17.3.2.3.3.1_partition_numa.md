# 17.3.2.3.3.1 — Partition under numactl: NVMe-bound, no NUMA win

**Date:** 2026-05-04
**Workload:** SF100 (72 GB) partition into 4 buckets, under
`numactl --cpunodebind=0 --preferred=0`
**Status:** PASS — confirms partition is I/O-bound, not memory-bound

## Result

```
Sample + sort + splitters: 519 ms
Pass 1 (count): 10982 ms       (6.55 GB/s read = ~ NVMe peak)
Pass 2 (route): 6646 ms        (21.7 GB/s effective r+w)
Total: 21465 ms                (3.35 GB/s end-to-end)
```

real time 21.5 sec, sys time 7m7s (NVMe + page table churn).

## Why no NUMA win

Pass 1 reads 72 GB at 6.55 GB/s — that's the **NVMe sequential read
ceiling** on /mnt/data (`nvme1n1`, single device, 6.2 GB/s peak per
`dd if=... iflag=direct`). NUMA binding doesn't accelerate disk I/O
on a single device.

Pass 2 does 72 GB read + 72 GB write at 21.7 GB/s effective. Output
is to `/dev/shm` (tmpfs), which is RAM-backed. Here NUMA *could*
help, but the read from NVMe is still serialized at 6.2 GB/s, so
it's mostly read-bound.

## Rules out a partition optimization

The 119-sec SF500 partition step (17.3.2.3.3) is at the NVMe I/O
ceiling and cannot be sped up by:
- Numactl wrap (no effect, this experiment)
- More CPU threads (already 64 threads; CPU is idle waiting on I/O)

To speed up partition, you need:
- **Faster storage** — RAID0 across multiple NVMe, or NVMe-of-fabric
  with higher aggregate bandwidth.
- **Pre-partitioned input** — store data already partitioned to a
  per-bucket file, eliminating the partition step entirely.
- **GPU-resident partition** — read input directly into GPU,
  partition there. Bypasses CPU entirely but adds a PCIe upload
  bottleneck.

For our paper, the partition step is now identified as the
fundamental bottleneck of the distributed pipeline. Single-GPU sort
is at GPU-near-peak; only partition sits at I/O.

## Comparison

| Workload | Wall | GB/s |
|----------|------|------|
| SF100 partition + numactl | 21.5 s | 3.35 |
| SF500 partition (17.3.2.3.3) | 119 s | 3.02 |
| Single NVMe peak (dd direct) | — | 6.2 |

So end-to-end is 1.85-2.06× SLOWER than NVMe peak because Pass 1 +
Pass 2 do 360 + 360 = 720 GB of I/O for 360 GB of input. So the
"effective" rate is 720 GB / 119 sec = 6.05 GB/s — basically NVMe
peak, just amortized over total work.

## Conclusion

Partition is at the storage ceiling. Move on; no win available
without different hardware.
