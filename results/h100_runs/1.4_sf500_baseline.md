# 1.4 — TPC-H SF500 baseline + USE_BITPACK (FAIL: OOM on single H100 NVL)

**Date:** 2026-05-02
**GPU(s):** 1× NVIDIA H100 NVL (94 GB HBM, CUDA_VISIBLE_DEVICES=0)
**Scale:** TPC-H SF500 (360.00 GB, 3,000,028,242 × 120-byte records)
**Wall time:** ~2 min before crash (NVMe load + OOM)
**Status:** **FAIL** — `cudaMalloc` for 198 GB key buffer fails on a 94 GB GPU.

## Command(s)

```bash
source ~/gpu-research/h100/env.sh
cd ~/gpu-research/gpu_crocsort
CUDA_VISIBLE_DEVICES=0 ./external_sort_tpch_compact \
    --input /mnt/data/lineitem_sf500.bin --runs 3
```

## Raw numbers

```
Allocating 360.00 GB pinned host memory... OK
Loading from /mnt/data/lineitem_sf500.bin... 360 GB in 108294 ms (3.32 GB/s NVMe)
[ExternalSort] GPU: 98.60 GB free, budget: 64.09 GB
[ExternalSort] Triple-buffer: 3 × 21.36 GB (178 M records each)
[ExternalSort] Sorting 3000028242 records (360.00 GB)
== Step 1+2: Key Upload ==
  Runtime compact map: 27/66 bytes vary, 32B compact key (4 LSD passes)
== Step 1+2: Key Upload ==
CUDA error at src/external_sort.cu:2510: out of memory
```

`d_keys_10byte` allocation at line 2510 = `num_records × KEY_SIZE` = 3B × 66 = **198 GB**, more than 2× the H100 NVL's 94 GB.

## What worked

- The 360 GB host-side load worked (pinned memory + mmap + huge pages).
- NVMe sustained 3.32 GB/s on the cold read of SF500.
- The runtime cmap detection on 3 B records still finished cleanly.

## What failed

The path-selection logic at `external_sort.cu:2069`:

```c
bool use_ovc = (total_keys_bytes + est_arena > free_mem_now * 0.9) &&
               (ovc_total + 2*buf_bytes + 512*1024*1024 < free_mem_now * 0.95);
```

At SF500 with ~88 GB free on a 94 GB H100:
- `total_keys_bytes` = 198 GB, est_arena ~72 GB. First condition (keys too
  large): **true**.
- `ovc_total` = 60 GB, `2*buf_bytes` = 42 GB, plus 0.5 GB scratch = 102.5 GB
  required vs ~84 GB available. Second condition (OVC fits): **false**.

So `use_ovc = false`, the binary skips the OVC path and falls through to
the unchunked single-pass path at line 2510 which tries to malloc 198 GB
and crashes. There's no fallback to a "chunked OVC" path.

## Surprises

1. **The "Step 1+2: Key Upload" header prints twice.** Once before the path
   decision and once after. Cosmetic but suggests the control flow was
   patched together; the path picker isn't well-isolated from its consumers.

2. **SF500 *almost* fits in aggregate HBM across 4 H100 NVLs** — total
   aggregate HBM = 4 × 95830 MiB ≈ 376 GB. SF500 is 360 GB. So with a
   **multi-GPU** version of the OVC path, SF500 fits entirely in HBM
   without any host-staging at all. That's exactly what 15.4
   (partition_then_sort_4gpu) is designed for.

3. **The single-GPU SF500 limit is somewhere between SF300 (works) and
   SF500 (OOMs).** A binary search would identify the exact OOM scale —
   a useful chart for the paper. Probably around SF400 / 290 GB.

## Interpretation

This FAIL is informative: it pins down the exact scale where the current
single-GPU OVC path runs out of HBM (somewhere between SF300 and SF500),
and it strongly motivates 15.4 (4-GPU partition-and-sort) as the
*next-natural-architecture* to push past SF300. Filing 3.2 (chunked_merge)
also remains a valid one-GPU alternative if multi-GPU isn't ready.

The H100 NVL's 94 GB HBM is, ironically, *just under* 1 GiB per record at
SF1 — meaning the hardware can hold a sorted output for ~SF1 worth of
records as 32 B compact keys (94 GB / 32 B ≈ 2.9 B records ≈ SF480 of TPC-H).
We're at the natural single-GPU edge here, not a code-quality issue.

## Follow-ups

- **1.4.1 (new) — binary-search for the exact single-GPU OOM scale.** Try
  SF400 and SF450. Likely SF400 still fits, SF450 OOMs. Useful for the
  paper's "scale envelope" chart.
- **1.4.2 (new) — add a chunked-OVC fallback.** When `use_ovc = false`
  due to ovc_total + 2*buf_bytes not fitting, chunk the OVC pass too.
  Currently the binary just OOMs.
- **1.4.3 (new) — 4-GPU SF500 via 15.4 path.** Aggregate 376 GB HBM
  across 4 GPUs covers the 360 GB SF500. Should beat any single-GPU
  number by a wide margin.
- 1.5 generate_sf1000 is **gated** on 1.1.1 (chunked encode) since the
  current encoder peaks at ~1.2 TB host RAM at SF1000.
