# 17.3.2.3 — 4-GPU SF500 with NUMA-spread (per-GPU numactl wrap)

**Date:** 2026-05-04
**GPU(s):** 4× NVIDIA H100 NVL (concurrent)
**Workload:** SF500 (3.0 B records, 360 GB) split into 4 contiguous chunks
of ~750 M records (~90 GB each); each GPU sorts its own chunk.
**Method:** GPU 0/1 wrapped with `numactl --cpunodebind=0 --membind=0`,
GPU 2/3 wrapped with `numactl --cpunodebind=1 --membind=1`.

## Result

Per-GPU walls (3 runs: cold + 2 warm):

| GPU | NUMA | run 1 | run 2 | run 3 |
|-----|------|-------|-------|-------|
| 0   | 0    | 5813  | 5477  | **3654** |
| 1   | 0    | **3785** | 5454 | 5752 |
| 2   | 1    | **3879** | 4993 | 5534 |
| 3   | 1    | 5674  | 5159  | **3665** |

(All wall times in ms. Bold = each GPU's best.)

Per-run aggregate (max wall across 4 GPUs = single-pass time):

| Run | max wall | Aggregate GB/s |
|-----|----------|-----------------|
| 1   | 5813 ms  | 61.9            |
| 2   | **5478 ms** | **65.7 ← best** |
| 3   | 5752 ms  | 62.6            |

## Comparison to 15.4 baseline (no numactl)

| Method | Aggregate wall | Aggregate GB/s | vs single-GPU SF300 (numactl) |
|--------|----------------|-----------------|---------------------------------|
| 15.4 (no numactl, max-of-warm) | 6770 ms | 53.2 | 1.92× |
| 17.3.2.3 (numactl spread, run 2 best) | **5478 ms** | **65.7** | **2.37×** |

NUMA-spread gives **1.23× aggregate speedup** over the original 15.4
result.

## Best single-GPU runs in 4-way concurrent

GPU 0 (run 3) and GPU 3 (run 3) both hit ~3660 ms = 24.6 GB/s — same
as our single-GPU SF50/100 numactl numbers (24-26 GB/s). So when 2
GPUs share a node WITHOUT contending for the same input file pages,
each GPU achieves single-GPU throughput.

But in the same run, GPU 1 (the partner of GPU 0 on node 0) only
got 5752 ms. The 1.6× difference between GPU 0 and GPU 1 in the
same run shows **two GPUs sharing one NUMA node still contend for
memory channels** — node 0's DDR5 bandwidth has to be split across
both GPU pipelines. That contention costs ~57% of GPU 1's
throughput.

## Theoretical headroom

If every GPU could hit its single-GPU best (3654 ms) concurrently:
- Aggregate wall: 3654 ms
- Aggregate GB/s: 360/3.65 = 98.6 GB/s

The gap from 65.7 (achieved) to 98.6 (theoretical) = **33% headroom**
left by NUMA-pair contention. Closing this would require:

- Splitting input file across 4 NUMA partitions (SF500 stored as 4 ×
  90 GB files on different node-local mounts), so each GPU reads
  from a different memory channel pool.
- OR using GPU-resident input (HBM only), avoiding host RAM bandwidth
  entirely. Possible at SF50 per GPU but not SF125-per-GPU on H100 NVL.

## Why GPU 0 and GPU 3 are fast on run 3 but GPU 1 and GPU 2 are slow

Hypothesis: page placement is set on first-touch. The two GPUs on
node 0 (GPU 0 and GPU 1) start their input scans at offsets 0 and
750 M records (different positions in the 360 GB file). Whichever
finishes its first-pass touch first locks in node-0 pages; the
other ends up with cross-node pages or page conflicts. This would
also explain why GPU 1's BEST run was the cold (run 1) — fewer
pages were committed by the time it scanned.

A definitive answer would need numa_maps inspection during the
sort. Out of scope for this micro-experiment.

## Recommendations for paper

- **Multi-GPU aggregate at 65.7 GB/s is the new headline** for
  4-GPU position-partitioned (15.4 / 17.3.2.3 with NUMA spread).
- Cite the `numactl --cpunodebind=N --membind=N` two-line wrap as
  the deployment recipe for any number of GPUs (1, 2, 4).
- Note the 33% headroom for future scaling work.

## Caveats

- Single sweep, not repeated. Variance across sweeps unknown.
- The "1/4 partition" sort produces 4 sorted slices, **not a globally
  sorted output**. This is the 15.4 model. For a globally sorted
  result, see 15.5.3.
- Adding numactl wrap to 15.5.3 (sample-partition + paired sort)
  is a logical next step (17.3.2.3.1).
