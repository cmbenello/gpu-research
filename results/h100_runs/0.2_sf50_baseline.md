**Date:** 2026-05-02
**GPU(s):** 1× NVIDIA H100 NVL (CUDA_VISIBLE_DEVICES=0)
**Scale:** TPC-H SF50 (36.00 GB, 300,005,811 × 120-byte records)
**Wall time:** ~3 min/config (5 runs each)
**Status:** PASS (all 10 runs)

## Command(s)

```bash

    --input /mnt/data/lineitem_sf50.bin --runs 5

    --input /mnt/data/lineitem_sf50.bin --runs 5
```

## What worked

- Both configurations PASS verification on all 5 runs (post-fix from 0.1).
- Run-to-run wall time stable to ~5% on warm runs (excluding cold first-run).
- BITPACK detection found 165 active key bits → 21 B packed (24 B with 8-byte
  CUB padding). This is meaningful compression (32 → 24 B = 25% reduction in
  the sort key) but the actual wall-time win was within noise at SF50.
- SF50 traverses the slow path (300 M records > 179 M `buf_records`), so the
  run-gen + K-way merge code path is exercised. After the 0.1 fast-path fix,
  the slow path also continues to PASS.

## What failed

Nothing — all 10 runs PASS. Note one operational gotcha: the SF50 generation
took 234 s (153 MB/s effective), bottlenecked by the polars→numpy→mmap encode
step in `gen_tpch_fast.py` rather than tpchgen-cli itself. Worth thinking
about for SF1000 (would be ~78 min if scaling holds — most of the gen budget
is encode, not parquet).

## Surprises

1. **USE_BITPACK didn't help at SF50.** Pre-fix expectation: 25% smaller key
   → fewer LSD passes → faster sort. Reality: warm runs of bitpack and baseline
   are within 1% of each other (median 2810 vs 2828 ms). The CPU prepack cost
   appears to fully offset the GPU sort savings at this scale. Hypothesis: the
   sort itself is small (~1.7 s of the 2.8 s total wall time is run-gen which
   *includes* the upload, and only ~1 s is merge); cutting the merge by 25%
   saves ~250 ms but the CPU prepack on 300 M records adds ~300 ms. Net wash.
   This suggests bitpack only wins when the sort phase dominates — likely at
   SF300+ where merge passes are a larger fraction of total wall time.

2. **First-run "cold" penalty is huge** — for baseline: 4083 ms vs 2625 ms
   (best warm) = 1458 ms slower. The bitpack first-run penalty is even bigger
   (7891 ms vs 2789 ms = 5101 ms slower). This is the page-cache warmup of
   the 36 GB input on NVMe. The bitpack cold case is much worse because the
   bitpack path scans all records to compute per-position min/max before
   encoding — cold pages double the disk pressure. Implication: any
   benchmarking has to ignore run 1 and report run-2-onward statistics.

3. **PCIe compression is real and substantial** — 36 GB raw records → 21 GB
   on the wire (1.71× reduction). At PCIe 5 ceiling of ~32 GB/s, the floor for
   PCIe alone is 21 GB / 32 GB/s = 660 ms. We measured 1725 ms run-gen +
   ~1050 ms merge = 2775 ms. So PCIe is ~24% of total wall time — confirms the
   bitpack hypothesis above (sort phases dominate).

## Interpretation

SF50 with the compact path runs in 2.6-2.9 seconds at 12.7-13.7 GB/s
effective on a single H100 NVL, beating the RTX 6000 baseline of 1.6 s only
because the RTX 6000 baseline must have been at warmer state — at the same
scale we're 1.7-1.8× slower in *wall time*, not better. The H100 advantage
will only emerge at scales where (a) sort merge dominates wall time
(SF300+), and (b) the data exceeds host RAM so PCIe compression saves
*re-reads* and not just one upload. **The SF50 number is not the headline;
it's the floor on which SF100 / SF300 numbers will sit.**

The non-obvious finding is that USE_BITPACK does nothing at this scale. We
either need to push to SF100/SF300 to see its win, or rethink the prepack
to be cheaper (e.g. fuse with the H2D upload to overlap CPU pack with PCIe).

  ms savings at SF50, more at larger scales.
