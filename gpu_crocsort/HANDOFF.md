## Architecture Overview

### Path 1: Small keys (KEY_SIZE <= ~16B, fits in GPU)
Used for GenSort and TPC-H 10B key.
1. **Strided DMA**: cudaMemcpy2D reads only KEY_SIZE bytes per record from host
2. **GPU CUB radix sort**: LSD on uint64 chunks (2 passes for 10B, 11 for 88B)
3. **Download permutation**: 4B per record
4. **CPU gather**: apply perm to original h_data → sorted output

PCIe: ~8.4GB for 60GB data (0.14x amplification). Time: 7.2s.

### Path 2: Large keys (KEY_SIZE > 16B, keys exceed GPU memory)
Used for TPC-H 88B keys at SF50+.

**Current best (13.2s for SF100):**
1. **Run generation**: Triple-buffered pipeline, per-chunk LSD radix sort
   - H2D full records → GPU sorts → D2H sorted records
   - With compact key (-DUSE_COMPACT_KEY): strips constant bytes, 4 CUB passes instead of 9
   - Pipeline: sort-bottlenecked at ~400ms/chunk (compact) or ~600ms/chunk (standard)
2. **CPU K-way merge**: Sample-based partitioning, 48-thread parallel K-way heap merge
   - Single pass over all runs

**Experimental OVC architecture (6.3s run-gen + fixup too slow):**
1. **Run generation WITHOUT D2H**: Upload records, GPU sorts, extract 8B prefix + perm on GPU, SKIP downloading sorted records. Run gen: 6.3s (was 9.7s = 35% faster!)
2. **GPU prefix merge**: CUB sort on 8B prefix + global_perm. 233ms.
3. **CPU gather**: 3.3s (apply perm to original h_data)
4. **CPU fixup**: Sort within 8B prefix groups. THIS IS THE BOTTLENECK (34s for 3,817 groups of 157K records)

## The Critical Problem to Solve

The OVC/prefix architecture saves 3.4s on run-gen by eliminating 76GB of D2H. But it introduces a CPU fixup that takes 34s because:

- 8B prefix (bytes 0-7) has only **3,817 unique values** for TPC-H SF100
- Average group: **157,201 records** — too large for fast CPU sort
- 16B prefix (bytes 0-15) has **~965K unique values**, avg 621 records — much better
- But 16B prefix merge on GPU needs more memory than we have for alt buffers

### Projected times if 16B prefix merge works:
```
Run gen (no D2H):    6.3s
GPU 16B merge:       0.5s  (2 CUB LSD passes + gather)
CPU gather:          3.3s
CPU fixup (965K grp): ~1-2s (621 avg records, fits in L2 cache)
TOTAL:              ~11.1s (vs 13.2s current = 16% faster)
```

### The Specific Technical Blocker

CUB SortPairs(key, value) carries only ONE satellite array. For 2-pass LSD merge on 16B prefix, we need to reorder THREE arrays (prefix1, prefix2, global_perm) together.

**Solution (partially implemented but OOM):**
1. Sort (key=prefix2, val=identity_index) — CUB gives us the reordering index
2. Use identity_index to gather prefix1 and global_perm via custom gather kernels
3. Sort (key=prefix1, val=global_perm) — final sort

**GPU memory breakdown:**
- Persistent: d_prefix_buffer (4.8GB) + d_ovc_buffer (4.8GB) + d_global_perm (2.4GB) = 12GB
- After freeing d_buf (11.7GB) + sort_ws (~2GB): ~13.7GB available
- But CUDA fragmentation leaves only ~10.5GB contiguous
- Need: d_pfx_alt2 (4.8GB) + d_idx_buf (2.4GB) + d_idx_alt (2.4GB) + CUB temp (0.5GB) = 10.1GB
- **BARELY fits after freeing d_ovc_buffer (4.8GB) before alloc**

The gather kernels exist (`gather_uint64_kernel`, `gather_uint32_kernel`). The code structure is in place. The issue is pure memory management — need to carefully free and re-allocate in the right order to avoid fragmentation.

## Key Source Files

### `src/external_sort.cu` (~1700 lines)
The main file. Key functions:
- `sort_chunk_on_gpu()` (line ~586): Per-chunk LSD radix sort. Has OVC extraction path (line ~649): when `d_ovc_buffer != nullptr`, extracts dual prefixes + global perm instead of reordering records.
- `generate_runs_pipelined()` (line ~672): Triple-buffered run generation. Skips D2H when `d_ovc_buffer != nullptr` (OVC mode).
- `sort()` (line ~1060): Main entry point. Decision tree:
  - Single-chunk: direct sort
  - Keys fit in GPU: single-pass strided DMA + CUB sort
  - OVC mode (line ~1363): `use_ovc=true` when keys don't fit but 20B/record OVC buffers do
  - Fallback: run-gen + CPU K-way merge (the working 13.2s path)

### `src/merge.cu` (~500 lines)
GPU merge kernels:
- `merge_2way_kernel`: Full-record 2-way merge
- `merge_keys_only_kernel`: Key-only merge with permutation
- `merge_ovc_kernel` (line ~450): OVC-based merge path (NOT used, OVC deltas are relative not absolute)
- `launch_merge_ovc`: Extern C launcher for OVC merge

### `include/ovc.cuh` (~170 lines)
OVC encoding/decoding: `ovc_compute_delta()`, `ovc_encode()`, flag checks.
`key_compare()` function used throughout for byte-by-byte key comparison.

### `include/loser_tree.cuh` (~165 lines)
Complete GPU loser tree with OVC support. Not currently used but fully implemented.

### `include/record.cuh`
Configurable via compile flags: `KEY_SIZE`, `VALUE_SIZE`, `RECORD_SIZE`.

### `Makefile`
Important flags: `-DUSE_COMPACT_KEY` enables the 26-byte compact key extraction for TPC-H. `--extended-lambda` needed for device lambdas in thrust calls.

## What to Try Next (Priority Order)

### 1. Fix the 16B GPU prefix merge (highest impact, ~11s total)
The code structure is there. Need to solve the memory management:

```cpp
// In sort() OVC path, after run gen:
// 1. Free d_ovc_buffer (4.8GB) — not needed after prefix extraction
// 2. Allocate d_idx_buf (2.4GB) for identity tracking
// 3. CUB sort (key=d_ovc_buffer_SAVED, val=d_idx_buf)  
// 4. Gather prefix1 and perm using d_idx_buf
// 5. CUB sort (key=prefix1, val=perm) — final merge

// Problem: d_ovc_buffer is freed at step 1 but needed at step 3.
// Fix: do step 3 BEFORE step 1, or copy d_ovc_buffer to d_pfx_alt2 first.
```

Alternative: allocate buffers more carefully. The 16B merge needs:
- Pass 1 input: prefix2 (4.8GB, read-only) + identity (2.4GB, read-only after init)
- Pass 1 output: sorted_idx (2.4GB) → used to gather prefix1 (need 4.8GB temp) + gather perm (reuse identity buffer)
- Pass 2 input: prefix1 (4.8GB) + perm (2.4GB)
- Pass 2 output: sorted perm (2.4GB)

Total unique buffers: prefix1 (4.8) + prefix2 (4.8) + perm (2.4) + pfx_alt (4.8) + idx (2.4) + CUB temp (0.5) = 19.7GB. With 25GB GPU: 5.3GB slack. Should fit if allocated in the right order.

### 2. Optimize the compact key extraction (medium impact)
The `build_compact_keys_kernel` reads from `__constant__` memory for the byte mapping. Could potentially hardcode the 26 byte positions for better register usage.

### 3. Try running on the server directly with Claude Code

### 4. Explore segmented sort for prefix groups
Instead of CPU fixup, use CUB's `DeviceSegmentedRadixSort` for the second sort pass within 8B prefix groups. The 3,817 segments of ~157K records each are in CUB's sweet spot.

### 5. Use a more discriminating prefix
Instead of bytes 0-7, extract bytes that maximize uniqueness. E.g., orderkey (bytes 46-53) has 262K unique values. But this would give wrong sort order (orderkey is not the most significant column). Could use a COMPOSITE prefix: hash(first 16 key bytes) → 8 bytes with near-unique values. But hash doesn't preserve sort order.
