# H100 plug-and-play autoresearch

The plan: get an H100 box, run **two commands**, and let it iterate experiments autonomously for the full week.

## Day 0 — bootstrap (one command)

On the freshly delivered H100 box (assumed: Ubuntu 22.04+, NVIDIA driver + CUDA 12.4+ already installed, your SSH key in your agent, `ANTHROPIC_API_KEY` env set):

```bash
curl -L https://raw.githubusercontent.com/cmbenello/gpu-research/research/overnight-runs-cs-uchicago/h100/setup.sh | bash
```

Or if you've already cloned the repo:

```bash
cd ~/gpu-research && bash h100/setup.sh
```

What `setup.sh` does (idempotent):
1. Detects GPU + CUDA + sets compute capability (`sm_90` for H100)
2. Installs system deps (gcc, tmux, htop, parallel, numactl, ...)
3. Installs Python deps (duckdb, polars, pyarrow, matplotlib, ...)
4. Installs Node.js + Claude Code via nvm
5. Clones / updates the repo on `research/overnight-runs-cs-uchicago`
6. Builds `gpu_crocsort` + `external_sort_tpch_compact` + variants for sm_90
7. Generates TPC-H SF10 (~7 GB) for verification
8. Runs an end-to-end smoke test (`PASS` expected)
9. Writes `h100/env.sh` with the resolved paths
10. Drops `.bootstrap_done` so the autoresearch script knows it's safe to start

Total time: ~5 min if dependencies are pre-installed, ~15 min on a fresh box.

## Day 0 onward — start the loop (one command)

```bash
bash ~/gpu-research/h100/start_autoresearch.sh
```

This spawns a `tmux new-session -d -s autoresearch` running `h100/.loop_body.sh`. Each iteration:

1. `git pull` the latest queue + code
2. Send `h100/iterate_prompt.md` to Claude Code in headless mode (`claude --print --dangerously-skip-permissions ...`)
3. Claude reads `h100/research_queue.md`, picks the highest-priority unticked experiment, marks it `[~]`, runs it (capped at 60 min wall time), writes findings to `results/h100_runs/<exp_id>.md`, ticks it `[x]` (or `[!]` on failure), and commits + pushes
4. Wait 5 minutes (cache-friendly), repeat

The loop runs **forever** until you `bash h100/stop_autoresearch.sh` or kill the tmux session.

### Watch / tail

```bash
tmux attach -t autoresearch       # follow the live loop output (Ctrl-b d to detach)
tail -f ~/gpu-research/results/h100_runs/loop_logs/iter_*.log
git log --oneline -10              # see what experiments landed
cat h100/research_queue.md         # see what's left
```

### Stop

```bash
bash ~/gpu-research/h100/stop_autoresearch.sh
```

## What's in the queue

[`h100/research_queue.md`](research_queue.md) is the source of truth for what gets run. The autoresearch loop is allowed to **add new items** at the bottom under "Running notes" when findings warrant follow-ups, so the queue grows over the week.

Tiers (top-down priority):
- **Tier 0 — Sanity:** SF10/SF50/SF100 verify
- **Tier 1 — Scale envelope:** SF300, SF500, SF1000 (assuming disk allows)
- **Tier 2 — Compression at scale:** codec ratios, PCIe sweep, dictionary
- **Tier 3 — Architecture:** sample-sort, chunked merge, GPU gather, record compression
- **Tier 4 — Cross-validation:** DuckDB/Polars/cuDF baselines
- **Tier 5 — Real-world data:** NYC Taxi 12mo, BTC, wide records
- **Tier 6 — Profiling:** Nsight traces, HBM saturation, phase breakdown sweep
- **Tier 7 — Paper / report material:** charts + writeup

Edit the queue freely — the loop reads it fresh each iteration.

## What "plug and play" actually requires

For the ideal experience, before the H100 arrives:
- [ ] SSH key on the box (so `git push` works)
- [ ] `ANTHROPIC_API_KEY` env var set (or run `claude login` once interactively)
- [ ] At least 2 TB of fast NVMe for SF1000 (=720 GB) to fit
- [ ] Adequate cooling (sustained sort runs warm the H100 quickly)

If any of these aren't ready, `setup.sh` will tell you and stop early. Re-run after fixing.

## Safety rails

- Loop never pushes to `main` — only to `research/overnight-runs-cs-uchicago`
- Each experiment is wall-time capped at 60 min
- Each gen step is disk-capped at 80% of `$DATA_DIR`
- GPU OOM is handled (logged, marked `[!]`, loop continues)
- If the queue is fully exhausted (`Tier 0–6` all `[x]` or `[!]`), the loop runs Tier 7 once, then idles at 1-hour intervals — won't burn API budget on nothing
